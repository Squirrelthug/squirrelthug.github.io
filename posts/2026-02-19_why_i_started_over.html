Why I Started Over — Again

On moving from distributed microservices to a modular monolith, and why the distinction matters.

There's a version of this project that lived as four separate GitHub repositories. Each one was a standalone service with its own dependencies, its own API surface, its own deployment concerns. lilith-context-manager. lilith-memory-sdk. lilith-orchestrator. lilith-scl-engine. It looked clean on a diagram. It looked like something a real company would ship.

I threw all of it away.

This is the explanation of why, because I think the reasoning is worth writing down — both for anyone following along and for myself, so that the next time I'm tempted by an architecture that looks good in a box diagram, I have somewhere to point back to.

The Problem With Building What Looks Right

Microservices are the correct answer for a team of engineers managing independent domains that need to scale and deploy independently. I am not a team. I am one person who works in construction during the day and writes code at night.

The overhead of running four services during development is not theoretical friction — it is actual, daily friction. Every time I wanted to trace a bug across a request lifecycle, I was looking at four log streams. Every time I wanted to change a data contract, I was touching four repositories. Every time I wanted to understand whether a performance issue was in the context manager or the orchestrator, I was doing archaeology.

I had designed a system that looked like what I wanted to build eventually, not what I needed to build right now. The distinction is important.

You don't frame walls before the foundation cures. And you don't design the foundation for the shed when you're planning to build the house.

The microservice architecture was the second of those mistakes. I was building the infrastructure for a distributed system that multiple engineers would work on, before I had proven that the core loop — voice in, reasoning, voice out, personality consistent — worked the way I needed it to work.

What a Modular Monolith Actually Is

There's a misconception that "monolith" means "unstructured mess." It doesn't. A modular monolith is a single deployable process with clean internal module boundaries — enforced by code discipline rather than by network calls. The modules can't import each other's internals. They communicate through a shared internal event bus. The data contracts are just as strict as they would be in a distributed system. But there's no HTTP overhead on every internal call. There's no serialization tax. There's no network partition to worry about. And there is exactly one log stream.

If I ever need to pull a module out into its own service — because some specific component genuinely needs to scale independently, or because a future version of this runs on multiple physical devices — the module boundary is already there. The separation already exists. I just add a network layer on top of an interface that already exists. I don't have to restructure anything.

That's the key insight. I'm not giving up the option to distribute later. I'm just not paying the cost of distribution before I need it.

The Deeper Lesson

The architecture I was building before was impressive-looking. It had the kind of diagram that makes sense to an interviewer or a senior engineer who works with distributed systems professionally. It signaled competence in the right vocabulary.

But I kept catching myself optimizing for the diagram instead of for the working system. That is a reliable early warning sign that something has gone wrong in the design process. When you're making decisions to make the architecture look correct rather than to make it work correctly for your actual situation, you've already lost the thread.

So: new foundation. Same final vision. But a foundation designed for a solo developer who needs to be able to hold the entire system in his head, debug it alone, change it quickly, and actually use it every day. The v5 documents describe that architecture in detail. What I'm building now is what v5 describes — not what the old repos implied.

The system is smaller now than it was when it had four repositories. It will be much larger soon. But this time, it'll be built in the right order.
