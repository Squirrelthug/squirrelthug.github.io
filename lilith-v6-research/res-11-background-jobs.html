Q2. APScheduler Job Priorities and Ordering

APScheduler does not provide a general priority scheduler. It controls overlap with max_instances, missed-run behavior with coalescing/misfire settings, and execution pools. If a sync job takes longer than its interval, max_instances=1 prevents pile-up; use coalesce=True so missed runs collapse into one run. Triggered jobs can be run immediately by calling scheduler.add_job(func, trigger="date", run_date=None, args=[...]) or by pushing onto Lilith's own enhancement queue.

from apscheduler.schedulers.background import BackgroundScheduler
from apscheduler.executors.pool import ThreadPoolExecutor
from datetime import datetime, timezone

scheduler = BackgroundScheduler(
    executors={"default": ThreadPoolExecutor(max_workers=4)},
    job_defaults={"coalesce": True, "max_instances": 1, "misfire_grace_time": 60},
)

scheduler.add_job(sync_google_calendar, "interval", minutes=15, id="google_calendar")
scheduler.add_job(consolidate_memory, "cron", minute="*/30", id="memory_consolidation")

def trigger_enhancement(record_id: str):
    scheduler.add_job(run_l1_l2_enhancement, "date", run_date=datetime.now(timezone.utc),
                      args=[record_id], id=f"enhance:{record_id}", replace_existing=True)

def stop_background_jobs():
    request_shutdown.set()
    scheduler.shutdown(wait=True)

Q3. SQLite WAL Mode for Concurrent Reads

Enable WAL after opening and keying each SQLCipher database: PRAGMA journal_mode=WAL;. SQLite persists WAL mode in the database file after it is set, but Lilith should still assert it at open. WAL allows readers to continue while a writer appends to the WAL; checkpointing or schema changes can still cause brief contention. Use PRAGMA busy_timeout=5000; and short transactions.

Use one SQLCipher connection per thread/job. Do not share a connection across worker threads. If pooling is added later, make it a pool of thread-local connections, not one global connection.

Q4. Graceful Shutdown on SIGTERM

On window close or SIGTERM, set a global shutdown event, stop accepting new jobs, cancel queued enhancement jobs, and call scheduler.shutdown(wait=True). Python threads cannot be force-killed safely; long LLM/API calls must use client timeouts and periodically check the shutdown event between records. In-progress enhancement should be checkpointed using a job table with states such as queued, running, succeeded, failed_retryable. On next startup, reset stale running jobs to queued.

In PySide6, call the shutdown routine from closeEvent, accept the event after shutdown completes, and keep the UI responsive by showing a short shutdown overlay if needed.

Q5. Threading and sqlite3 in Python

Python's sqlite3 module enforces check_same_thread=True by default, meaning a connection may only be used in the creating thread. The underlying SQLite library is usually serialized/thread-safe, but sharing connections still creates transaction and cursor hazards. check_same_thread=False is only safe when the application serializes all access with its own lock; Lilith should avoid that design.

Best practice: each job opens its own keyed SQLCipher connection using pathlib.Path, sets required pragmas, runs a short transaction, closes the connection, and logs mutations via log_event().

Final recommendation: Use APScheduler 3.x for periodic jobs and immediate timer-style triggers, backed by a small explicit enhancement queue where priority matters. Use connection-per-job SQLCipher access with WAL, busy timeout, and short transactions. No blocker found.

Action Plan — Background Jobs
  • Periodic jobs: APScheduler 3.x BackgroundScheduler — Google sync, memory consolidation, staleness checks.
  • Triggered jobs: Internal priority queue for L1/L2 enhancement, triggered by DB writes.
  • DB access: WAL mode + busy_timeout = 5000 + one connection per thread.
  • Shutdown: scheduler.shutdown(wait=True) from PySide6 closeEvent — in-progress jobs complete before exit.
  • Safety: max_instances=1 and coalescing enabled on periodic jobs — no pile-up if a job runs long.
