RES-01 — Qt6 / PySide6 Licensing

Sprint: Research · Agent 1 · Report 1 of 7
Researched: June 28, 2026
Library version: PySide6 6.11.1 (Qt 6.8+ LTS compatible)

Q1. LGPL v3 Requirements for a Closed-Source PySide6 App

To use PySide6 (under LGPLv3) in a closed-source desktop application without releasing the proprietary source code, the following requirements must be met:

  • Relinking / Swapping: The end-user must have the technical ability to replace the PySide6 libraries with their own modified versions. Since Python is interpreted, importing PySide6 dynamically from the environment naturally meets this, provided the user can access and overwrite the PySide6 package directory.
  • PyInstaller Caveat: Packaging the application as a single executable (--onefile) makes replacement extremely difficult, as it extracts to a temporary folder at runtime. To ensure LGPL compliance, we must:
    1. Distribute the application using One-Folder mode (--onedir), where the Python interpreter, proprietary code (compiled to .pyc), and PySide6 shared objects (.so / .dll / .dylib) are stored in separate files inside an accessible directory structure. This makes it trivial for users to swap files.
    2. Alternatively, provide the PyInstaller .spec build script and intermediate object files (or instructions) so that an advanced user can re-bundle the application with a modified version of PySide6.
Design note — LGPL compliance is friction only, not a code risk Following LGPL v3 compliance has zero effect on Lilith's runtime behavior, codebase stability, or reliability. It is a distribution packaging requirement only. The practical steps: use PyInstaller --onedir (one folder) instead of --onefile (single executable), include attribution notices in an About dialog, and link to Qt source. The --onedir approach is actually preferable on its own merits — startup is faster because files don't need to be extracted at runtime, and the folder structure is easier to inspect and debug. None of this touches application code.

Q2. Does the Paid-Sync SaaS Model Trigger Qt Commercial License?

No, a paid cloud-sync service does not trigger commercial licensing requirements under LGPLv3. The GNU LGPLv3 obligations are triggered strictly by the distribution of the software to end users.

  • The server-side sync backend resides on our servers and is not distributed to the customer. Therefore, server-side code has zero LGPL/GPL copyleft obligations (often referred to as the "SaaS loophole").
  • Since Qt/PySide6 does not use the AGPL (Affero GPL), network access does not trigger any source-sharing obligations for the server or client sync components.
  • The local client application is distributed and must comply with LGPLv3, but its connection to a paid SaaS sync backend is perfectly valid under LGPL.

Q3. Static vs Dynamic Linking in Python / PyInstaller

Python's import mechanism is dynamically resolved at runtime, which satisfies the LGPLv3 dynamic linking preference. However, the packaging step is where linking-equivalent issues arise:

  • Dynamic Linking: PySide6 contains compiled C++ extension modules (shared libraries). If these are distributed as separate files (e.g., PySide6.QtCore.pyd), it represents standard dynamic linking, and users can swap them.
  • Static Linking Equivalent: Using PyInstaller's --onefile compiles all resources into a single monolithic binary. This acts as a static link equivalent because the user cannot swap out individual components without extracting and rebuilding the executable. To maintain LGPL compliance, One-Folder mode (--onedir) is required.

Q4. LGPL v3 Attribution and Distribution Requirements

For compliant distribution of a closed-source app using PySide6 under LGPLv3, the application must include:

  • License Copies: Include the full text of the GNU GPLv3 and GNU LGPLv3 licenses in the application's installation folder and accessible via the UI.
  • Prominent Attribution Notice: Place a notice in the application's "About" or "Legal" dialog stating: "This application uses PySide6/Qt6, licensed under the GNU Lesser General Public License (LGPL) version 3. The Qt Toolkit and PySide6 are copyright of The Qt Company and/or its contributors."
  • Source Code Offer: Provide a link to the official Qt source code repository and include instructions explaining how the user can modify and swap out the PySide6 libraries (e.g., by replacing the folder contents in a --onedir distribution).

Q5. Qt Quick 3D — Separate Licensing Terms?

CRITICAL BLOCKER: Yes. Unlike core Qt/PySide6 modules which are dual-licensed under LGPLv3 and Commercial, Qt Quick 3D is licensed under GPLv3 (and Commercial) only. It is NOT available under LGPLv3.

If we distribute a closed-source desktop application that imports or links to Qt Quick 3D under the open-source license, the GPLv3's strong copyleft triggers. This would legally obligate us to release the entire proprietary source code of Lilith v6 under the GPLv3. This directly conflicts with the closed-source business model of Lilith v6.

Decision made — ModernGL (MIT) chosen for the orb Qt Quick 3D is dropped. The animated orb is implemented using ModernGL 5.x (MIT license) embedded in a PySide6 QOpenGLWidget. ModernGL attaches to Qt's existing OpenGL context inside initializeGL(), requiring zero additional windowing setup. The orb is a UV sphere mesh rendered with a custom GLSL Fresnel glow shader; amplitude from Kokoro TTS drives a u_amplitude uniform each frame. Full implementation details in RES-03 (ModernGL — Animated Orb). No commercial Qt license is required. This blocker is fully resolved.

Q6. Qt Commercial License — Cost and Scope

If LGPLv3/GPLv3 is insufficient (due to the Qt Quick 3D blocker), a commercial license is required:

Tier Cost (Per Developer) Key Features / Scope
Qt for Application Development (Small Business) ~$499 - $618 / year For startups with <$1M revenue/funding and <5 employees (max 3 dev licenses). Permits closed-source use of GPL-only modules like Qt Quick 3D and static linking.
Qt for Application Development (Standard) ~$3,624+ / year (starts ~$302/mo) For larger teams. Includes full Qt Commercial license, premium technical support, static linking, and proprietary module access.

Note: PySide6 itself is included under the Qt for Application Development license. If PyQt6 from Riverbank Computing was used instead, a separate commercial license would have to be purchased from them.

Considering — Qt Small Business license vs MIT 3D alternative The $499–618/yr Small Business tier is viable if Lilith generates revenue — roughly 8–10 subscriptions at a typical indie pricing tier covers it annually. The question for the MIT alternative is not implementation complexity (a cloud reasoning model can handle custom GLSL shader code) but cross-platform rendering quality and consistency. Tools like ModernGL and Vispy are MIT-licensed and proven on Linux, macOS, and Windows. The orb requires: a sphere mesh, a GLSL emission/glow shader, real-time amplitude as a uniform, and smooth positional animation — all achievable without Qt Quick 3D. A decision will be logged in decisions.md and reflected in RES-03.

Q7. Official Guidance on Local App + Paid Cloud Sync Under LGPL

There is extensive precedent for commercial, closed-source desktop apps using Qt under the LGPL and charging for supplementary services. Examples include:

  • Autodesk Maya: Uses PySide/Qt under LGPL for its UI extensions, while keeping the main program closed-source and commercial.
  • Adobe Photoshop / Premiere Elements: Use dynamically linked Qt libraries under LGPL, distributed commercially as closed-source desktop applications.
  • Plex Media Player: Closed-source desktop client that uses LGPL Qt libraries and interacts with the paid Plex Pass cloud subscription.
Verdict: RESOLVED — PySide6 LGPL compliance is clear, no commercial license needed Qt Quick 3D has been dropped. ModernGL (MIT) is used for the orb instead. PySide6 core modules (LGPL v3) are fully compatible with closed-source commercial distribution when distributed via PyInstaller --onedir, with attribution notice in the About dialog and a link to Qt source. No Qt commercial license is required. The SaaS sync model does not trigger additional Qt licensing. All distribution requirements are straightforward packaging steps — zero codebase impact.
