RES-14 — sounddevice Audio I/O

Researched: June 28, 2026 · sounddevice 0.5.5 · PortAudio v19 API family

Q1. Linux Audio Backend — PulseAudio, PipeWire, ALSA

sounddevice uses PortAudio. On Linux, PortAudio commonly reaches consumer devices through ALSA; PipeWire systems usually provide compatibility through pipewire-alsa and/or pipewire-pulse. Ubuntu 22.04+ PipeWire setups generally work if those compatibility layers are installed. For consumer headsets, prefer the default input/output devices unless the user explicitly selects devices in Lilith settings. Avoid hard-coding ALSA device names.

Sample-rate negotiation can differ between PulseAudio and PipeWire. Lilith should request 16 kHz mono for capture, but be ready to open the device at its default rate and resample to 16 kHz if PortAudio rejects the format.

Q2. WASAPI on Windows

PortAudio supports several Windows host APIs. sounddevice exposes host APIs through query_hostapis() and device records. To explicitly select WASAPI, choose a device whose hostapi points to the WASAPI host API or pass WASAPI-specific settings. WASAPI generally has lower latency and better modern-device behavior than MME; DirectSound/MME are fallback paths. Avoid exclusive mode by default because it can conflict with calls, browsers, and games.

Q3. CoreAudio on macOS / Apple Silicon

sounddevice wheels support macOS and use PortAudio/CoreAudio; Apple Silicon users should install native arm64 Python/wheels to avoid Rosetta. CoreAudio devices often run natively at 44.1 or 48 kHz. A MacBook built-in mic may not expose direct 16 kHz capture, so Lilith should test device support and resample to 16 kHz for VAD/Whisper when needed.

Q4. Microphone Stream to VAD Pattern

The callback must do the minimum: copy the chunk and enqueue it. VAD and transcription run on worker threads. This prevents PortAudio underflow/overflow caused by model inference or database writes in the audio callback.

import queue, threading
import numpy as np
import sounddevice as sd

SAMPLE_RATE = 16000
BLOCKSIZE = 480
audio_q: queue.Queue[np.ndarray] = queue.Queue(maxsize=128)
utterance_q: queue.Queue[np.ndarray] = queue.Queue()
stop = threading.Event()

def mic_callback(indata, frames, time_info, status):
    chunk = indata[:, 0].copy()
    try:
        audio_q.put_nowait(chunk)
    except queue.Full:
        pass

def vad_loop(vad_iterator):
    buffer = []
    in_speech = False
    while not stop.is_set():
        try:
            chunk = audio_q.get(timeout=0.1)
        except queue.Empty:
            continue
        event = vad_iterator(chunk)
        if event and "start" in event:
            in_speech = True
            buffer = [chunk]
        elif in_speech:
            buffer.append(chunk)
        if event and "end" in event and in_speech:
            utterance_q.put(np.concatenate(buffer))
            buffer = []
            in_speech = False

def whisper_loop(model):
    while not stop.is_set():
        try:
            utterance = utterance_q.get(timeout=0.1)
        except queue.Empty:
            continue
        segments, _ = model.transcribe(utterance, language="en")
        publish_transcript(" ".join(s.text for s in segments))

stream = sd.InputStream(samplerate=SAMPLE_RATE, channels=1, dtype="float32",
                        blocksize=BLOCKSIZE, callback=mic_callback)

Q5. Streaming TTS Playback

Use sounddevice.OutputStream with a callback and a bounded queue/ring buffer. sounddevice.play() is for complete arrays and is not the right primitive for streaming TTS. If synthesis outruns playback, the queue fills; either block the synthesis producer briefly or drop only after a policy decision. Track playback completion by counting queued samples and setting an event when the producer is done and the buffer drains.

import queue, threading
import numpy as np
import sounddevice as sd

play_q: queue.Queue[np.ndarray] = queue.Queue(maxsize=32)
producer_done = threading.Event()
playback_done = threading.Event()

def output_callback(outdata, frames, time_info, status):
    out = np.zeros((frames, 1), dtype=np.float32)
    filled = 0
    while filled < frames:
        try:
            chunk = play_q.get_nowait()
        except queue.Empty:
            break
        n = min(len(chunk), frames - filled)
        out[filled:filled+n, 0] = chunk[:n]
        filled += n
        if n < len(chunk):
            play_q.queue.appendleft(chunk[n:])  # use collections.deque in production
            break
    outdata[:] = out
    if producer_done.is_set() and play_q.empty() and filled == 0:
        playback_done.set()

def kokoro_producer(kokoro_stream):
    for samples in kokoro_stream:
        play_q.put(samples.astype(np.float32))
    producer_done.set()

with sd.OutputStream(samplerate=24000, channels=1, dtype="float32",
                     blocksize=1024, callback=output_callback):
    playback_done.wait()

Q6. Device Disconnection and Hot-Plug

PortAudio/sounddevice does not provide a simple cross-platform hot-plug callback at the Python level. A disconnected device can surface as callback status flags, stream errors, or exceptions on subsequent reads/writes. Lilith should catch stream failures, stop the stream, re-query devices, fall back to the current default device, and show a non-blocking UI notification. Polling device lists every few seconds while a stream is failed is acceptable.

Q7. License
