RES-10 — LLM Providers & API Abstraction

Researched: June 28, 2026 · anthropic 0.112.0 · Claude Sonnet family pricing/model docs checked as current public docs

Q1. Anthropic Prompt Caching Implementation

Anthropic prompt caching is enabled by marking content blocks with cache_control. Cache breakpoints can be placed on system, tool, and message content blocks. For Lilith, cache the stable system/personality prompt and stable user context/personal-truths block. Do not cache the latest user turn or volatile database excerpts.

Anthropic documents minimum cacheable prompt lengths by model family, with current Claude Sonnet/Opus cache blocks requiring at least about 1,024 tokens. The default cache TTL is 5 minutes; Anthropic also documents longer TTL options for supported models. The cache is server-side and keyed by prompt prefix, so it can be reused across Python process restarts as long as the exact prefix is sent again before TTL expiry.

from anthropic import Anthropic
client = Anthropic()

resp = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=300,
    system=[
        {
            "type": "text",
            "text": LILITH_SYSTEM_PROMPT,
            "cache_control": {"type": "ephemeral"},
        },
        {
            "type": "text",
            "text": STABLE_USER_CONTEXT,
            "cache_control": {"type": "ephemeral"},
        },
    ],
    messages=[{"role": "user", "content": "What is next on my calendar?"}],
)

Q2. Cost Difference: Cached vs Uncached Tokens

Pricing changes, so Lilith should keep model pricing in configuration and label estimates by date. Anthropic's public pricing model has separate prices for base input, cache writes, cache reads, and output. Cache writes cost more than base input, but cache reads are much cheaper.

Token TypeTypical Sonnet Public Pricing PatternImplication
Input tokensBase input rate per MTokPaid every uncached turn.
Cache writeHigher than base inputPaid when warming/changing cached prompt.
Cache readMuch lower than base inputMain savings for repeated Lilith system context.
OutputSeparate output rate per MTokUnaffected by prompt caching.

Example using the common Sonnet 5-minute cache pattern of $3/MTok input, $3.75/MTok cache write, $0.30/MTok cache read, and $15/MTok output: a 2,000-token system prompt over 1,000 turns costs about $6.00 uncached input. With one cache write plus 999 reads, it costs about $0.61, saving roughly $5.39 per 1,000 turns before output tokens.

Q3. Streaming with Anthropic SDK — Tool Call Deltas

Anthropic streaming emits typed events: message start, content block start, deltas, content block stop, and message stop. Text arrives as text deltas. Tool use arrives as a tool_use content block with incremental input_json_delta fragments. Lilith can stream text until a tool_use block is encountered; once a tool is requested, finish assembling the JSON, execute the tool, then send a follow-up tool_result message to resume generation.

Do not execute a partial tool call before the content block is complete. For conversational latency, keep tool schemas small and prefer read-only internal queries.

Q4. Rate Limits for Claude Sonnet

Anthropic rate limits vary by usage tier and are documented in the console/docs as requests-per-minute, input-tokens-per-minute, and output-tokens-per-minute. A single-user personal assistant is unlikely to hit paid-tier limits except during batch enhancement jobs. Lilith should isolate artifact/enhancement work behind a scheduler so conversational calls get first use of the user's quota.

The API returns HTTP 429 rate-limit errors and the SDK raises an API status/rate-limit exception. Lilith should retry background jobs with exponential backoff and show a short voice-safe error for conversational requests rather than spinning.

Q5. Provider Abstraction Layer

Use local adapters, not LiteLLM, for v6 v1. Anthropic prompt caching and streaming tool deltas are important enough that a lowest-common-denominator proxy adds risk. LiteLLM is useful for broad provider coverage, retries, and OpenAI-compatible normalization, but it introduces another dependency, another API surface, and possible feature lag for provider-specific capabilities. Anthropic does not provide a full OpenAI-compatible API as the primary integration path; use the official SDK.

from dataclasses import dataclass
from typing import Iterable, Protocol, Literal

@dataclass
class LLMChunk:
    type: Literal["text", "tool_call", "done", "error"]
    text: str = ""
    tool_name: str | None = None
    tool_input: dict | None = None

class LLMProvider(Protocol):
    name: str
    def count_tokens(self, messages: list[dict], system: list[dict] | None = None) -> int: ...
    def stream_conversation(self, *, system, messages, tools=None) -> Iterable[LLMChunk]: ...
    def complete_artifact(self, *, system, messages, tools=None, timeout_s: float | None = None) -> str: ...

Q6. Ollama for Local Inference

Ollama exposes its own REST API and an OpenAI-compatible endpoint at local base URLs, so Lilith can use either direct HTTP or the OpenAI SDK with a base_url override. Streaming responses are supported. For 8-20 GB VRAM, use small quantized models: Qwen2.5/3 7B, Llama 3.1/3.2 8B, Mistral 7B, or Phi-class models for fast conversational routing. First inference can take seconds to tens of seconds while the model loads; keep the model warm if Tier 1 conversational latency matters.

Q7. Token Counting for Anthropic

The Anthropic SDK/API includes message token counting. Lilith should count the exact conversational payload before sending and enforce a hard budget: system/cacheable blocks are separate from the volatile 1,200-token conversational context budget, while database-derived context must come only from L1/L2 enhanced data. If the count exceeds the budget, trim oldest conversational turns first, then low-priority context snippets.

Final recommendation: Build a small in-house adapter interface with provider-specific Anthropic and Ollama implementations. Do not adopt LiteLLM for v6 v1 because prompt caching, streaming tool deltas, and token counting should be controlled directly. No blocker found.
