RES-07 — python-keyring

Sprint: Research · Agent 1 · Report 7 of 7
Researched: June 28, 2026
Library version: keyring 25.7.0 (OS Keychain API)

Q1. Linux backends — what is available and what is the default?

On Linux, python-keyring interacts with multiple desktop-specific backends:

  • SecretService Backend (GNOME Keyring / KDE Wallet): Communicates via D-Bus with the Freedesktop.org Secret Service API. This is the **default and recommended** Linux backend.
  • KWallet Backend: Directly communicates with KDE Wallet via custom D-Bus bindings.
  • keyrings.alt (Plaintext Fallback): Stores passwords unencrypted in ~/.local/share/python_keyring/keyringrc.cfg. This is insecure and unacceptable for Lilith.
  • Out-of-the-Box Availability: SecretService is available by default on a fresh installation of Ubuntu Desktop (22.04 and 24.04 LTS) and all GNOME-based distributions. No configuration is required.
  • Forcing SecretService in Code: The active backend can be set programmatically during application boot: 
    import keyring
from keyring.backends.SecretService import Keyring as SecretServiceKeyring

# Force the SecretService backend
keyring.set_keyring(SecretServiceKeyring())

Q2. Headless Linux — what happens without a display or D-Bus session?

  • Failure Behavior: If Lilith runs headlessly (e.g., via SSH, server, or Docker container with no D-Bus session), python-keyring fails to load SecretService and falls back to PlaintextKeyring (if keyrings.alt is installed) or returns a fail.Keyring which throws exceptions on read/write.
  • Headless Detection & Safety: We must detect and block insecure fallback backends on startup, refusing to run if a secure keyring is unavailable: 
    import keyring
from keyring.backends.fail import Keyring as FailKeyring

def verify_keyring_security():
    active_backend = keyring.get_keyring()
    backend_name = type(active_backend).__name__
    
    # 1. Block keyrings.alt Plaintext fallback
    if "Plaintext" in backend_name or "keyrings.alt" in active_backend.__class__.__module__:
        raise RuntimeError("CRITICAL SECURITY ERROR: Insecure plaintext keyring detected. Refusing to start.")
        
    # 2. Block failed backends
    if isinstance(active_backend, FailKeyring):
        raise RuntimeError("CRITICAL SECURITY ERROR: No secure OS keychain backend is available on this system.")
        
    print(f"Secure keyring verified: {backend_name}")

Q3. macOS Keychain — user-facing behavior

  • User Authorization: If the application is **properly signed** (with a Developer ID Application certificate), macOS prompts the user for access **only on the first attempt** to read or write a secret. Selecting "Always Allow" whitelists the application binary, enabling silent background access on subsequent launches. Unsigned or self-signed apps will trigger security prompts repeatedly or fail.
  • Keychain Item Class: python-keyring maps values to the **Generic Password** (secGenericPassword) class in macOS.
  • Persistence: Stored credentials survive application reinstallation, updates, and macOS system upgrades.

Q4. Windows DPAPI / Windows Credential Manager

  • Windows Backend: Uses WinVaultKeyring which calls the Windows Credential Manager.
  • Security Properties: Windows Credential Manager encrypts credentials using the **Windows Data Protection API (DPAPI)**. DPAPI keys are derived directly from the user's Windows login credentials. This guarantees that credentials can only be decrypted when the user is logged into their Windows session.
  • Persistence: Stored secrets survive Windows updates, application updates, and user profile migrations.
  • Caveats: Entries are visible to users via the Windows Control Panel (under "Credential Manager"). A user can manually delete them, which the application must handle by prompting for re-encryption.

Q5. Size limits per keychain entry

  • macOS Keychain: Up to 16 MB per entry (effectively unlimited).
  • SecretService (Linux): No strict size limit (supports up to several MBs of payload).
  • Windows Credential Manager: Modern Windows OS supports up to **2,560 bytes** (defined by CRED_MAX_CREDENTIAL_BLOB_SIZE). Older legacy APIs had a limit of 512 bytes.
  • Lilith Credentials:
    • The 32-byte Database Master Key (64 characters hex) is well within limits on all platforms.
    • Google OAuth Refresh Tokens (100–200 characters) are well within the 512-byte and 2,560-byte limits on Windows.

Q6. Service name and username namespace

  • Namespace Convention: Use reverse domain name style for the `service` parameter (e.g., "com.squirrelthug.lilith") to prevent collisions.
  • Enumeration Limitation: The python-keyring API **does not support enumerating credentials** under a service name (e.g. no `list_credentials` exists). To clear credentials, Lilith must maintain a list of stored secret keys inside its encrypted local database and delete them sequentially: 
    # Delete a secret
try:
    keyring.delete_password("com.squirrelthug.lilith", "database_master_key")
except keyring.errors.PasswordDeleteError:
    pass # Already deleted
  • Missing Secrets: If a credential is missing, keyring.get_password() returns None. It does not throw an exception.

Verdict: CLEAR (Using python-keyring with safety guardrails) python-keyring is the correct architecture choice for secure local-first credential storage in Lilith v6. It uses the MIT license and is highly compatible across macOS, Windows, and Linux desktops.

Action Plan:
  1. Namespace all credentials under "com.squirrelthug.lilith".
  2. Add a **security verification wrapper** at boot to inspect keyring.get_keyring(), explicitly blocking PlaintextKeyring and preventing execution on headless Linux without a secure keyring daemon.
  3. Store the list of active credential keys inside the encrypted database to allow for a clean "reset/clear all credentials" loop, since keyring enumeration is not supported.
