Key Data Flows

Trace a message through the system — the diagram above shows components; these show movement.

Flow 1 — Conversational Voice Query
User: "What time is my dentist?"
  🎙 Microphone → VAD → STT (Whisper)
  → "What time is my dentist?"
  → Endpoint Manager tags: desktop, voice input
  → Classifier: CONVERSATIONAL (calendar.lookup intent)
  → CWM: loads [system prompt, conv history, calendar block]
  → Calendar block: L2 enhanced (today's events)
     staleness gate: data is 2h old — OK
  → LLM: 28-token response assembled
  → "Your dentist is at 2 PM."
  → TTS → speaker
  → UCL log: latency, tokens, tool used, mode=conv
Time to audio: < 2 seconds
Flow 2 — Artifact Mode (Restaurant Search)
User: "Find Italian restaurants near downtown for 6"
  🎙 Microphone → STT → Endpoint Manager
  → Classifier: ARTIFACT (tool.search intent)
  → Dispatcher: immediate acknowledge
  → TTS: "On it — looking for Italian options near downtown."
  → Background task starts: restaurant search tool
  → Tool runs (6 seconds)
  → L1 enhancement: rank results, add metadata
  → LLM: format handoff summary
  → TTS: "Found three. Osteria Marco fits 6 people..."
  → Chat screen: full artifact with comparison table
  → UCL log: artifact.started, tool.call, artifact.complete
Flow 3 — Data Sync Pipeline
Sync trigger (interval or user request)
  → Google Calendar API (OAuth token from OS keychain)
  → Pull: events in [-7d, +30d] window
  → Write: calendar_db (SQLCipher)
  → UCL log: sync.calendar.complete, records=847
  → L1 Enhancer runs (background job)
     → aggregates: events by day, participants, location
     → writes: calendar_l1_summary table
  → L2 Enhancer runs
     → writes: narrative blocks ("3 meetings tomorrow")
     → sets: staleness_ok = True
  → Next query: CWM fetches L2 block
     → staleness gate: passes (data is fresh)
     → LLM receives pre-computed summary, not raw rows
Flow 4 — Lilith Self-Debugs
User: "You've been slow lately."
  → Classifier: CONVERSATIONAL (system.introspection)
  → CWM: loads [system prompt, telemetry block]
  → Telemetry block: L2 system stats (pre-computed)
     → avg_response_ms: 4,200 (↑ from 1,800 yesterday)
     → slowest tool: get_calendar (p95: 8,200ms)
     → last_calendar_sync: 11:15 PM, pulled 847 events
  → LLM assembles diagnostic:
     "The slowdown started last night after a calendar sync
     that pulled 847 events. The get_calendar tool is
     running at 8 seconds for complex queries.
     I'll run a targeted sync to rebuild the summary tables."
  → tool: calendar_sync.targeted_rebuild()
  → Latency returns to baseline
  → UCL logs: self_debug.introspection event

Scale Reality — The Backend is Heavy

The architecture diagram is drawn bottom-heavy intentionally. The frontend is a thin UI layer over a substantial backend. Here's what "bottom-heavy" means in practice:

Frontend (thin layer)
7 screens + orb
Reads data from backend via IPC
Sends events, receives responses
~10% of total codebase
Processing Layer (core)
Voice pipeline · Classifier · CWM
Narrative layer · Formatters
Tool implementations · Sync
~45% of total codebase
Data Layer (heaviest)
6 SQLCipher databases
L1/L2 enhancement jobs
UCL telemetry schema
~45% of total codebase

This is the right shape for Lilith. A thin, beautiful UI over a deep data and intelligence backend. The frontend doesn't do computation — it displays what the backend has already computed. The LLM doesn't touch raw data — it reads what the enhancement layers have already prepared. The enhancement layers run continuously in the background so query time is fast.
