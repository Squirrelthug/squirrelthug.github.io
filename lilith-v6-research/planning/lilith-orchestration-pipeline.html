The friction points in the current workflow
  1. You still have to manually transcribe speech → x.txt → prompt files (or do it yourself in text)
  2. Monitoring three terminals simultaneously requires attention that could be automated
  3. The post-agent review (diff/check/build) is manual — you can catch errors but it's time-consuming
  4. Task assignment to agents is manual — you decide who gets what
  5. There's no automated "did this break anything" check before the merge

The collapsed-code view for watching agents

You want to see agents working but not be flooded with code. The right monitor setup for the current workflow:

  • Left/right terminals show agent narrative output, not code diffs (pipe through a filter or use agent's status.txt)
  • Code diffs go to the center terminal on request only (git diff when you want to check)
  • Agents write a one-line status update every N minutes to their status.txt — center terminal tail -f both
# Center terminal: watch both agents without reading code

2. The Path: Lilith Becomes the Orchestrator

The long-term vision: you speak to Lilith. She understands the sprint, breaks the work down, assigns it to agents, monitors their progress, reviews the output, runs the build, and tells you what happened. The 3-terminal setup becomes a UI that Lilith manages. You watch from outside.

  CURRENT WORKFLOW                    TARGET WORKFLOW

  You speak                           You speak
      │                                   │
      ▼                                   ▼
  x.txt updated                       Lilith transcribes (STT)
      │                                   │
  You write prompts          ─────→   Lilith writes agent prompts
      │                                   │
  Agents work                         Agents work
      │                                   │
  You monitor terminals               Lilith monitors status.txt
      │                                   │
  You review diff                     Lilith reviews diff (LLM)
      │                                   │
  You run build/tests                 Lilith runs build/tests
      │                                   │
  You merge/push             ─────→   Lilith reports to you:
                                      "Both agents done. Tests pass.
                                       3 files changed. Ready to merge?"
                                          │
                                      You say "merge it"
                                          │
                                      Lilith merges and pushes

This is the pair programmer vision — the thing that makes Lilith the pair programmer you want. The orchestration logic already has a shape: read sprint doc → extract tasks → assign to agents → monitor → review → build → report. All of this can be a Lilith tool chain.

Lilith's orchestration tools (what gets built)

Why this is the right project to build Lilith's pair programmer capability
You understand this workflow deeply because you built it. The tooling maps naturally to Lilith's existing architecture (tools → LLM → decision → next action). And it creates a feedback loop: Lilith helping you build Lilith creates telemetry on what a coding orchestrator actually needs. The tools you build for managing agents in this project become the template for managing any multi-step autonomous process.

3. Windows Build Toolchain — What "Deliberate Attention" Actually Means

The previous document said the Windows build toolchain requires "deliberate attention." That's abstract. Here's the concrete reality.

What the toolchain problem actually is

On Linux, building a Python app is simple: Python is first-class, system libraries install via apt, C extensions compile cleanly with gcc. The entire stack is POSIX and assumes Linux is the default.

On Windows, several things break this assumption:

Problem 1: C extension compilation

Several Python packages you likely use have C extensions (native code that compiles when you pip install). On Linux this uses gcc (already installed). On Windows it requires MSVC (Microsoft Visual C++ build tools). The CI runner must have these installed, and some packages behave differently between their Linux and Windows compiled versions.

# What works on Linux
pip install cryptography   ← uses prebuilt wheel, no problem

# What might fail on Windows CI without MSVC
pip install some-audio-package  ← needs to compile C extension
                                ← fails if build tools not installed

# Fix in CI runner (GitHub Actions Windows)
- uses: microsoft/setup-msbuild@v1
- run: pip install --prefer-binary package-name
     # --prefer-binary uses prebuilt wheel if available, avoids compilation

Problem 2: Path separators

Every hardcoded path in your Python code using / or string concatenation is a potential Windows bug. The fix is already known: use pathlib.Path everywhere. But any path that was written as a string with slashes will silently break on Windows.

# Wrong (breaks on Windows)
config_file = "~/.config/lilith/config.json"
log_path = base_dir + "/logs/" + name + ".log"

# Right
from pathlib import Path
config_file = Path.home() / ".config" / "lilith" / "config.json"
log_path = Path(base_dir) / "logs" / f"{name}.log"

Problem 3: File system case sensitivity

Linux file systems are case-sensitive. Windows is not. A file named Config.py and config.py are different on Linux, the same on Windows. Any import that relies on exact case matching can fail silently — or worse, import the wrong file. The CI runner will catch this if it's a real problem.

Problem 4: Process spawning

Python's subprocess and os.fork() behavior differs on Windows. Multiprocessing uses "spawn" by default on Windows (vs "fork" on Linux), meaning child processes don't inherit parent state. Any code that relies on forked state will need a different approach.

Problem 5: SQLCipher on Windows

SQLCipher requires a native C library. On Linux it's a system package. On Windows it must be bundled. The sqlcipher3 Python package has prebuilt Windows wheels but they must be verified. This is a one-time setup cost, not ongoing friction.

The actual workflow for handling this

  YOU WRITE CODE ON LINUX (normal dev workflow)
             │
             ▼
  Push to GitHub
             │
             ▼
  GitHub Actions: Linux CI runner (fast, verifies correctness)
             │
             ▼
  GitHub Actions: Windows CI runner (parallel)
             │
             ├── If Windows CI passes → no Windows-specific issue
             │
             └── If Windows CI fails →
                     Read CI log → identify which package/path failed
                     Fix it → push → both CIs green → done

In practice, most Windows CI failures are one of the five problems above. They're fixed once and don't recur. The first pass of getting the app running on Windows CI probably takes 2–3 days of fixing. After that, it's green and stays green as long as you avoid re-introducing path string hacks or OS-specific assumptions.

4. Windows Audio Testing — VM vs Real Hardware

The question: can you test audio input/output on Windows in a virtual machine, or do you need actual Windows hardware?

What Windows audio actually is

Windows has three audio driver models, and which one you're talking to affects how Lilith's audio pipeline works:

Driver modelWhat it isLatencyUsed for
WASAPI
(Windows Audio Session API)		Microsoft's current standard. Two modes: shared (normal apps) and exclusive (low-latency apps)Shared: ~10–30ms
Exclusive: ~1–5ms		Everything modern including Python sounddevice
WDM/KS
(Kernel Streaming)		Older low-level driver access. Rarely needed by app developers.~1–3msPro audio DAWs, not Lilith
MME / DirectSoundLegacy Windows audio. Still present for compatibility. High latency.~50–150msOld apps only

For Lilith, WASAPI shared mode is the target. That's what PyAudio and sounddevice use by default on Windows.

What a VM can and cannot do

VMware / VirtualBox / Hyper-VReal Windows Hardware
Audio output (speakers)Works — VM routes to host speakers via virtual audio deviceWorks
Audio input (microphone)Partial — can forward host mic, but with added latency and potential dropout. USB microphone passthrough is more reliable than integrated mic.Works natively
WASAPI shared modePartial — the VM presents a virtual audio device. It behaves like WASAPI shared but is actually a layer of abstraction. Functional for testing, not identical to real hardware.Real WASAPI
WASAPI exclusive modeDoes not work — VM cannot grant exclusive hardware accessWorks
Bluetooth audioVery limited — BT passthrough in VMs is unreliable; BT drivers rarely work correctlyWorks natively
UI and app behaviorWorks — no difference for UI testingWorks
Installer testingWorks — clean VM is ideal for testing first-run installsWorks
Bottom line on VM audio testing

A VM is good enough for initial development and most functional testing. You can verify that the Python audio pipeline connects to Windows audio, that STT input works, that TTS output plays. The things that won't be testable in a VM:

  • Real latency (VM audio has added overhead)
  • Bluetooth audio devices
  • WASAPI exclusive mode (not needed for Lilith at this stage)
  • Microphone array or noise cancellation hardware behavior

For pre-launch validation you want one real Windows machine (or access to one). For development, a VM will get you 90% of the way there.

Practical recommendation for audio on Windows

  DEVELOPMENT PHASE
  Use VM (VMware/VirtualBox on Linux) for:
  ├── UI development and testing
  ├── Installer testing
  ├── Python pipeline smoke testing
  └── Initial audio I/O verification (using USB mic passthrough)

  PRE-LAUNCH PHASE
  Borrow or buy a cheap Windows laptop for:
  ├── Real audio device behavior
  ├── Microphone quality under real WASAPI
  ├── Bluetooth audio pairing
  ├── Full performance profiling
  └── Real-user first-run experience

  Budget: $200–400 used/refurbished Windows laptop is enough
  for hardware-accurate testing. Not a development machine, just
  a test device.

5. CI Runners Explained

GitHub Actions gives you managed virtual machines — "runners" — that run your code. You tell GitHub what kind of machine you want and what to do on it. The key insight: you don't own these machines, you don't configure them, you don't maintain them. GitHub spins one up for each job, it runs, it shuts down.

  YOUR GITHUB REPO
        │
        │ push to main
        ▼
  GitHub Actions triggered
        │
        ├─────────────────────────────────┐
        │                                 │
        ▼                                 ▼
  Ubuntu runner (Linux)           Windows runner
  ubuntu-latest                   windows-latest
        │                                 │
  pip install requirements         pip install requirements
  python -m pytest                 python -m pytest
  build AppImage                   build Windows .exe
        │                                 │
  ✓ Tests pass / ✗ fail            ✓ Tests pass / ✗ fail
        │                                 │
        └──────────────────┬──────────────┘
                           │
                     All green → merge allowed
                     Any red → PR blocked

The GitHub Actions config file (`.github/workflows/ci.yml`) controls all of this:

name: Build and Test
on: [push, pull_request]

jobs:
  test-linux:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt
      - run: timeout 180 python -m pytest tests/

  test-windows:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install --prefer-binary -r requirements.txt
      - run: python -m pytest tests/

  test-macos:
    runs-on: macos-latest          # Apple Silicon runner available
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt
      - run: python -m pytest tests/

Free tier: GitHub provides 2,000 minutes/month on free accounts, 3,000 on Pro. Linux runners use 1× multiplier, macOS uses 10× (more expensive), Windows uses 2×. For the current stage of development, you'll likely stay well within the free tier.

What you don't need a Windows machine for
Building the Windows .exe happens on the Windows CI runner. You never need to boot Windows yourself to produce a Windows binary. Your entire development workflow stays on Linux. The CI runner is your Windows machine.

6. Hardware Capability Detection & Routing Tiers

The concept: Lilith boots on different hardware and detects what she can do locally before deciding what to send to the cloud. The hardware detection happens once at startup and produces a capability profile that routes all subsequent decisions.

  STARTUP SEQUENCE

  Lilith starts
       │
       ▼
  Hardware detection (< 1 second)
  ├── GPU: detected via nvidia-smi / Metal / DirectX
  │         → VRAM: how much?
  ├── CPU: core count, architecture (x86 / ARM)
  ├── RAM: available system memory
  └── Audio: microphone present? Speaker present?
       │
       ▼
  Capability profile written to session state
       │
       ▼
  Route all decisions based on tier

Capability tiers

Tier 0 — Cloud-Only (No Local Inference)
MacBook Air M1 with no GPU · low-RAM devices · any machine without a dedicated GPU
  • STT: Whisper tiny (CPU — fast on any hardware) ✓
  • TTS: Kokoro or similar lightweight model (CPU) ✓
  • LLM: Cloud only (Claude API, OpenAI, etc.)
  • Personality model: Not available → falls back to baked-in system prompt
  • Local tools: All tools available (they're Python, not GPU-dependent)
  • Voice pipeline: Fully functional, cloud-routed inference
Tier 1 — Hybrid (Local STT/TTS + Optional Local LLM)
RTX 3060 (12GB VRAM) — current dev machine · or equivalent 8GB+ VRAM
  • STT: Whisper small or medium (GPU-accelerated) ✓ — noticeably faster than Tier 0
  • TTS: Higher quality model (GPU) ✓
  • LLM: Can run a 7B quantized model locally (e.g., Mistral-7B-Q4) if needed, but still defaults to cloud for quality
  • Personality model: Can run 3B–4B personality model when PERS-2 is implemented
  • Memory: Enough VRAM to run STT + personality simultaneously
Tier 2 — Local-First (High-VRAM Workstation)
RTX 4090 24GB · or 48GB+ VRAM · future hardware
  • STT: Whisper large-v3 (GPU) ✓ — near-perfect transcription
  • TTS: High-fidelity neural TTS (GPU) ✓
  • LLM: 13B–34B quantized model locally — approaches cloud quality for many tasks
  • Personality model: Full 7B dedicated personality model
  • Privacy: Conversation can be fully local, no cloud required

The detection code (simplified)

# modules/startup/hardware_detect.py

def detect_hardware_tier() -> HardwareTier:
    vram_gb = _get_vram_gb()   # nvidia-smi, or Metal on macOS, or 0 if no GPU
    ram_gb = psutil.virtual_memory().total / (1024**3)
    cpu_cores = psutil.cpu_count(logical=False)

    if vram_gb >= 20:
        return HardwareTier.TIER_2_LOCAL_FIRST
    elif vram_gb >= 8:
        return HardwareTier.TIER_1_HYBRID
    else:
        return HardwareTier.TIER_0_CLOUD_ONLY

def _get_vram_gb() -> float:
    try:
        result = subprocess.run(
            ["nvidia-smi", "--query-gpu=memory.total", "--format=csv,noheader,nounits"],
            capture_output=True, text=True, timeout=3
        )
        return float(result.stdout.strip()) / 1024
    except Exception:
        return 0.0  # No NVIDIA GPU, or nvidia-smi not available

The tier isn't just for the personality model — it changes behavior throughout the system:

7. MacBook M1 Considerations

The M1 MacBook is currently used as a Mac node/thin client (Sprint ND-1). Its capability profile matters for understanding the routing decisions.

M1 vs. the 3060 desktop

CapabilityRTX 3060 (Linux desktop)MacBook M1 (16GB unified)
VRAM for inference12GB GDDR6 discrete~7–10GB usable from 16GB unified (shared with CPU)
STT (Whisper)small/medium via CUDAtiny/small via Metal (MPS) — respectable, slower than CUDA
TTSGPU-acceleratedCPU/Metal — functional, slightly higher latency
Local LLM7B Q4 comfortably7B Q4 via Metal, possible but warm/throttled on extended use
Hardware tier classificationTier 1 (Hybrid)Tier 0–1 boundary (treat as Tier 0 for safety)
Best role for LilithPrimary workstation — all featuresThin client (ND-1) — sends audio to Linux master for processing

The M1 as a development machine: possible but adds friction. The PyTorch Metal backend (MPS) works but is less stable than CUDA, some packages don't have Metal-accelerated versions, and the cross-compilation story for producing Windows binaries gets complicated. The Linux dev → CI-for-all-platforms approach remains the right architecture regardless of whether you occasionally use the MacBook for coding.

If you ever develop on the MacBook temporarily
The Python backend will run fine. CUDA-specific calls fail — always guard with torch.cuda.is_available() and fall through to CPU/MPS. The CI pipeline still handles building for all three platforms regardless of which machine you write code on. The constraint is that you can't run the full local inference stack on M1, so development that depends on GPU-accelerated STT/TTS needs to test against the Linux machine (or accept that audio testing on M1 uses CPU-speed models).
