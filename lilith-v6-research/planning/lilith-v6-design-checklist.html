Foundation Layer
01
Project Structure & Conventions
The ground rules every file in the project follows. Must be decided before the first file is written so agents don't drift.
  • Root directory layout: exact top-level directories named and their purpose stated. No guessing — every directory has a declared role. Standard
  • Path handling: pathlib.Path everywhere, no string concatenation for paths, no hardcoded separators. Rule applies to ALL file I/O, all subprocess calls, all DB paths. No exceptions. Standard
  • Python version: which exact version (3.11 / 3.12 / 3.13), why, and how it's enforced (.python-version file, pyproject.toml). Decision needed
  • Package management: pip + requirements.txt or uv or poetry — pick one, explain the choice, document how to add a dependency. Decision needed
  • Naming conventions: files (snake_case), classes (PascalCase), constants (UPPER_SNAKE), database tables (snake_case), event type names (dot.namespaced). All stated explicitly.
  • Module boundary rules: which modules may import from which. Circular imports are a recurring failure mode — define the dependency direction at project start.
  • Error handling philosophy: where errors are caught vs propagated, what constitutes a recoverable vs unrecoverable error, what gets logged vs what gets surfaced to the user.
  • Logging to stdout vs UCL: clear rule for which events go to the UCL database vs which go to stdout/stderr. UCL gets behavioral events; stdout gets startup/debug only.
  • Configuration management: how app config (not credentials) is stored and read. What format (TOML/JSON/YAML), where the file lives per OS, what the defaults are, how schema changes are handled.
  • Git workflow: branch naming convention, PR requirement for all merges, no direct commits to main, commit message format. Stated so agents follow it.
  • Sprint task done-criteria standard: the baseline that every task inherits — telemetry event on every write path, security review, test coverage. Written here, referenced by every sprint doc.

02
Security & Credential Management
How secrets are stored, accessed, and never exposed. Must be the first implementation concern — retrofitting security is how v5 accrued debt.
  • OS keychain library: python-keyring as the abstraction — confirm it covers libsecret (Linux), Keychain (macOS), DPAPI (Windows) with the same API. Document the service name and key naming convention used throughout the app.
  • What counts as a credential: explicit list — API keys, OAuth tokens, database passphrase, user password. Everything on this list goes to keychain. Everything not on this list is not a secret.
  • Credential access pattern: how the app retrieves a key at runtime — the exact call sequence, where it happens in startup, what the failure behavior is if keychain is unavailable.
  • .env file policy: .env is for local development only, never committed, never read in production. What replaces it in production (keychain). How CI/CD accesses secrets it needs (GitHub Secrets, not .env).
  • OAuth token lifecycle: how access tokens are acquired, stored (keychain), refreshed on expiry, and revoked when the user disconnects a service. Token never written to disk as plaintext.
  • What never leaves the device: explicit list of data categories that are never sent anywhere unencrypted. Contacts, conversation content, finance data — state this explicitly so agents don't accidentally send it.
  • LLM call data hygiene: what is and isn't sent in the context window to cloud LLMs. Raw PII handling — names, addresses, account numbers — and how they're handled before being included in a cloud-bound prompt.
  • Audit trail for credential access: does the app log when it reads from keychain? Should it? Decision stated either way.

03
Encryption at Rest — SQLCipher
Every database is encrypted from the first migration. Not added later. This section defines the exact technical approach so all six databases are consistent.
  • Python package choice: sqlcipher3 vs pysqlcipher3 vs apsw with SQLCipher — pick one, confirm it has pre-built wheels for all three target platforms, document the install command.
  • Key derivation: how the SQLCipher encryption key is derived. User passphrase → PBKDF2 or Argon2 → key. Which algorithm, which parameters, where the salt is stored. Decision stated with rationale.
  • Passphrase storage: the derived key or the raw passphrase stored in keychain — which one and why. What the user must provide on each app start vs what is auto-retrieved.
  • Database file locations per OS: exact paths using OS-appropriate conventions — XDG_DATA_HOME on Linux (~/.local/share/lilith/), ~/Library/Application Support/Lilith/ on macOS, %APPDATA%\Lilith\ on Windows. All stated explicitly.
  • Which databases exist: complete list — calendar_db, contacts_db, finance_db, memory_db, conversation_db, ucl_db. All SQLCipher, no plaintext SQLite anywhere.
  • Schema migration strategy: how schema changes are applied without data loss. Migration runner design — numbered migrations, applied on startup, idempotent. No ad-hoc ALTER TABLE in code.
  • Connection management: one connection per database or a pool — which, and how connections are opened, kept alive, and closed. Thread safety requirements.
  • Backup strategy: how the encrypted databases are backed up. VACUUM INTO for online backup. When it runs, where backups go, how many are kept.
  • First-run key setup: what happens on first launch — passphrase prompt, key derivation, database creation. What happens if the user forgets their passphrase (data is unrecoverable — is this stated clearly in onboarding?).

Cross-Platform Foundations
The rules that make the same codebase run on Linux, macOS, and Windows without platform-specific branches scattered through the code.
  • Platform detection: how and where the OS is detected — once at startup, stored in session state, accessed via a platform module. Never detected ad-hoc in feature code.
  • OS-specific path roots: the canonical function that returns the right base path per OS for data, config, logs, and cache. Called by all modules — never hardcoded per-module.
  • Platform plugin system: how OS-specific capabilities (keychain backend, audio API, notification system) are loaded at startup based on detected OS. The interface each plugin must implement.
  • C extension handling: list of packages with C extensions that need special handling on Windows (--prefer-binary in CI). Process for validating new dependencies cross-platform before adding them.
  • Process spawning: any code that uses subprocess, multiprocessing, or os.fork must note the Windows behavior difference (spawn vs fork). Decision on how multiprocessing is used (if at all).
  • File system case sensitivity: policy for file and import naming — always lowercase filenames, always exact-case imports. Enforced by linter or CI check.
  • Line endings: .gitattributes policy. LF everywhere, Windows CI runner converts on checkout.
  • Packaging per platform: AppImage (Linux), .dmg or .pkg (macOS), NSIS installer or .msix (Windows). Which tools produce each. When in the CI pipeline they're built.

Infrastructure Layer
05
Hardware Detection & Capability Tiers
Runs once at startup. Determines which local models can run, how routing decisions are made, and what features are available. The rest of the system reads from this result — never re-detects.
  • Detection method per platform: nvidia-smi for NVIDIA GPU VRAM (Linux/Windows), Metal device query for Apple Silicon, psutil for CPU and RAM. Exact library/command for each, timeout behavior if detection hangs.
  • Tier definitions (complete): Tier 0 (no GPU / low VRAM), Tier 1 (8–20GB VRAM), Tier 2 (20GB+ VRAM) — exact thresholds stated. What changes at each tier for STT, TTS, LLM, personality model.
  • Capability profile schema: the exact data structure written to session state — which fields, which types. All downstream consumers read this structure.
  • Where tier is stored: session state only (not persisted to DB). What happens if detection fails — safe default (Tier 0 / cloud-only) stated explicitly.
  • How tier affects routing: explicit mapping — Tier 0 routes STT/LLM/TTS to cloud, Tier 1 runs STT and TTS locally with cloud LLM, Tier 2 adds local LLM option. This mapping is the single source of truth.
  • User visibility: does the user see their detected tier? Where? Can they override it manually? Decision stated.
  • UCL telemetry event: what gets logged at startup — detected GPU, VRAM, tier assigned, models selected as a result. Event name and payload fields.

Process Lifecycle & Recovery
How Lilith starts, stays alive, restarts after a crash, and shuts down cleanly. The appliance behavior — it just works, silently.
  • Startup sequence order: the exact ordered steps — hardware detect → keychain open → database open → config load → platform plugins load → voice pipeline init → ready. No step may be skipped or reordered.
  • Process supervisor per platform: systemd unit file (Linux), launchd plist (macOS), Windows Service or Task Scheduler (Windows). Which approach for v6 v1, with the others noted as future.
  • Crash recovery: what constitutes a crash vs a clean exit. How the supervisor restarts the process. Backoff strategy for repeated crashes (don't loop-crash infinitely).
  • Graceful shutdown: what must be flushed or closed before exit — DB connections, audio streams, in-flight LLM calls. SIGTERM handler behavior.
  • Log file management: where logs are written, rotation policy (size or time based), how many old files kept. Log format (structured JSON vs plain text).
  • State that survives restart: what is persisted to DB (durable) vs what is reconstructed fresh on each start (session state, capability tier, active conversation context).
  • Update mechanism: how a new version replaces the running one — AppImage swap on Linux, pkg replace on macOS. Whether updates are applied while running or require restart. Who initiates the update.
  • First-run vs subsequent start: how the process knows it's running for the first time (flag in DB or config). Onboarding flow gated on this flag.

CI/CD Pipeline
GitHub Actions configuration. What runs automatically on push and merge. The machine that ensures the codebase is always in a releasable state on all three platforms.
  • Trigger events: which GitHub events fire the pipeline — push to main, PR open, PR update. What runs on each event (not everything needs to run on every push).

  • Windows-specific install flags: --prefer-binary for packages with C extensions. List of packages known to need this. How to detect new ones that do.
  • Test execution: how pytest is invoked in CI, timeout wrapper (no stalling CI), which test markers run vs which are skipped (integration tests that need real hardware excluded from CI).
  • Build artifacts: which jobs produce a binary (AppImage, .dmg, .exe), where they're uploaded (GitHub Releases or Actions artifacts), version numbering scheme.
  • Secrets in CI: which secrets are needed (API keys for integration tests?), how they're stored (GitHub Secrets), how they're injected into the CI environment without appearing in logs.
  • Failure behavior: does a Windows CI failure block merge? Should it for v6 v1 (Linux primary)? Policy stated explicitly so there's no ambiguity when it happens.

08
Testing Standards
What must be tested, how, and what is not worth testing. Prevents both the "no tests" failure and the "mocked everything" failure from v5.
  • No DB mocking policy: integration tests hit real SQLCipher databases — in-memory or temp file, but real SQLCipher. No mocking the database layer. Mocked DB tests caught v5 bugs too late. Standard
  • Test database setup: how a clean test DB is created and torn down for each test or test session. Fixture design (pytest fixtures). Key derivation in tests (fixed test passphrase, not random).
  • What requires a test: every new tool, every database write function, every enhancement layer job, every sync function. Unit tests for pure functions. Integration tests for anything touching a DB or external API.
  • External API testing: LLM calls, OAuth flows, Google API calls — how these are tested without hitting real APIs in CI. VCR cassettes, response fixtures, or skip markers with real-API test run separately.
  • Voice pipeline testing: STT and TTS cannot run in headless CI without audio hardware. Policy: unit test the pipeline logic, mark audio I/O tests as hardware-only, exclude from CI.
  • Test file organization: where tests live relative to the code they test (tests/ mirroring modules/ structure or colocated). Import convention (absolute imports in tests).
  • Test timeout: per-test timeout (pytest-timeout), suite-level timeout (wrapping pytest call). Values stated — not left as defaults.
  • Coverage requirements: minimum coverage threshold enforced in CI? Which paths are excluded from coverage (third-party wrappers, platform-specific branches that can't run on CI runner)?

Audio I/O & Platform Audio APIs
The hardware interface layer for microphone input and speaker output. Platform-specific below this point; everything above treats audio as streams.
  • Python audio library: sounddevice vs PyAudio vs soundcard — pick one, confirm it supports WASAPI on Windows, ALSA/PulseAudio on Linux, CoreAudio on macOS. Document why.
  • Audio device selection: how the app selects input and output devices at startup. Default device vs user-configured. What happens when the configured device is disconnected.
  • Microphone capture format: sample rate, bit depth, mono vs stereo, chunk size for streaming. These values affect STT model performance — chosen to match Whisper's expected input.
  • VAD (Voice Activity Detection): library choice (Silero VAD vs WebRTC VAD vs energy threshold). Threshold tuning. What constitutes end-of-utterance. How VAD output feeds into STT.
  • Audio buffer and latency management: target end-to-end audio latency budget by tier. How buffers are sized. What causes latency spikes and how they're mitigated.
  • Platform audio API per OS: WASAPI shared mode (Windows), CoreAudio (macOS), PulseAudio/PipeWire (Linux) — which is targeted for v6 v1, what the fallback is.
  • Amplitude signal: how microphone amplitude is exposed to the orb for real-time visual reaction. Which value (RMS, peak), at what sample rate, over what channel.
  • UCL telemetry: what audio events are logged — device selected, dropout detected, latency spike, sample rate mismatch.

Speech-to-Text (STT)
Converts microphone audio to text. Model choice and inference path are tier-dependent. Output feeds directly into the router.
  • Model selection per tier: exact model names — Tier 0: whisper tiny (CPU), Tier 1: whisper small or medium (CUDA), Tier 2: whisper large-v3 (CUDA). Language setting (auto-detect vs English-locked).
  • Inference library: openai-whisper vs faster-whisper vs whisper.cpp Python bindings — pick one for each tier or one library that handles all tiers. Confirm CUDA support for Tier 1/2.
  • Model loading: when the model is loaded (startup vs first use), where model weights are cached (OS-appropriate path), how loading time is handled UX-wise.
  • Streaming vs batch transcription: does STT run on a fixed audio chunk after VAD detects end-of-utterance, or is it a streaming transcription? Tradeoff stated, decision made.
  • Output format: what the STT module returns — plain text string, or text with timestamps/confidence. What downstream components expect.
  • Failure handling: what happens when transcription fails or returns empty string. Silent failure vs retry vs user notification.
  • UCL telemetry: latency from audio capture end to text output, model used, character count output, confidence score if available.

11
Text-to-Speech (TTS)
Converts LLM text output to audio. Must be fast enough to meet the conversational mode latency budget. Quality and speed are tier-dependent.
  • Model selection per tier: Tier 0: lightweight CPU model (Kokoro or piper-tts), Tier 1: mid-quality GPU model, Tier 2: high-fidelity GPU model. Exact model names or packages.
  • Streaming TTS: does TTS begin playing before the full text is generated (streaming), or does it wait for the complete response? Streaming is critical for conversational mode latency — decision and rationale.
  • Voice selection: which voice(s) are available, how voice is configured per user, whether voice can change based on context (e.g., Lilith's personality modes).
  • Amplitude output for orb sync: how TTS audio amplitude is extracted in real time and sent to the orb. This must be a low-latency signal, not post-processed.
  • Interruption handling: what happens when the user speaks while Lilith is speaking. Stop immediately vs finish current sentence vs ignore. Policy stated.
  • Endpoint-aware output: how TTS knows whether to play to local speakers vs send audio stream to a remote thin client (kitchen speaker). The output routing decision.
  • UCL telemetry: latency from text input to first audio byte, model used, character count input.

Intelligence Layer
12
LLM Integration & Provider Abstraction
The layer that talks to inference models. Must be provider-agnostic so cloud models can be swapped without touching the rest of the system. BYOK is a core requirement.
  • Provider abstraction interface: the exact methods every provider must implement — complete(messages, tools, stream) at minimum. No provider-specific code above this interface.
  • Supported providers for v6 v1: which providers ship in vertical slice — Anthropic Claude (primary), OpenAI (secondary), local via ollama or llama.cpp. Others noted as future.
  • BYOK implementation: API keys retrieved from keychain at provider instantiation, never stored in code or config. Key validation on startup (light API call to confirm key works).
  • Default model per provider: which specific model string is used for each provider in v6 v1. How model choice is overridden by user config.
  • Prompt caching: Anthropic prompt caching — which parts of the context are marked as cacheable (system prompt, stable context blocks). Cache TTL awareness. Whether other providers have equivalent feature.
  • Streaming vs non-streaming: which call paths use streaming (conversational mode for low latency) and which use non-streaming (artifact mode where full response is needed before proceeding).
  • Error handling and retry: rate limit (429) backoff strategy, timeout values, model unavailable fallback chain. Maximum retry count before surfacing error to user.
  • Token counting: how tokens are counted before sending (to enforce context budget) and after receiving (for cost tracking). Which library (tiktoken, Anthropic's counter, provider-reported).
  • UCL telemetry: every LLM call logged — provider, model, prompt tokens, completion tokens, latency, cache hit/miss, cost estimate in USD, success/error.

13
Two-Mode Router
Every input is classified as conversational or artifact before anything else happens. This classification gates the entire downstream pipeline. The most load-bearing architectural decision in v6.
  • Classifier implementation: how classification happens — small LLM call, keyword/pattern matching, embedding similarity, or a combination. Latency budget for the classification step itself.
  • Classification output schema: what the classifier returns — mode (conversational/artifact), intent labels (calendar.lookup, people.search, etc.), confidence score. This schema is the contract between routing and downstream systems.
  • Conversational mode constraints (hard limits): maximum output tokens (50), maximum time-to-first-audio-byte (3 seconds), maximum context tokens (1,200). These are ceilings, not targets. How they're enforced.
  • Artifact mode flow: immediate acknowledge → background task dispatch → bridge timer → handoff. What happens at each step, what triggers the handoff, what happens if the task exceeds the bridge timer.
  • Ambiguous query handling: when confidence is below a threshold, what happens — default to conversational (safer/faster) or ask for clarification. Threshold value stated.
  • Intent taxonomy: complete list of intent labels the classifier can output. This list is the contract — CWM uses it to select context blocks, tool system uses it to decide which tools to surface.
  • Multi-intent handling: a query with two intents (calendar AND people) — which mode wins, which context blocks load, how the response handles both.
  • UCL telemetry: every classification logged — input text hash (not raw text), mode selected, intent labels, confidence, classifier latency.

14
Context Window Manager
Assembles the context that goes into each LLM call. Wrong context = wrong answer. Unnecessary context = slow, expensive calls. This is the efficiency engine of the system.
  • Intent-to-block mapping (complete): for every intent label in §13's taxonomy, which context blocks are loaded. Must be a complete table — no unmapped intents. Open
  • Block catalog: complete list of all context blocks — system prompt, personal truths, conversation history, calendar, contacts, finance, memory recall, telemetry stats, pending tasks. Each block's source, size budget, and staleness policy.
  • Token budget per block: how many tokens each block is allowed to consume. What happens when a block's data exceeds its budget (truncate, summarize, or paginate).
  • Block loading behavior: blocks not in the whitelist for the current intent are NOT fetched — not fetched and discarded, not fetched. The DB query never happens. This is critical for conversational mode latency.
  • Conversational mode context ceiling: 1,200 tokens total — how this is enforced across all blocks. Which blocks get priority when budget is exceeded.
  • Conversation history compression: when does history get summarized vs truncated, how many raw turns are kept before compression, what the summary format is.
  • Staleness gate integration: how blocks check freshness before injection. What "stale" means per block (calendar: 4 hours, contacts: 24 hours, etc.). What happens in the context when a block is stale (note injected, or block withheld).
  • Block assembly order: the order blocks appear in the context matters for LLM attention. Defined ordering stated — system prompt first, stable context, then dynamic blocks, then history last.
  • UCL telemetry: which blocks were loaded, token count per block, total tokens assembled, blocks skipped due to intent filter, staleness flags triggered.

15
Tool System
How tools are registered, called, and their results returned to the LLM. The interface between the LLM and all data/action capabilities.
  • Tool registration: how tools are discovered and registered at startup. Auto-discovery from a directory vs explicit registration list. Which approach and why.
  • Tool schema format: how each tool's input schema is defined (JSON Schema). Whether the schema is in code or in a separate YAML/JSON file. How it's validated.
  • Tool result format: what every tool returns — success/error flag, payload, metadata (latency, source, freshness). The LLM always receives a structured result, never raw exception text.
  • Tool call loop: how the multi-step tool call loop works — LLM requests tool, tool runs, result injected, LLM continues. Maximum tool calls per request. How to detect and break infinite tool loops.
  • Tool authorization: which tools require explicit user confirmation before running (any tool that writes data, deletes data, or sends a message). Which run silently. Policy stated and enforced in the tool runner, not per-tool.
  • Blocking vs non-blocking tools: tools that return immediately vs tools that kick off a background job (the artifact mode tools). How the router knows which type a tool is.
  • Tool availability by tier: which tools are unavailable at certain hardware tiers (personality model tool only at Tier 1+). How the LLM's tool list is filtered before each call.
  • Error handling: tool throws an exception — what gets returned to the LLM, what gets logged to UCL, whether the LLM is told to retry or to report failure to user.
  • UCL telemetry: every tool call logged — tool name, input hash, latency, success/error, called in mode (conversational/artifact).

Data Layer
16
UCL Telemetry
The nervous system of the entire application. Every behavioral event goes here. Lilith reads this to understand herself. Must be designed once and followed without exception.
  • Schema (complete): ucl_events table with all columns — id, timestamp (UTC microseconds), event_type (dot-namespaced text), endpoint_id, session_id, payload (JSON), duration_ms, success (bool), error_msg. No other tables in ucl_db at v1.
  • Event taxonomy (complete): every dot-namespaced event type in the system listed. llm.request, llm.response, tool.call, tool.result, voice.stt.complete, voice.tts.start, router.classify, context.assemble, sync.calendar.complete, memory.write, user.message_sent, error.unhandled — full list, not examples.
  • Write path: the exact function called to log an event. Synchronous or asynchronous write. What happens if the write fails (telemetry must never crash the main path).
  • No raw table LLM access: the LLM never queries ucl_events directly. It reads L2-enhanced views only. This is a hard architectural rule, not a preference. Standard
  • Retention policy: how long raw events are kept. After retention period, are they deleted or archived? Are L1/L2 summaries kept longer than raw events?
  • Indexes: which columns are indexed for query performance — at minimum timestamp and event_type. Any compound indexes for common query patterns.
  • Payload schema per event type: for each event type in the taxonomy, the exact JSON fields in the payload. Not vague — specific field names, types, and what they mean.
  • Enforcement: how the "telemetry on every write path" rule is enforced during review. Checklist item in PR review. No automated enforcement planned for v1 — stated as a discipline requirement.
Touches: §17 Enhancement Layers · §12 LLM · §13 Router · §14 CWM · §15 Tools
17
Enhancement Layers (L1/L2)
Raw data is never LLM-readable. Enhancement layers transform it into purpose-built views. Every data domain has at least L1. L2 produces the narrative the LLM actually reads.
  • L1 definition: aggregation and normalization of raw data. Runs as a background job, not on-demand. Output is structured summary tables, not narrative text. Idempotent — safe to re-run.
  • L2 definition: interprets L1 output into LLM-readable context blocks. May involve a small LLM call or a template engine. Output is text or structured text. Also runs as a background job.
  • Enhancement domains for v1: calendar, contacts, finance, UCL/telemetry, memory. Each domain listed with its L1 computation and L2 narrative output.
  • Job scheduling: when L1 and L2 jobs run — after sync completion, on a timer, or triggered by data change. Maximum staleness tolerance per domain before a job is forced.
  • Output storage: where L1 and L2 outputs are stored — in the same DB as raw data (separate tables) or in a separate enhanced_db. Decision stated with rationale.
  • Staleness tracking: how the CWM knows if an L2 block is fresh enough to use. Timestamp column on enhancement output, compared against the staleness threshold defined per domain in §14.
  • Failure isolation: if an L2 job fails, the L1 data is still valid. If L1 fails, raw data is still in the DB. Enhancement failures degrade context quality but never crash the main path.
  • UCL telemetry: enhancement job start, complete, duration, records processed, success/failure.

18
Memory System
Persistent recall across sessions. Three tiers: warm store (recent, fast), cold store (archived, slower), personal truths (stable facts about the user, permanent).
  • Warm store definition: what goes here — recent conversation summaries, recently mentioned entities, active tasks. Retention window (last N days or last N entries). Query method.
  • Cold store definition: what goes here — older summaries beyond warm window, significant events, archived conversations. How items move from warm to cold (age threshold, or triggered by compression).
  • Personal truths store: structure of a truth entry — content, confidence level, source (explicit/inferred), timestamp, last_confirmed. The schema is a design decision. Open — PERS-1 D1
  • Truth write triggers: what causes a truth to be written — explicit user statement ("I am vegetarian"), repeated mention, or subtle pattern. Which of these are in scope for v1.
  • Recall tool behavior: how the memory_recall tool searches — keyword, semantic similarity, or both. What it returns — text snippets, structured entries, or both. How results are ranked.
  • Memory injection into context: does memory go into the CWM block system (yes), and under what intent labels does the memory block load?
  • Session isolation: whether in-session conversation state is kept separate from persisted memory until session end. How a session is defined (time gap, explicit end, or device change).
  • UCL telemetry: memory.write (with tier), memory.recall (with match count), memory.compress (when warm→cold migration runs).

19
Data Sync Pipeline
How external data (Google Calendar, Google Contacts) gets into the local encrypted databases safely and efficiently. Sync failures must never corrupt local state.
  • Sync sources for v1: Google Calendar and Google Contacts only. No other sources in v6 v1. Others listed as future.
  • OAuth flow: which OAuth library (google-auth-oauthlib), scopes requested (calendar.readonly, contacts.readonly), how the browser is opened for the consent screen, how the callback is handled locally.
  • Token storage and refresh: access token and refresh token stored in OS keychain. Refresh logic — check expiry before each sync, refresh if within threshold, update keychain with new token.
  • Incremental sync: how each sync avoids pulling everything every time — sync tokens for Google Calendar API, etags for Contacts. What triggers a full sync vs incremental.
  • Write safety: sync writes to DB in a transaction. If the sync fails mid-way, the transaction rolls back — no partial state. How this is implemented.
  • Conflict handling: what happens when a local record differs from the remote (edit on both sides). For v1: remote wins for calendar and contacts (read-only sources). Stated explicitly.
  • Sync schedule: how often each source is synced — interval (minutes), startup sync, and manually triggered sync. Whether sync runs when the device is on battery (mobile consideration).
  • Rate limit handling: Google API rate limits — how they're detected (429 response), backoff strategy, how the user is notified if sync is blocked.
  • UCL telemetry: sync.start, sync.complete (with record count and duration), sync.error (with error type), sync.rate_limited.

Application Layer — Core Apps
20
Contacts / People App
The first of three vertical slice apps. People data is central to most queries — who is this person, when did we last talk, what do I know about them.
  • Database schema (contacts_db): every table, every column, types, constraints, indexes. People, relationships, interaction_history, dossier_entries — exact schema.
  • Google Contacts field mapping: which Google Contacts fields are pulled and how they map to contacts_db columns. Fields not mapped are discarded — stated explicitly.
  • L1 enhancement: what L1 computes for contacts — interaction frequency, last contact date, relationship strength score. How these are computed from raw contact and calendar data.
  • L2 enhancement (dossier block): what the LLM-readable context block for a person looks like. What fields, what format, token budget per person entry.
  • get_contacts tool: input parameters (name search, relationship filter, recency), output format, how it queries (L1 or L2 data, not raw), error cases.
  • Dossier concept: free-form notes per person that Lilith writes from conversation. Schema for dossier entries, who can write them (only Lilith via tool, not sync), how they surface in context.
  • Privacy handling: contacts data is sensitive — what's included in cloud LLM calls and what's withheld. Names included; home addresses potentially excluded from context block. Policy stated.
  • UCL telemetry: contacts.sync.complete (record count), contacts.query (tool call), contacts.dossier.write.

21
Finance App
The most sensitive data in the system. Financial data must be treated with the strictest privacy controls. For v6 v1, manual entry only — no bank sync.
  • Database schema (finance_db): every table and column — transactions, accounts, categories, budgets. Types, constraints, indexes. Account numbers never stored in plaintext even within SQLCipher.
  • Data ingestion for v1: manual entry via tool call or UI. No bank sync, no Plaid, no CSV import in v6 v1. Stated clearly so no scope creep.
  • L1 enhancement: what L1 computes — spending by category (last 30/90 days), income vs expense summary, budget utilization. How often it runs (after any transaction write).
  • L2 enhancement: the narrative block the LLM reads. "You spent $X on Y last month, up Z% from the month before." Token budget, update frequency.
  • get_finance tool: input parameters (date range, category, account), output format, what level of detail is returned by default vs on explicit request.
  • Cloud LLM privacy: specific fields that NEVER appear in a cloud-bound LLM context — account numbers, full transaction merchant names (hash or category instead?). Policy written as a rule, not a suggestion.
  • UCL telemetry: finance.transaction.write, finance.query (tool call). No amounts or merchant names in telemetry payload — only category and relative size (small/medium/large).
Touches: §2 Security · §17 Enhancement Layers

Conversation / Chat App
The persistent log of every interaction across every endpoint. The chat screen is its UI. Cross-device continuation is its killer feature.
  • Database schema (conversation_db): sessions table, messages table — every column. Message fields include: id, session_id, timestamp, role (user/assistant), content, endpoint_id, mode (conversational/artifact), source_type (voice/text).
  • Endpoint tagging: how every message is tagged with its origin endpoint at write time. The endpoint_id format and registry — where endpoint IDs are assigned and stored.
  • Session definition: what constitutes a conversation session — time gap threshold, device change, explicit new conversation. How sessions are created and closed.
  • Cross-device continuation: how a conversation started on the kitchen speaker is continued in the desktop chat window. Session ID is the link. How the desktop UI signals "continuing session X."
  • Conversation history block for CWM: format and token budget of the history block injected into context. How many raw turns before compression. What compression produces.
  • Search and retrieval: how the user finds an old conversation. Full-text search (FTS5 in SQLite), date filter, endpoint filter. What the UI shows for search results.
  • Artifact messages: how artifact-mode responses are stored — the acknowledge, the task result, and the handoff as distinct message entries or as a single structured message. How the UI renders them differently.
  • UCL telemetry: conversation.session.start, conversation.session.end (with duration and message count), conversation.message.write.

Personality & Response Layer
23
Personality System & System Prompt
Who Lilith is, in text. The system prompt is the most read piece of the entire system. Its structure determines how reliably Lilith behaves. Every word in it is a design decision.
  • System prompt structure: the ordered sections of the system prompt — identity, behavioral rules, tone guidelines, personal truths (stable), current context (volatile). Order matters for LLM attention.
  • Stable vs volatile sections: which parts of the system prompt never change (identity, rules) vs which are injected fresh each call (personal truths summary, current context note). Cache-eligible sections marked for Anthropic prompt caching.
  • Personal truths injection: where in the system prompt truths appear, how many truths are injected at once, whether all truths are always included or selectively filtered by relevance to the current query. Open — PERS-1 D2
  • Personality document format: the file that defines Lilith's character — what it contains, what format it's in, where it lives, how it's loaded into the system prompt. Not the content — the structure and format decision.
  • Two-mode tone differences: how conversational mode system prompt differs from artifact mode. Conversational: brief, warm, immediate. Artifact: acknowledging, managing expectations. Whether these are two prompts or one with mode-injected instructions.
  • System prompt versioning: how prompt changes are tracked. Whether a prompt version is stored in the DB so telemetry can correlate behavior changes to prompt changes.
  • Personality model (PERS-2): hardware-gated, Tier 1+ only, design-only for v6 v1. Where in the pipeline it fits (artifact mode only, post-LLM style pass). Noted but not implemented.

Narrative Layer
The voice of the artifact mode pipeline. Every tool has acknowledge, bridge, handoff, and failure copy. This is what the user hears while Lilith works.
  • YAML schema for tool narratives: exact fields — tool_name, acknowledge (spoken immediately), bridge (spoken at interval if task is slow), handoff (spoken on completion), failure (spoken on error). Format of the YAML file per tool.
  • Narrative file location and loading: where narrative YAML files live (tools/ directory alongside tool definition, or separate narratives/ directory), how the loader discovers them at startup.
  • Acknowledge timing: the acknowledge phrase is spoken before the tool call begins — how quickly after the user's input it fires, and how this is enforced technically.
  • Bridge timer: how long before a bridge phrase fires if the tool is still running. Fixed duration vs adaptive (based on tool's average historical duration from UCL). Open — CONV-2 D2
  • Handoff summary source: does the handoff phrase come from the tool result directly (template fill), from a small LLM call summarizing the result, or from the narrative YAML? Open — CONV-2 D3
  • Failure narrative: what Lilith says when a tool errors. Generic fallback vs tool-specific failure copy. Whether the error detail is shared with the user or abstracted.
  • Tools requiring narratives for v1: complete list of tools that need narrative YAML files in the vertical slice. Every artifact-mode tool must have one before it ships.
  • UCL telemetry: narrative.acknowledge.fired, narrative.bridge.fired (with delay), narrative.handoff.fired (with task duration).

Onboarding Flow
The first-run experience. Everything in this section happens once. It must end with Lilith knowing who the user is and having initial data to work with — or it failed.
  • First-run detection: how the app knows it's the first run — flag in a config file or a dedicated DB table. What sets the flag to "done" (completion of onboarding flow, not just first launch).
  • Passphrase setup: the step where the user sets their database passphrase. Requirements (minimum length, confirmation), how it's tested (decrypt/re-encrypt a test value), what happens if they skip it (is skipping allowed for v1?).
  • Google sign-in: Google SSO step — what is requested (contacts.readonly + calendar.readonly), how the consent screen is shown (system browser), what feedback the app gives while waiting for completion.
  • Initial sync during onboarding: what gets synced at first run, how long it takes, what the user sees while it runs. Progress indicator? Lilith speaks through it?
  • Hardware detection disclosure: whether the user is shown their detected hardware tier and what features it enables. If not shown, when they can find it.
  • First conversation after onboarding: what Lilith says when onboarding is complete and the first real conversation begins. Should demonstrate immediate value from the synced data.
  • Onboarding skip / defer: can any step be skipped and completed later? Which steps are blocking (passphrase) vs optional (Google sign-in, can be done from settings later)?
  • UCL telemetry: onboarding.step.complete for each step, onboarding.complete (with total duration), onboarding.error (if any step fails).

Frontend Layer
26
Frontend Framework & Application Shell
The largest open design decision. Framework choice determines everything about the orb, the backend-to-UI event bus, the build toolchain, and the touch/mouse parity story. Must be decided before any UI code is written.
  • Framework selection: DECIDED — PySide6 6.11.1 (Qt6, LGPL v3). PyInstaller --onedir distribution. No commercial license required. Qt Quick 3D dropped (GPLv3); orb uses ModernGL 5.x (MIT) in QOpenGLWidget instead. See RES-01, RES-03, RES-15. Standard
  • Backend-to-frontend event bus: how Python backend sends navigation and UI commands to the frontend. The protocol — event names, payload format, how the frontend registers handlers. Must be defined before any screen is built.
  • Frontend-to-backend calls: how user actions in the UI trigger backend functions. Named commands, parameters, response format. Same interface regardless of framework choice.
  • Touch and mouse parity contract: every interactive element must respond to both tap and click. Right-click menus must have long-press equivalents. Hover tooltips must have tap-to-show equivalents. Design rule, not framework feature.
  • UI command protocol for Lilith-controls-UI: how Lilith issues UI commands (navigate, highlight, resize, run_demo_step). The full set of commands she can issue, the payload format, and how the frontend maps them to actions.
  • Element addressability: every notable UI element has a semantic ID known to the backend. The naming convention for element IDs and how they're registered (or defined in a shared schema).
  • App window behavior: initial size, minimum size, full-screen behavior, whether the app runs in the system tray when minimized. Per-platform behavior noted.
  • Build output: what the frontend build produces, how it's bundled with the Python backend, how the resulting package installs on each OS.

27
The Orb
Lilith's physical presence in the UI. 3D, animated, voice-reactive. This is the element that makes Lilith feel like an entity rather than a chat window. Its implementation is framework-dependent.
  • 3D implementation path: DECIDED — ModernGL 5.x (MIT) embedded in PySide6 QOpenGLWidget. UV sphere mesh (numpy, 48×24 sectors), GLSL Fresnel glow shader, Kokoro RMS amplitude → u_amplitude uniform, spring-physics positional animation. Qt Quick 3D dropped (GPLv3 incompatible). Full implementation spec in RES-03. Standard
  • Orb state machine: complete set of states — idle, listening, processing, speaking, error, background_task. Transition triggers (from audio events, from router, from TTS events). Visual behavior per state.
  • Voice-reactive animation: how microphone amplitude (from §9) drives orb pulse, scale, or particle density in real time. Maximum acceptable latency between audio and visual (target: <50ms).
  • TTS amplitude sync: how TTS audio amplitude (from §11) drives orb behavior while Lilith speaks. The signal path from TTS output to orb animation.
  • Orb positioning: where the orb lives in the layout — fixed corner, floating, or responsive to the active screen. Whether its position is controlled by Lilith (she can move herself) or fixed by layout.
  • Pointing / attention direction: how the orb "points" at a UI element when Lilith wants to direct attention to it. Whether this is an animation (glow toward element, particle trail) or a visual indicator (arrow/highlight).
  • Idle behavior: the orb when Lilith is not speaking and no input is coming. Not a frozen image — subtle drift, occasional micro-animation, internal light shift. Described in implementation terms, not just aesthetics.
  • Performance budget: GPU/CPU budget for the orb at idle and during active animation. Must not spike system resources. Target frame rate (60fps or 30fps acceptable).

Screen Architecture & Navigation
The map of every screen in the vertical slice and how the user moves between them — both manually and when Lilith navigates for them.
  • Screen inventory (complete for v1): every screen in the vertical slice listed by name — Chat, Calendar, Contacts, Finance, Assistant, Settings. No others in v6 v1. Stated as a constraint.
  • Navigation model: how the user moves between screens — sidebar/nav rail, tab bar, gesture-based, voice command. Choice must work for both mouse and touch. Decision stated with rationale.
  • Lilith-driven navigation: how the UI command navigate(screen_id) is handled — transition animation type and duration, what happens to ongoing content on the current screen.
  • Data loading per screen: how each screen gets its data — from a backend query on every load, from a persistent subscription, or from a cached frontend store. How loading states are shown.
  • Chat screen specifics: message list rendering (virtualized for long histories), source tags per message, visual distinction between conversational and artifact responses, continuation thread indicator for cross-device sessions.
  • Calendar screen specifics: default view (day/week/month), how events from contacts_db appear alongside calendar_db events, how Lilith can highlight a specific event.
  • Deep link / element addressing: how a UI command like highlight(element_id: "contact_sarah") maps to a specific rendered element. Whether the screen must be active for this to work, or if navigation happens first.
  • Demo mode: the mechanism that allows Lilith to run a scripted sequence of UI commands from a markdown instruction file. How the sequence is parsed, timed, and executed. What can interrupt a demo in progress.
