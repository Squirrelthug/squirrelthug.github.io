1. What Actually Differs Across Operating Systems

When you look at a cross-platform codebase from a bird's eye view, there are two types of code: the stuff that's the same on every OS and the stuff that isn't. Understanding that boundary is the whole game. The parts that differ aren't random — they map onto specific areas where each OS made different design decisions.

WHAT CHANGES ACROSS OSes (for Lilith specifically)

  Area                  Linux               macOS               Windows
  ──────────────────    ──────────────────  ──────────────────  ──────────────────
  Secret storage        libsecret (DBUS)    Keychain (Security) DPAPI (Win32)
  Audio system          PipeWire/PulseAudio CoreAudio           WASAPI/DirectSound
  File paths            /home/user/         /Users/user/        C:\Users\user\
  Path separator        /                   /                   \ (backslash)
  App config dir        ~/.config/lilith/   ~/Library/Applic.   %APPDATA%\Lilith\
  Process signals       SIGTERM, SIGKILL    Same (POSIX)        No SIGTERM; use Win32
  System tray           xdg-utils           NSStatusItem        Shell_NotifyIcon
  Desktop overlay       X11/Wayland         Cocoa overlay       Win32/DirectComposition
  Package format        .AppImage / .deb    .dmg / .pkg         .exe / .msi
  Python audio libs     pyaudio (ALSA/PA)   pyaudio (CoreAudio) pyaudio (MME/WASAPI)
  SQLCipher bindings    gcc builds fine     clang builds fine   MSVC or MinGW required
  STT/TTS native        Whisper/Kokoro fine Same                Same (but CUDA differs)
  Auto-start at login   systemd user unit   launchd plist       Registry Run key

The list looks long. But most of those differences are handled by the framework you choose for the frontend and by the Python libraries you use for backend platform integration. You don't write OS-specific code for most of them — you use a library that does it for you. The ones that are genuinely harder on Windows are marked below.

The Three Genuinely Hard Windows Problems

  1. Build toolchain. Linux and macOS both use gcc/clang, which most Python C extension wheels are built against. Windows requires MSVC (Microsoft Visual C++) for native extensions. SQLCipher, some audio libraries, and certain ML libraries either need pre-built Windows wheels or a working MSVC installation to compile from source. This isn't insurmountable — it just requires deliberate attention.
  2. Audio system variance. Windows audio has three APIs (MME, DirectSound, WASAPI) and they behave differently depending on device drivers. Voice applications on Windows frequently have audio device selection issues that don't exist on Linux/macOS. Latency characteristics are different. pyaudio behaves differently under WASAPI vs. MME.
  3. Overlay behavior. The MiniOrb — a transparent, always-on-top window that floats over other applications — is native to Linux (X11/Wayland compositors handle it), clean on macOS (Cocoa makes it easy), and tricky on Windows because of how Win32 handles layered windows and focus stealing. Not impossible. Just the one UI element that needs the most platform-specific testing.
The practical upshot
If you choose a framework that abstracts the desktop (Tauri or Flutter), most of the table above gets handled for you. What remains — secret storage, audio device selection, the overlay — are the places you'll write platform-specific code or find platform-specific libraries. That's maybe 5–10% of the codebase, not a rewrite.

2. Why Linux-First Dev Made Sense — and Still Does

The perception that AI/ML development is a Linux-first ecosystem isn't wrong — it's just worth understanding why, so you can judge whether it applies to what you're building.

Why the AI community developed Linux-first

CUDA — Nvidia's GPU programming framework — was Linux-first for most of its history. Training runs on research clusters (AWS, GCP, Lambda Labs) all run Linux. Docker containers, the standard packaging unit for ML services, are Linux-native. The Python AI ecosystem (PyTorch, TensorFlow, Transformers) is tested most thoroughly on Linux. Conda/mamba, the package manager of choice for ML environments, handles Linux dependency resolution better than Windows. Local inference tools like Ollama, llama.cpp, and vLLM all develop and test on Linux first.

But this mostly applies to the training and serving infrastructure layer — the stuff data centers run. For a desktop application using pre-trained models via API or small local models via Ollama, the Linux-first advantage is real but much smaller. Whisper.cpp, Kokoro, Ollama — all of these work on all three operating systems. The cases where Linux gives you an edge are edge cases in a desktop app context.

The reason Linux made sense for Lilith specifically:

  • It's where you are. You develop there. The friction of switching is real.
  • Terminal tooling (bash, cron, systemd) integrates cleanly with Lilith's background processes.
  • The developer community you'd want early feedback from skews Linux and macOS.
  • No licensing costs. No activation. No unexpected update behavior that breaks your dev environment mid-sprint.

None of these reasons go away for v6. Linux is still the right development OS. What's worth reconsidering isn't where you develop — it's the assumption that Windows would require a fundamentally different codebase. It doesn't. It requires platform-specific polish in about three areas, not a rebuild.

3. Linux → macOS: The Easy Jump

You already proved this one. The early Tauri app ran on both. The early node prototype — treating the MacBook as a lightweight client — worked. The reason it was easy is structural: both Linux and macOS are Unix-derived POSIX systems. Shell commands are the same. Python behaves the same. File paths use the same separator. Signal handling is the same. The differences are surface-level.

POSIX SHARED HERITAGE

  UNIX (1969)
    ├── BSD → macOS (Darwin kernel + BSD userland)
    └── GNU/Linux → your dev machine

  Both speak the same language:
  - POSIX filesystem API
  - fork/exec process model
  - Signal handling (SIGTERM, SIGKILL, SIGUSR1...)
  - Same Python, same pip, same venv
  - Same audio abstractions (CoreAudio ≈ PulseAudio for Lilith's purposes)

The main difference for Lilith's stack: macOS uses CoreAudio instead of PulseAudio. pyaudio handles both. Kokoro TTS works on both. Whisper works on both. SQLCipher builds on both. The secret store is Keychain on macOS instead of libsecret, but python-keyring abstracts that away.

Shipping on macOS from a Linux dev environment is a CI/CD problem, not a codebase problem. You can build the macOS binary on macOS (in GitHub Actions, using a macOS runner), triggered from your Linux dev machine. You never need to touch the macOS machine for daily development.

Practical implication
Linux + macOS is essentially one target. Design once, test on both, CI builds for both. If v6 launches on Linux, it launches on macOS for nearly free. These two are the first distribution targets.

4. Linux → Windows: The Real Work

Windows is genuinely different. Not insurmountably different — but it won't just work the way macOS does. Here's an honest breakdown of what actually needs attention.

ComponentLinux/macOSWindows deltaDifficulty
Python runtime System Python or venv, works everywhere Same, but packaging with PyInstaller/cx_Freeze is more fiddly on Windows Medium
SQLCipher gcc builds the C extension fine Needs MSVC or pre-built wheel; sqlcipher3 on PyPI has Windows wheels now Medium — one-time setup
Audio (STT/TTS) PulseAudio/PipeWire — clean WASAPI works well; PortAudio (used by pyaudio) supports it; test with multiple drivers Medium — needs device testing

Kokoro TTS Works Works; audio output path needs Windows driver testing Likely fine

Nothing in this table is a "different codebase" problem. They're all "write this one thing differently" or "test this more carefully" or "add a CI step for Windows." The concern you had about Windows being fundamentally incompatible was partly true in the early days of Lilith's stack (some ML libraries genuinely didn't have Windows wheels) — but it's less true in 2026. The ecosystem has matured.

The one Windows thing that's actually hard
The Windows audio driver ecosystem is genuinely more chaotic than Linux or macOS. Different audio interfaces, different driver models (WDM, WASAPI exclusive mode, WASAPI shared mode), different latency characteristics. For a voice-first application that depends on reliable low-latency audio I/O, Windows audio needs thorough testing with a variety of hardware configurations. This isn't a reason to skip Windows — it's a reason to test audio on Windows specifically, not just assume it works because the code compiled.

5. Framework and the Cross-Platform Story

The frontend framework choice from the v6 vertical slice document matters a lot here. The frameworks don't behave the same way cross-platform, and that affects the Windows story.

FrameworkHow it handles WindowsOverlay supportBuild from Linux
Tauri v2 Uses WebView2 (Edge-based) on Windows. Ships with it bundled or requires installation. Very good Windows support — Tauri team tests all three platforms in CI. Tauri window flags handle transparent always-on-top on Windows. Tested. Cross-compile with cargo for Windows target. Or use GitHub Actions Windows runner.
Flutter Desktop Renders its own UI — no OS webview dependency. Same Dart code, same rendering everywhere. Best OS-rendering-parity story of any framework. Flutter window plugin supports always-on-top + transparency on all three OS. Cannot cross-compile Flutter for Windows from Linux — needs a Windows machine or CI runner.
Electron Excellent Windows support. Slack, VS Code, Notion all ship on Windows via Electron. Transparent overlay windows work but are heavy. electron-builder handles cross-platform builds. Windows builds need wine or CI runner.

The key tradeoff: Tauri can cross-compile for Windows from Linux using a Windows cross-compilation target (experimental but functional), or you use GitHub Actions with a Windows runner for Windows builds. Flutter cannot cross-compile for Windows — you either have a Windows machine or you use a CI Windows runner. For a solo developer on Linux, GitHub Actions provides free Windows and macOS CI runners. You develop on Linux, push to GitHub, and the CI builds and signs for all three platforms automatically.

What this means practically
You never need to boot Windows to develop Lilith. Linux is your development environment. GitHub Actions does the Windows and macOS builds. You test those builds in a VM or by asking someone with a Windows machine to test for you. This is how most cross-platform desktop applications developed by small teams actually work.

7. Which OS to Launch On First

You said if you had to pick one it would probably be Windows because it's the biggest market. That logic makes sense from raw numbers. But here's the counterargument, laid out without trying to convince you either way.

Case for Windows first
  • 72% of the desktop market
  • If you can make it work on Windows, you've solved the hardest cross-platform problem
  • Enterprise eventual path goes through Windows
  • Early testers more representative of the general population
  • Shows you're not just building for developers
Case for Linux + macOS first
  • You develop on Linux — you test what you build, instantly
  • macOS users pay for software, ask fewer dumb setup questions
  • Developer demographic understands BYOK, gives better feedback
  • Linux → macOS is basically free — one launch covers both
  • Your early adopter base is already there
  • Tauri v2 and Flutter both make Windows phase 2 easy
The case nobody makes but is probably right
Launch on all three simultaneously — but don't let "all three" slow you down during development. Develop on Linux. Use GitHub Actions CI to build and test Windows and macOS binaries from day one. By the time the vertical slice is ready, all three builds have been in CI for months and you launch all three. The overhead is CI config upfront, not extra development time.

This is actually how most cross-platform tools launch. The frameworks (Tauri, Flutter) are specifically designed to make "build for all three from one codebase" a CI/CD problem, not a development problem. You write once on Linux. The CI builds everywhere. You test the CI builds in VMs or with a GitHub Actions artifact download. On launch day, all three installers exist.

The exception: if the Windows audio situation causes real bugs that take time to debug, Windows might slip a couple weeks. That's fine — launch Linux + macOS, follow with Windows within a month. But design for all three from the start.

8. One Repo vs. Many — The Real Industry Answer

This is where the "one repo for everything" instinct comes from: you're used to a single GitHub repository for Lilith, and you've been imagining a world where everything — desktop, phone, embedded device — lives in that one repo. The instinct to keep things unified is good. The application of that instinct to fundamentally different runtime environments is where the pain comes from.

THE REAL QUESTION: what shares enough to live together?

  ┌─────────────────────────────────────────────────────┐
  │  SAME RUNTIME ENVIRONMENT → same repo               │
  │                                                     │
  │  lilith-core/  (one repo)                           │
  │  ├── backend/      Python — runs on all three OSes  │
  │  └── desktop/      Tauri/Flutter — all three OSes   │

  ┌─────────────────────────────────────────────────────┐

  │                                                     │
  │  lilith-android/   (separate repo, future)          │

Desktop (Windows + macOS + Linux) is one repo. All three platforms share the Python backend verbatim. The frontend code is 95% shared, with platform-specific config files (not code) for things like app signing and package format. This is how every major cross-platform desktop app works today.

The API Boundary That Makes This Work

The key architectural decision that determines whether one repo scales: the protocol between backend and frontend must be clean and documented. For Lilith, that's the WebSocket interface between the Python backend and the frontend. If that interface is well-defined — a spec that says "these are the messages, these are the data shapes, this is the event model" — then the frontend can be any technology and the backend doesn't care. A macOS Tauri app, a future Android app, and a HomePod-like embedded client are all just "WebSocket clients that speak Lilith's protocol."

THE PROTOCOL BOUNDARY

  Python backend (one codebase, all platforms)
       ↕ WebSocket — documented message protocol
  Frontend (platform-native)
  ├── Desktop (Tauri/Flutter) — lilith-core repo
  ├── Android app (future) — lilith-android repo
  └── iOS app (future) — lilith-ios repo

  The backend doesn't know which client is connected.
  The client doesn't know what the backend runs on.
  Changes to the backend don't break clients if the protocol stays stable.

This is how companies with large engineering teams handle multi-platform. The shared layer is the protocol, not the code. Each platform owns its own code, but all platforms speak the same language to the backend.

Industry context — monorepo vs. multiple repos

The "one repo for everything" approach (monorepo) is used by very large engineering teams — Google, Meta, Vercel, Expo. The tooling for it (Bazel, Nx, Turborepo) is complex and has a steep learning curve. It's justified when hundreds of packages need to share code and be kept in sync across hundreds of engineers. For a solo developer, it's unnecessary overhead.

The "one repo per deployment target" approach is used by most small-to-medium teams. Desktop is one repo. Mobile is one or two repos (iOS, Android). Embedded is separate. The coordination happens through the API protocol, not through shared code. This scales well because each repo is independently deployable — you don't need to sync all platforms before shipping a backend change.

What "One Repo" Still Means for Desktop

The one thing you don't want to do is build the Windows desktop app, the macOS desktop app, and the Linux desktop app as three separate codebases. That would require three separate UIs, three separate backend integrations, three separate test suites. Every feature addition happens three times. That's the nightmare scenario — and it's not necessary.

Desktop is one repo. Linux, macOS, and Windows builds all come from that one repo. The CI handles the per-platform compilation. That's the right structure, and it's what the framework guides you toward.

Why v6 Will Be Faster

Not just "you know more" — specific reasons:

  • No design discovery. The two-mode architecture, the scope (3 apps), the BYOK model, the desktop-as-hub structure — all decided. None of this gets re-litigated mid-build.
  • No zigzagging. Zigzagging happened because the design changed mid-build. When the design doesn't change, the code doesn't zigzag.
  • Better tools. Antigravity, Grok, Codex, and the delegation workflow are all more mature now. The sprint/design task system exists. Parallel implementation is routine.
  • Tighter scope. 3 apps instead of 12 is literally 4× less surface area. The implementation tracks in each sprint are scoped tighter. The design tasks are already written.
  • Known patterns. The UCL telemetry pattern, the provider model, the context assembly pipeline — you know what these look like from v5. You're not figuring them out, you're rebuilding them clean.
  • Security upfront. Not retrofitting. SQLCipher from day one means the database schema doesn't need to be migrated mid-build.

4–6 weeks to a working alpha of the vertical slice is achievable. Not guaranteed — there will be friction in the new framework choice, in the Windows CI setup, in first-time SQLCipher integration. But the order of magnitude difference from v5 is real. This is a known problem with a known solution. You're building, not discovering.

10. The Multi-Device Fantasy vs. the Practical Path

You named this one yourself: the aspiration to have Lilith on every device — phone, tablet, TV, HomePod-like speaker, all synchronized, all integrated — was clouding the decisions. It's worth separating what's vision from what's scope creep in disguise.

The Vision (still valid)

Lilith as a personal AI hub that serves all your devices. Desktop is the brain. Everything else is a thin client. End-to-end encrypted sync. You talk to Lilith on your phone in the car; she continues the context on your desktop. That vision is correct and worth building toward.

The Scope Creep in Disguise

Building the Android app, the iOS app, the TV app, and the embedded device client simultaneously with the desktop, as part of v6 v1 — that's not the vision. That's the vision's worst enemy. Each of those is a separate engineering effort of comparable scope to the desktop app. A solo developer building four of them at once is guaranteed to build all four of them badly.

The hacked-together Android app for personal use — hardcoded IP, your API keys baked in — is the right v1 mobile story. You've built World of Warcraft addons. Hacking an Android connection to Lilith's WebSocket backend is orders of magnitude simpler than what you've already built. When you're ready for it, it'll take a weekend. Not a sprint series.

The embedded device question
A HomePod-like speaker running Lilith — a Raspberry Pi with a microphone and speaker in a nice enclosure — is a completely valid personal home project and a great showcase if you ever want to demo to people. It's also a completely separate engineering effort from the desktop app. It doesn't need to be designed for or architected around in v6. The WebSocket protocol that the desktop client uses can be spoken by any device you put a network interface and a microphone on. Design the protocol cleanly, and the embedded device is just another client.

11. The Verdict

Cross-platform and repo structure — summary decisions

Development environment

Stay on Linux. The reasons you chose it are still valid. The concern about Windows being architecturally inaccessible from Linux is no longer a meaningful constraint — GitHub Actions CI handles Windows builds without you touching a Windows machine.

Distribution targets

Design for all three desktops (Windows, macOS, Linux) from day one. Set up CI for all three platforms on the first sprint. The CI overhead is a one-time investment, not per-feature work. Linux + macOS launch together (they're nearly the same target). Windows follows as soon as audio behavior is tested and the three genuinely hard Windows problems above are verified.

Which OS first

Linux and macOS simultaneously — this is effectively free given shared POSIX heritage. You already proved the Tauri app runs on both. Windows follows within weeks of the Linux/macOS launch, not months. Don't pick Windows first just because it's the biggest raw market — your first users are the developer-adjacent, privacy-conscious, BYOK-comfortable demographic, and they live on macOS and Linux.

Repository structure

One repo for the desktop (all three OS targets). That repo contains the Python backend and the frontend code. The CI produces Windows, macOS, and Linux builds from the same commit. Separate repos for future mobile clients, built when the desktop is solid. The coordination layer between them is the WebSocket protocol — document it once, and every future client just implements the protocol.

Mobile and embedded

Not in v6 v1. Not even in the architecture consideration for v6 v1 beyond "the WebSocket protocol must be documented." When you want an Android client for personal use, hack it together in a weekend — a WebSocket client with voice I/O forwarded to the desktop. That's enough for personal use and doubles as a proof-of-concept for investors if you want it to.

The timeline

4–6 weeks to a working alpha of the vertical slice is realistic. The design overhead that consumed most of v5's calendar time is gone. What remains is building a known solution to a known problem, with better tools than you had in February.
