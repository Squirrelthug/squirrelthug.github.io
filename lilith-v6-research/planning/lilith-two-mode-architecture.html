The Core Realization

The architecture you described was always the right one

Everything in the last article — the context window manager, the dynamic retrieval, the query classifier, the hybrid vector DB — those are all still real work that needs to happen. But they were being framed as solutions to the wrong problem. The problem was never "how do we fit more context into a small window." It was: how does Lilith always feel present and fast to talk to, even when the underlying system is slow?

The answer isn't better hardware or smarter routing. It's a design paradigm: Lilith is a personality that exists in the conversational layer. Everything else — tool results, long documents, background research, cloud model outputs — is an artifact that arrives from a system. She receives artifacts. She delivers them. She is not them.

The Paradigm Shift
Before: Lilith generates a response. If it's slow, that's a latency problem.
After: Lilith speaks immediately — conversationally, always. When something takes time, it's a background task completing, not a slow response. The framing makes latency architecturally invisible.

This isn't a workaround. It's actually how trust is built with any assistant — human or AI. A good assistant doesn't go silent for 47 seconds and then deliver a monologue. They say "let me pull that together" and keep the conversation moving while the work happens.

K-2SO from Rogue One. TARS from Interstellar. What made those characters work wasn't that they were fast — it was that they were present. Personality in the foreground, capability in the background. That's the original design of Lilith, and you arrived back at it by working through all the hardware friction.

Note on prediction
You don't need to generate the full response to know it'll be long. Query classification (the same intent layer from the context window article) can predict output size before any model sees the prompt. "What's the weather?" → short. "Explain everything about my finances" → artifact. This decision can be made in milliseconds.

The Bandwidth Truth

Voice has fundamentally lower information bandwidth than text. You can scan a well-structured document in 30 seconds and absorb the key points. Hearing someone read that same document aloud takes 10 minutes. This isn't a failure — it's the nature of the medium. Voice is rich and human and trusted. It's just low bandwidth.

The implication: voice should never be used to deliver information that would be better scanned. Anything dense — a finance summary, a code diff, a list of calendar events — belongs in a visual artifact with Lilith speaking a short summary: "Your March spending is in the panel. Main thing: Costco was way up." That's two seconds of audio plus something you can actually read. This is better for the user AND better for the architecture.

Conversational Mode — Design

The fast path · Always ≤ 50 output tokens · Always feels immediate

Conversational mode is the default. Every interaction starts here and only escalates if the query demands it. The entire design of this path is optimized for one metric: time-to-first-audio.

  CONVERSATIONAL MODE PIPELINE
  ────────────────────────────────────────────────────────────

  User speaks
       │
       ▼
  Query Classifier  (~5ms)
  ├── Intent: simple / conversational / quick-lookup
  ├── Predicted output: ≤ 50 tokens
  └── Context needed: minimal
       │
       ▼
  Context Assembly  (~10ms, runs in parallel with classifier)
  ├── dossier         (always — ~200 tokens, who you are)
  ├── last 3–5 turns  (~500 tokens max)
  └── 1–2 recall hits (~200 tokens)
  Total context: ~1,000 tokens (not 22,000)
       │
       ▼
  Local Model  (Mistral NeMo or fast cloud)
  ├── 1K token prefill  →  ~2s on RTX 3060
  ├── Output budget: HARD CAP at 60 tokens
  └── If model would exceed cap → truncate + append artifact flag
       │
       ▼
  Personality Layer  (baked into system prompt — no extra model pass)
  └── EDICTS + VOICE already embedded; no second inference needed
       │
       ▼
  format_for_speech()  → Kokoro TTS
  └── Audio begins streaming

  ────────────────────────────────────────────────────────────
  Total latency target: ≤ 3 seconds to first audio
Conversational Mode — What It Handles
  • "What's the weather right now?"
  • "Remind me what I was working on yesterday"
  • "When's my next appointment?"
  • "What time is it in Tokyo?"
  • "Set a timer for 20 minutes"
  • "What did we talk about last week with the finance stuff?"
  • Casual back-and-forth — any reply to a reply
  • Short acknowledgements, confirmations, small talk

The Hard Output Cap

The most important enforcement mechanism in conversational mode is a hard cap on output tokens. The model is told in the system prompt that in conversational mode, responses must fit within the token budget. If a question genuinely requires more — it says so briefly and signals artifact mode.

This is not clipping the model's ability. It's training the right behavior: "I can give you the short version now, or pull together the full picture — which do you want?" That's how a skilled assistant actually operates.

Artifact Mode — Design

The background path · No latency pressure · Lilith speaks immediately anyway

Artifact mode is not slow mode. It's parallel mode. Lilith speaks within 2 seconds regardless of how long the background task takes — she has a narrative response ready before the work is even started.

Artifact Mode — What It Handles
  • Finance summaries, spending trends, monthly breakdowns

Lilith's Relationship to Artifacts

The key framing: artifacts don't originate from Lilith — they originate from systems Lilith has access to. She's the one who requested them, received them, and is delivering them. She is not the artifact. She is the assistant who pulled it from the system and is handing it to you.

This matters for how she speaks about them: "The finance module came back with this" not "I calculated your spending." The subtle difference builds the right mental model for the user: Lilith is present and responsive; the underlying systems are capable but have their own latency.

Why This Builds Trust
A human assistant who says "let me look into that and come back to you" is more trustworthy than one who goes silent and returns 45 seconds later with an answer. The verbal acknowledgement + brief explanation + follow-through is the entire trust loop. Lilith just needs to execute that loop on every artifact task.

The Narrative Layer

What Lilith says while the work is happening — per tool, per situation

Every tool and every artifact class needs a defined set of filler narratives — short spoken responses Lilith delivers immediately while the background task runs. These aren't filler in the pejorative sense. They're conversational acknowledgements that maintain presence and set expectations.

The Structure of a Narrative

Each narrative has three parts, delivered across the interaction:

  1. Acknowledgement — confirms the request was understood, task is underway (spoken immediately)
  2. Bridge — optional, keeps conversation alive if processing is taking longer (spoken mid-wait)
  3. Handoff — delivers the artifact summary when the result arrives (spoken on completion)

trigger: calendar range query · task queued
Web Research · Acknowledgement
"That's going to take me a minute to pull together — I'm searching on it."

Large Document / Long Generation · Acknowledgement
"There's a lot there. I'm going to work through it in the background — might take a bit. What else is on your mind?"
trigger: estimated output > 500 tokens · routed to background
System Self-Query · Acknowledgement
"My system's generating a report on that — I'll have it for you in a moment."
trigger: user asks about Lilith's own internals · telemetry query queued

Bridge Responses (For Long Waits)

If a task is predicted to take longer than ~8 seconds and Lilith has gone quiet, a bridge response fires automatically — something that continues the conversation or signals the wait is expected. She doesn't go silent. Silence is the UX failure to avoid above all others.

Generic Bridge · fires at 8s if task incomplete
"Still working on that one — it's a bit to pull together."
trigger: task_elapsed > 8s AND no response yet

Where These Live in Code

Each tool definition already has a YAML schema in tools/definitions/. The right place to add narratives is a new narratives: key in each tool YAML — three optional string fields: acknowledge, bridge, handoff. The router reads these at dispatch time and fires the appropriate one to the TTS queue immediately, before the tool even starts executing.

The Personality Model — Where It Actually Belongs

A small model for voice, not for reasoning

You described running a small model on the upgraded GPU whose job is to take generated text and run it through Lilith's personality layer before it hits TTS. This is architecturally correct — but its role is different in each mode.

Conversational Mode

Personality in the Prompt, Not the Pipeline

  • EDICTS + VOICE already embedded in system prompt
  • A second model pass adds latency you can't afford here
  • Target is ≤ 3s total — no room for extra inference
  • Personality layer model: not used in this path
  • Solution: make the main model's personality baked-in tight enough that no pass is needed
Artifact Mode

Personality Model Earns Its Slot Here

  • Cloud model generates raw content (no personality constraint)
  • Raw content can be long, dense, structured — cloud doesn't care
  • Personality model takes: raw content + delivery context
  • Outputs: Lilith's spoken handoff (the 2-sentence summary)
  • Latency: runs while artifact is being formatted → parallel, free
  • Personality model: translates artifacts into Lilith's voice

The personality model's actual job description: given a dense output from a cloud model or tool, produce the short spoken handoff Lilith uses to deliver it. That's a narrow, fast task a 3B–7B model handles well. It doesn't need to reason. It needs to know Lilith's voice deeply and produce 1–3 sentences in it.

The Collaborative Personality Document

You mentioned the idea of having a document that defines how Lilith handles specific situations — written collaboratively with her, capturing the examples. Something like an extended personality layer beyond EDICTS and VOICE. This would be the primary context the personality model receives, distinct from EDICTS (what she must do) and VOICE (how she speaks). Call it something like character.md — the situational responses, the humor register, the way she handles being asked about herself, the way she handles long silences, the way she transitions between topics.

This is the document that makes her feel like K-2SO or TARS — not just capable, but distinct. You can edit it in conversation with her. You say "that response felt wrong — in that situation you'd say X instead." She edits her own character file. Over time it becomes the richest personality context in the system.

The Context Window Manager, Revisited

Its job changed — it's now a servant of the conversational path

After this design session, the context window manager is no longer the architecture. It's a subsystem that serves the two-mode split. Its job description is now narrower and more achievable:

Old Job Description

Handle all queries efficiently

  • Dynamic retrieval for every turn
  • Balance context quality vs. token count for all cases
  • Route local vs cloud based on complexity
  • Handle short queries and long ones in the same pipeline
New Job Description

Keep conversational mode skinny

  • Assemble ≤ 1,200 tokens for conversational path
  • Never inject finance/people/facts when not queried
  • Artifact mode gets full context — no budget pressure there
  • Route decision: conversational vs artifact (not local vs cloud)

The cloud vs local decision is now downstream of the conversational vs artifact decision. Conversational mode usually goes local (fast, cheap). Artifact mode usually goes cloud (capable, no latency pressure). But either can go either way — a simple query on a slow GPU might route cloud; a complex artifact might run local overnight. The primary split is user experience, not model selection.

Anthropic prompt caching still matters
For artifact mode cloud calls, the stable system prompt (EDICTS, VOICE, tools, identity) is already cached via the CACHE_BREAK split in cloud.py. Artifact mode hits cloud frequently. Cache hits here save real money. Keep the caching architecture exactly as-is — it's in the right place.

The Edge Cases

Three situations that don't fit cleanly into either mode

1. Steven dumps a 20-minute rant (like today)

Large wall of text → artifact mode, immediately. Lilith doesn't try to process it inline. She says something like "That's a lot — let me work through it properly. Keep going or give me a moment?" The full text goes to a cloud model for summarization + synthesis in the background. The result comes back as a structured artifact she then delivers conversationally.

The key is she doesn't go silent. She acknowledges the magnitude of the input and signals the mode shift explicitly.

2. A query that starts conversational but turns artifact mid-generation

The model starts generating a response and realizes it's going long. Two options:

  • The hard output cap truncates it and appends: "...there's more to this — I'll put the full version in the panel."
  • Better long-term: query classification is accurate enough that this rarely happens. "Explain everything about X" is always artifact before the model is called.

3. User is in rapid-fire mode (coding session, quick back-and-forth)

When queries are coming in faster than artifact tasks complete, the context window needs to know which tasks are pending and reference them naturally. "Still working on that search from a minute ago — should have it soon." This requires the context window to include a lightweight "pending tasks" block — a short list of what's processing in background. Small, always injected, keeps Lilith coherent across parallel work.

4. The user directly asks Lilith about herself

This is the introspection case from x.txt. Lilith's self-knowledge should come in layers:

  • General identity — always in context (who she is, what she does, in vague terms)
  • System specifics — queried on demand (UCL telemetry, current model, uptime, recent activity)
  • Deep internals — artifact mode (if she needs to generate a full system report, that's background)

The response to "what are you running on?" should never take 47 seconds. The quick answer is always available. The detailed answer is an artifact.

What This Changes About Build Order

Same work, different priority — the constraint came first

You said in x.txt: "I think we still need to do that, but with the constraint of locking ourselves into a new design." That's exactly right. The work from the context window article still needs doing. The priority order just changed.

Do First — Constraint

Define the two-mode split in code

Add mode: "conversational" | "artifact" to the query classification output. Wire the output token cap into conversational mode. This is the architectural anchor everything else plugs into. Without it, all the other work could be optimizing the wrong path.

Next — Narrative Layer

Add narratives to tool YAML definitions

Three fields per tool: acknowledge, bridge, handoff. Router fires acknowledge immediately on artifact dispatch. Bridge fires at 8s if task still running. Handoff fires on completion. This is additive — no existing behavior changes.

Next — Context

Query-intent block selection for conversational path

From the context window article: only inject blocks relevant to the query. Now easier to scope — conversational mode has a strict ≤ 1,200 token budget, so block selection isn't optional, it's enforced by the ceiling.

After GPU Upgrade

Personality model for artifact handoffs

Small model (3B–7B) loaded alongside Kokoro. Takes raw artifact content + VOICE.md, produces Lilith's spoken handoff summary. Runs in parallel with artifact formatting — no added latency on the critical path.

Later

character.md — the collaborative personality document

Situational responses, humor register, self-identity framing, how she handles long tasks, how she talks about the system. Written with her, iteratively. The thing that makes her feel distinct, not just functional.

Later Still

Conversation compression + history management

Still needed — especially for the conversational path's token budget. But less urgent now that artifact mode takes the heavy workload out of the conversational context entirely.

The Summary
The hardware problem was a design problem in disguise. The conversational path needs to be fast and it can be — not because the hardware got better, but because you stopped asking the conversational path to do things it was never supposed to do. Anything heavy is an artifact. Lilith speaks immediately. That's the design. It was always the design.
