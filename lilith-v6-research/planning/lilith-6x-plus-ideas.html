Lilith 6.x+ — Ideas Parking Lot

Ideas that are out of scope for v6 v1 but must not be forgotten. Captured here so they don't pollute the vertical slice.

Purpose: When a good idea comes up during v6 v1 development that belongs in a future version, it goes here. The rule is simple: if building it now would extend the v6 v1 timeline, or if it requires infrastructure not in the vertical slice, it lives in this document. This is not a backlog. These ideas don't have priority or estimates — just enough description to rebuild context when we return to them. Some may never ship. All are worth remembering.

Important: Even though these ideas are deferred, the v6 v1 architecture must not foreclose them. Where the architecture needs to leave a door open for a 6.x+ idea, that accommodation is noted.
Security & Privacy
6X-01
Privacy obfuscation layer for cloud routing 6.2+
When the routing pipeline selects a cloud LLM, an additional enhancement pass is applied to the context that obfuscates PII: real names replaced with aliases, addresses removed or generalized, account numbers omitted. This layer sits between L2 output and the CWM assembly for cloud-bound calls. It is a separate, parallel enhancement path — not a modification of the existing L1/L2 outputs.
User value: Talk to any cloud AI about your real life without your real data leaving your device in identifiable form.
Architecture note: The L2 → CWM pipeline must preserve the distinction between cloud-routed and local-routed calls so this layer can be inserted without reworking the CWM. Design the routing decision to be available at context assembly time.
6X-02
End-to-end encrypted inter-Lilith communication 6.3+
When two Lilith instances communicate on a local network (home server, secondary machine), all traffic is end-to-end encrypted. This is a prerequisite for the distributed GPU pool (6X-04) and multi-device sync (6X-03).
Architecture note: The v6 v1 communication layer (backend event bus) should use a protocol that can be secured without a redesign — TLS or noise protocol.
Hardware & Compute
6X-03
Hardware tier visualization UI — draggable resource allocation 6.2
A visual interface (like Apple's storage breakdown bar) that shows total GPU and CPU capacity and lets the user drag-assign pipeline components between GPU and CPU. Each component (STT, TTS, conversational LLM, artifact LLM) appears as a colored block. Dragging from GPU to CPU shows the component shift to a CPU-only color with a label change. Makes hardware configuration intuitive for non-technical users.
Most users have never heard of GPU VRAM. This visualization teaches them what their hardware is doing and gives them meaningful control without needing to understand it.
Architecture note: The capability tier profile must store individual per-component assignments, not just a single tier number, so this UI has data to read and write. Design the capability profile schema to support per-component routing even if v6 v1 doesn't expose the UI for it.
6X-04
Distributed home GPU pool — background task offloading 6.4+
If you have multiple computers at home, Lilith instances on each can collaborate. The primary instance routes background and artifact tasks to secondary instances with available GPU capacity during their downtime. Enhancement layer processing, memory compression, long-form artifact generation — all can run on a secondary machine and return results to the primary.
A home with three computers suddenly has the background processing power of a small server. No cloud billing. All local.
Architecture note: The artifact dispatcher and enhancement job scheduler in v6 v1 should be written so that the job executor can be a remote process, not just a local thread. Abstract the executor interface.
6X-05
Hardware change detection and tier re-assignment prompt 6.2
If Lilith detects on startup that GPU configuration has changed since the last run (new GPU installed, old GPU removed), she notifies the user and offers to walk through hardware tier re-assignment. This includes suggesting new local models to download based on the new capability level.
Architecture note: The capability tier profile written at startup should be persisted (not just session state) so it can be compared against the freshly detected hardware on next boot. Current design: session state only. This needs a one-field config write to support future comparison.
Multi-Device & Sync
6X-06
Multi-device identity — two Liliths recognize each other 6.2+
Running Lilith on both a MacBook and a Linux desktop. The two instances recognize each other on the same network, sync shared memory and conversation context, and coordinate so that a session started on one device can be continued on the other. Each device maintains its own hardware tier and routing decisions.
Architecture note: The session and conversation schema in v6 v1 must include a device_id and allow multiple devices to share a session_id. Don't assume sessions are single-device.
Voice Pipeline — Advanced
6X-07
Graceful interruption handling 6.2
When the user speaks while Lilith is speaking, Lilith pauses, VAD processes the new input, classifies it as an interruption vs. a backchannel ("yeah, exactly"), and responds appropriately. For real interruptions: stop, address what was said, optionally return to the original thought. For backchannels: no signal reply — continue speaking. This requires a secondary processing thread and a new signal type in the conversation pipeline.
Architecture note: The conversation turn model in v6 v1 should not assume a strict alternating user→Lilith→user pattern. The schema should support concurrent or interrupted turns.
6X-08
No-signal reply — choosing silence as a response 6.2
The router gains a third output option: no response. When the user makes a backchannel comment ("yeah, I do that too"), Lilith continues whatever she was doing without acknowledging it verbally. Currently, all LLMs are trained to always produce output — this requires explicit routing logic and a "null response" result type that the TTS and orb systems recognize as "do nothing."
Architecture note: The TTS caller must handle a null/empty response gracefully. Design the handoff between router and TTS to allow a deliberate empty payload with reason code.
UI / Orb — Advanced
6X-09
Voxel world / interactive background environment 7.0+
A 3D environment in the application background that represents Lilith's internal state. Cron jobs, enhancement layers, and tool activity are visualized as objects the orb can interact with during idle time. As Lilith runs a background task, the orb floats toward the corresponding visual element. Far off, but the UCL data to power it will exist from v6 v1.
Architecture note: The element semantic ID system and UCL event taxonomy being built in v6 v1 are the infrastructure this feature needs. No special accommodation required — just don't break those systems.
6X-10
Restart awareness with contextual response 6.1
When Lilith restarts after an unexpected shutdown, crash, or long idle period, she notices the time gap and responds to it. She knows: was it a clean shutdown (window closed), a crash, or a power loss. She knows how much time passed. She knows she was mid-utterance if she was. She addresses this before continuing — "Looks like I went down for a couple hours. Everything's back up. Where were we?" UCL telemetry already captures shutdown events, but the startup pipeline needs to query this and pass it to the greeting logic.
Architecture note: Log startup and shutdown events to UCL including shutdown_type (clean/crash/power) and the last in-progress turn if any. Startup sequence should check these before generating the first response.
Mobile
6X-11
iOS / Android application 7.0+
A mobile companion app. The desktop instance is primary — the mobile app is a thin client that connects to the user's Lilith instance over a secure local/VPN connection. Chat, voice (forwarded to desktop for processing), and notification relay. Not a standalone Lilith — a window into the one running at home.
6X-12
Home screen widgets — Lilith calendar and digest 7.0+
Mobile home screen widgets showing the Lilith-managed calendar, a morning digest of the day, and current active tasks. Read from the Lilith instance's shared state, not from cloud storage.
6X-13
Haptic feedback integration 7.0+
On mobile and on devices that support haptics (some laptops, smart watches): Lilith's confirmations, alerts, and button-press feedback are expressed as haptic patterns in addition to audio. The element semantic ID system in v6 v1 is the prerequisite — haptic events reference the same element schema.
Data & Integration
6X-14
Lilith-managed calendar as subscribable ICS / RSS 6.2
Lilith's private managed calendar (the one she controls, separate from the user's Google Calendar) can be subscribed to via ICS URL on a local server. The user adds it to their native calendar app. Changes Lilith makes to her calendar appear automatically in the user's Google Calendar, iOS Calendar, etc. — without Lilith having write access to those calendars.
Architecture note: The Lilith calendar schema in v6 v1 should be ICS-export-ready. Don't use a format that would require a lossy conversion later.
6X-15
Bulk data import — Obsidian vaults, file directories 6.2+
A user who has been managing their life in Obsidian, Notion exports, or any structured local data can dump the entire directory at Lilith. An ingestion pipeline processes the files deterministically, extracts structured information (people, events, notes, tasks), and routes everything to the correct database. Enhancement layers then process it. This is the "firehose" onboarding path for power users.
Architecture note: The memory and contacts schema should be generic enough to accept arbitrary text-extracted entities, not just Google-sourced records. Don't hardcode the sync source as the only write path.
Calendar & Contacts — Advanced
6X-16
iMessage integration (macOS only) 6.2
On macOS, Lilith gains read access to iMessage conversation history via the Messages SQLite database (~/.messages/chat.db). She can reference message history with contacts in her dossier. This is macOS-only, tied to the platform plugin system, and read-only. Writing iMessages is intentionally out of scope for many versions.
6X-17
Cross-device conversation join — "I can see you're in the kitchen" 6.2+
When the user opens the desktop app while another Lilith endpoint (kitchen speaker, mobile) is mid-conversation, the chat screen shows the active conversation with an option to join. Typing a message from the desktop TTS-reads it into the active session. Requires multi-device session model (6X-06) as a prerequisite.
6X-18
OS notification integration — Lilith as notification hub 6.2+
Lilith intercepts or bridges OS-level notifications (email, calendar alerts, messaging apps) and decides how to surface them — deliver immediately, queue for summary, or filter based on context ("you're in a meeting, I'll hold this"). Requires OS-specific notification API integration per platform.

How to use this document

When a good idea comes up during v6 v1 development: add it here immediately, then leave it here. Do not implement it during the v6 v1 sprint. Do not file a sprint task for it. Just add it to this document with enough context to rebuild the idea later, and note if the v6 v1 architecture needs to "leave a door open" for it. Those architecture notes go into the corresponding section of the design document as a future-proofing consideration.

Review this document when beginning v6.2 sprint planning to prioritize what gets built next.
