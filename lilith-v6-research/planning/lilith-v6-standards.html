Lilith v6 — Non-Negotiable Standards

These rules apply to every line of code, every sprint task, every agent session. No exceptions.

This document is law. Every sprint task starts by reading this. Every pull request is reviewed against this. Standards here are not suggestions — they are the floor below which code does not ship. If a standard creates friction with a specific implementation, the implementation changes, not the standard. If a standard needs to change, it is a design decision requiring a documented design session.
Infrastructure Standards
Path handling — pathlib.Path everywhere
Rule: All file paths use pathlib.Path. No string concatenation for paths. No os.path.join. No hardcoded directory separators (/, \). Every path is constructed from a base Path object and appended with /.
Why: String paths fail silently across platforms. pathlib is type-safe, cross-platform, and readable. Enforcing this from the start prevents an entire class of Windows bugs.
No exceptions. Even one-liner scripts in the project use pathlib.
OS-appropriate data paths
Rule: Data, config, and log paths use OS-appropriate conventions: ~/.local/share/lilith/ on Linux, ~/Library/Application Support/Lilith/ on macOS, %APPDATA%\Lilith\ on Windows. A single platform module returns the correct base path — no hardcoded paths anywhere else.
Why: Hardcoding Linux paths breaks every macOS and Windows user. Centralizing the logic means one fix works everywhere.
Module dependency direction
Rule: Dependencies flow downward only. UI imports from logic layer. Logic imports from data layer. Data layer imports from platform module. Nothing in the data layer imports from the logic or UI layer. Circular imports are bugs, not workarounds.
Why: Circular imports create initialization order problems and make testing impossible without mocking the entire stack.
Telemetry Standards
No blind spots
Rule: Every system event that has behavioral significance is logged to the UCL database. If Lilith cannot answer a question about her own behavior by querying the UCL, that is a bug. There is no such thing as "too much telemetry" — there is only telemetry that hasn't been enhanced yet.
Why: Lilith's metacognitive capability depends entirely on the completeness of the UCL. A blind spot is a place where she cannot self-debug.
log_event() on every write path
Rule: Every function that writes to a database, calls an external API, modifies application state, or produces an artifact must call log_event() at the end of its success path. This is a checklist item on every PR.
Why: Writing without logging means the action exists but Lilith cannot perceive it. She needs to perceive everything she does.
No exceptions. Including read operations that modify state (cache writes, sync token updates).
All UI elements have semantic IDs
Rule: Every interactive UI element — buttons, input fields, list items, navigation tabs, settings controls — has a stable semantic ID registered in a shared schema. The ID is present at render time, exposed to the backend event bus, and used in telemetry. IDs are human-readable and stable across app updates.
Why: Telemetry that says "user tapped element_0x3f" is useless. Telemetry that says "user tapped settings.voice.stt_model" is actionable. The same IDs enable Lilith's UI control (highlight, navigate, demo mode).
Hot-path deterministic enhancement only
Rule: Data written to the UCL in the hot path (during a live interaction) may not be enhanced by an LLM call in the same path. Hot-path enhancement must be deterministic — template-based, rule-based, or computed from fixed formulas. LLM-based enhancement runs in background jobs only.
Why: An LLM call in the hot path adds 500–3000ms latency. The conversational mode latency budget is 3 seconds total. Using any of that budget for telemetry enhancement breaks the experience.
Telemetry never crashes the main path
Rule: log_event() catches its own exceptions. A telemetry write failure is logged to stderr and counted, but never propagates to the calling code. The application continues running even if the UCL database is temporarily unavailable.
Why: Telemetry is observability infrastructure. Crashing the application because of a telemetry failure is worse than missing an event.
Security Standards
SQLCipher for all databases from first migration
Rule: All databases use SQLCipher with AES-256 encryption. No plaintext SQLite anywhere in the application. This applies from the first migration — retrofitting encryption is not acceptable. The six databases are: ucl_db, calendar_db, contacts_db, finance_db, memory_db, conversation_db.
Why: Encrypting databases after they've been created with user data requires a complete re-migration and user disruption. Starting encrypted costs nothing and protects everything.
OS keychain for all credentials
Rule: API keys, OAuth tokens (access and refresh), the database passphrase, and any other secret are stored in the OS keychain via python-keyring. They are never written to .env files, config files, code, or the application databases. .env is for local development scaffolding only and is never committed.
Why: Config files and code get committed. Databases get backed up to other places. The OS keychain is the only storage that is both persistent and OS-protected.
External APIs are read-only in v6 v1
Rule: Lilith may read from Google Calendar and Google Contacts. She may not write to them in any circumstance in v6 v1. Lilith manages her own separate calendar and contact roster. Writes to user's external accounts require an explicit, separate design decision in a future version.
Why: Write access to a user's calendar or contacts, if buggy, causes irreversible data loss. The read-only constraint eliminates this entire risk category while still enabling the core use case.
No PII in LLM cloud context without awareness
Rule: Before any data reaches a cloud LLM, the type of data being sent is known and considered. Contact home addresses, account numbers, and other high-sensitivity fields must not appear in cloud-bound context blocks unless explicitly required and noted in the design. The privacy obfuscation layer (future) will eventually handle this automatically.
Why: Cloud LLMs ingest and potentially use training data. Data sent to the cloud cannot be recalled. This is a one-way door that should be managed intentionally.
No raw data in LLM context window
Rule: The LLM never reads raw database rows. All data presented to the LLM passes through L1 or L2 enhancement. This is not a preference — it is an architectural invariant enforced by routing all context assembly through the Context Window Manager, which only reads from enhancement outputs.
Why: Raw rows are poorly formatted for LLM consumption, often contain irrelevant columns, and can expose more data than needed. Enhancement layers are the single place where "what the LLM sees" is controlled.
Voice Standards
STT and TTS are non-negotiable minimums
Rule: Whisper (STT) and Kokoro (TTS) run on every supported machine, including Tier 0 (cloud-only inference). There is no "text-only mode." There is no user option to disable voice to free up resources for local inference. Lilith is a voice-first application. Period.
Why: The differentiation of Lilith vs existing AI chat applications is the conversational, present experience. Removing voice removes that. Any machine that cannot run STT and TTS does not meet the minimum system requirements and will not support Lilith.
No exceptions. If a user wants to type at a keyboard and read text, they should use Claude.ai, ChatGPT, or Grok. Those products already exist.
Tier determines inference routing, not voice presence
Rule: Hardware tiers control whether local inference is available for the conversational and artifact paths. They do not control whether voice is present. Tier 0 = cloud inference only, but STT and TTS run locally. Tier 1 = local inference for conversational path added. Tier 2 = local inference for both paths.
Why: Conflating voice capability with inference tier would mean lower-hardware users get a degraded product. Whisper tiny runs on any machine that meets minimum specs.
Data & Enhancement Standards
Enhancement jobs never block the main path
Rule: L1 and L2 enhancement jobs run as background tasks with lower priority than all interactive path code. If there is resource contention, enhancement is preempted. An enhancement job failure degrades context quality but never causes a user-facing error.
Why: Enhancement latency is measured in seconds to minutes. Conversational mode latency budget is 3 seconds total. These cannot share a thread.
Staleness is acknowledged, not silently served
Rule: Every context block has a defined staleness threshold. The Context Window Manager checks freshness before injecting a block. Stale blocks are either flagged with a staleness note in the context or withheld entirely — never served silently as if current.
Why: Stale data presented as current causes Lilith to give confidently wrong answers. It erodes trust faster than simply saying "I don't have fresh data on that."
Artifact lifecycle is tracked end-to-end
Rule: Every artifact has a tracked lifecycle: created → stored (temp buffer with TTL) → handoff announced → user confirmed (telemetry: file opened/tapped) → saved or expired. A verbal handoff from Lilith does not count as confirmation. Telemetry must show the user accessed the artifact.
Why: Lilith saying "I put that in your folder" while the user had walked away is not a successful delivery. Delivery confirmation must come from user action, not agent action.
Documentation Standards
Every feature has a wiki page
Rule: Every new feature, module, or significant function gets a wiki page in the locally-hosted HTML documentation. The wiki page must exist before the feature is marked complete. It contains: purpose, logic summary, inputs/outputs, which systems it touches, known constraints, and developer notes. No code pasted — just links to the source location.
Why: Documentation written after the fact is documentation that doesn't get written. Building it into the done-criteria means it always exists.
Mermaid diagrams for all data flows
Rule: Any process that passes data through more than two components gets a Mermaid flowchart in its wiki page. Node labels in the diagram link to their respective wiki pages.
Why: Prose descriptions of multi-component flows are almost always misunderstood. Diagrams are the single fastest way for a new agent or developer to understand the system.
docs/history entry on direction changes
Rule: Any time a chosen approach is abandoned, a technical direction is reversed, or a design decision is changed mid-sprint, a docs/history entry is written explaining what was tried, why it failed, and what was chosen instead. This is a sprint done-criteria item, not an optional debrief.
Why: The v5 build spent significant time debugging problems that stemmed from abandoned approaches whose failure modes were never documented. The history directory exists to prevent re-entering dead ends.
Licensing Standards
Every dependency permits commercial SaaS distribution
Rule: Before any package is added to requirements.txt, its license is checked against the license audit (RES-15). The license must explicitly permit: (a) use in a closed-source desktop application, (b) distribution to end users, and (c) the developer selling a subscription service built around the application. Copyleft licenses (GPL, AGPL) require explicit evaluation before use.
Why: Discovering a license incompatibility after integration means a rewrite. Checking first costs 5 minutes per dependency.
PySide6 LGPL compliance — confirmed, no commercial license required
Rule: PySide6 6.11.1 is distributed under LGPL v3. Lilith distributes via PyInstaller --onedir (never --onefile), includes GPL/LGPL license texts in the licenses/ folder, and shows a Qt attribution notice with clickable Qt source link in the About dialog. Qt Quick 3D (GPLv3-only) is NOT used — the orb is implemented with ModernGL 5.x (MIT) embedded in a QOpenGLWidget. No commercial Qt license is needed. This was the pre-condition for the frontend build starting — it is now resolved. See RES-01 and RES-03.
Why: Confirmed via research sprint (RES-01, RES-03, RES-15). LGPL v3 is fully compatible with closed-source commercial distribution given --onedir packaging. ModernGL (MIT) eliminates the only remaining GPLv3 exposure (Qt Quick 3D).

Sprint task done-criteria — applies to every task, every sprint

A task is not complete until all of these are true.

  • Every new write path has a log_event() call at its success exit
  • No credentials in any file (code, config, .env) — all in keychain
  • All new databases use SQLCipher — no plaintext SQLite created
  • All paths use pathlib.Path — no string concatenation
  • Tests written and passing for all new functions that touch a DB or external API
  • Wiki page created for the feature/module implemented
  • CLAUDE.md updated if any architecture decision was made or changed
  • docs/history entry written if a direction was changed or abandoned
  • All new UI elements have semantic IDs registered in the element schema
  • Enhancement jobs that touch the new feature are tested for non-blocking behavior
  • Any new dependency has been license-checked against RES-15 criteria
