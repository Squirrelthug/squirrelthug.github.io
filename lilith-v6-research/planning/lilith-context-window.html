How Local Models Actually Process a Prompt

mistral-nemo:12b · Ollama · RTX 3060 12GB

When you send a message to Lilith and she routes it to the local model, here is what physically happens on your GPU — in full, not abstracted:

The KV Cache: Your Biggest Cost

Every transformer model works by computing attention — how much each token should "look at" every other token. The intermediate results of this computation (called Key and Value matrices, or the KV cache) are stored in VRAM so they don't have to be recomputed for tokens already processed.

  PROMPT ARRIVES
        │
        ▼
┌───────────────────────────────────────────────────┐
│  PREFILL PHASE  (slow — proportional to length)  │
│                                                   │
│  For every token in your prompt:                  │
│  ┌────────┐    ┌────────┐    ┌────────┐           │
│  │token 1 │───▶│token 2 │───▶│ ... N  │           │
│  └────────┘    └────────┘    └────────┘           │
│       │              │             │              │
│       ▼              ▼             ▼              │
│  [K₁,V₁]       [K₂,V₂]      [Kₙ,Vₙ]  ← KV cache│
│                                                   │
│  All N tokens run through all 40 layers in one   │
│  pass. Bigger prompt = more VRAM, more time.      │
└───────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────┐
│  DECODE PHASE  (one token at a time)              │
│                                                   │
│  NEW token ◀── attends to all KV cache           │
│  NEW token ◀── attends to all KV cache           │
│  ...                                              │
│                                                   │
│  Each output token = one GPU pass. Fast per-     │
│  token, but must read the ENTIRE KV cache.       │
└───────────────────────────────────────────────────┘

Why 32K Context Made Lilith Take 47 Seconds

When you bumped OLLAMA_NUM_CTX from 8192 → 32768, Ollama pre-allocates the full KV cache at startup for that window size. Even if your prompt is only 5K tokens, it reserved memory for 32K. This is why the first real 22K-token request took 47 seconds — the prefill phase had to process 22,000 tokens across all 40 layers of a 12B parameter model on a single GPU.

Key Insight
The local model has no persistent memory between requests. Every time you send a message, the entire conversation history + all context blocks are re-prefilled from scratch. There is no "warm" state. The KV cache is built fresh every turn and discarded after the response.

What This Means for Lilith's Local-First Vision

The local model is fast at short conversations about simple things. It degrades quickly as context grows because the prefill cost is linear: double the tokens, roughly double the time. The enriched, dense context blocks Lilith was built around are actually enemies of local inference speed — they're great for quality, terrible for latency.

The only way to make local inference fast again is to make the prompt dramatically smaller — not by removing information from the system, but by only injecting the information relevant to this specific query. That's exactly what the context window manager was always supposed to do.

How Claude (Cloud) Handles It — Prompt Caching

Anthropic API · claude-opus-4-6 · Already wired in Lilith

Cloud models face a different version of the same problem: you're paying per-token, every turn. Sending the same 3,000-token system prompt with every message means you're billed for those 3,000 tokens repeatedly, even when nothing changed.

Anthropic Prompt Caching — How It Works

Anthropic's caching system lets you mark parts of the prompt as cacheable by attaching a cache_control: {type: "ephemeral"} flag. When the same cached block is seen again (within a 5-minute TTL), Anthropic serves it from their cache instead of re-processing it — and charges you at a 90% discount (cache read tokens cost 0.1× normal input price).

  TURN 1  ─────────────────────────────────────────────
  ┌─────────────────────────────────────────────────┐
  │ SYSTEM PROMPT (stable)        cache_control: ✓  │  ← Anthropic caches this
  │  • EDICTS.md                                    │
  │  • VOICE.md                                     │    Full price this turn
  │  • TOOLS.md  (2,500 tokens)                     │
  │  • LILITH.md                                    │
  ├─────────────────────────────────────────────────┤
  │ VOLATILE BLOCK                (no cache flag)   │  ← Always re-sent
  │  • Current date & time                          │
  │  • Calendar, facts, finance blocks              │
  │  • Recall context                               │
  └─────────────────────────────────────────────────┘

  TURN 2  ─────────────────────────────────────────────
  ┌─────────────────────────────────────────────────┐
  │ SYSTEM PROMPT (stable)        cache_control: ✓  │  ← CACHE HIT — 90% cheaper
  ├─────────────────────────────────────────────────┤
  │ VOLATILE BLOCK                                  │  ← Full price
  │  • New datetime, updated context                │
  └─────────────────────────────────────────────────┘
Already Wired
Lilith's cloud router (modules/router/cloud.py line 188–200) already implements this correctly. The PersonalityEngine inserts a CACHE_BREAK marker between the stable and volatile sections. The router splits on that marker and attaches cache_control to the stable block only. This is working right now.

Cache TTL and the 5-Minute Problem

Anthropic's cache TTL is 5 minutes. If more than 5 minutes pass between turns, the cache expires and you pay full price again on the next turn. For a conversation that's actively happening (back-and-forth every minute), caching saves significantly. For Lilith running as an always-on assistant with gaps between interactions, cache hits will be inconsistent.

What About Context Window Limits for Claude?

Claude Opus 4 supports a 200K token context window. This is why routing to cloud solves the immediate problem — you're not hitting a hard wall. But you're paying for every token you send, so the economic argument for trimming context is just as strong as the performance argument for local.

The Irony
The recall system is a context window manager — it retrieves only what's relevant. But it sits alongside 5 other blocks that are always injected wholesale. The architecture has the right piece; it just needs to be the primary retrieval mechanism rather than an addendum to static dumps.

The Core Problem: Static Context, Dynamic Needs

Why the current architecture can't scale

The gap between what Lilith has and what you described is a single idea: the context assembled for every message should be shaped by what the message is asking for.

What Happens Now

  • "What's the weather?" → sends calendar + finance + people
  • "Remind me about my goals" → sends weather + finance data
  • "Help me debug this code" → sends Zoe's birthday + grocery spending
  • All 6 blocks injected regardless
  • 50-turn history loaded unconditionally
  • Local model gets 22K tokens → 47s response

What It Should Be

  • "What's the weather?" → sends only facts block
  • "Remind me about my goals" → sends goals + recent relevant turns
  • "Help me debug this code" → sends last 5 coding turns + recall hits
  • Query classifier routes to relevant blocks only
  • History compressed, only relevant turns included
  • Local model gets 3K tokens → <5s response

The enriched data pipeline (L1/L2 enhancement, TTL-based caching, the finance store, the calendar sync, the vector recall) — all of that is the data warehouse. What's missing is the query planner that sits in front of it and decides what to pull out for each specific question.

Best-Practice Context Architectures

What research and production systems actually do

1. RAG — Retrieval-Augmented Generation

The foundational pattern. Instead of injecting all known facts, you embed the user's query, search a vector store for semantically similar content, and inject only the top-K results. Lilith's recall.db already does this for conversation history — the missing piece is applying the same pattern to the fact blocks.

  User query: "what's my car payment this month?"
       │
       ▼
  EMBED query → [0.23, -0.11, 0.87, ...]
       │
       ▼
  SEARCH each knowledge store:
  ┌────────────────┬─────────────┬──────────────┐
  │ finance.db     │ dossier.db  │ recall.db    │
  │ score: 0.91 ✓  │ score: 0.21 │ score: 0.44  │
  └────────────────┴─────────────┴──────────────┘
       │
       ▼
  INJECT only finance block + top 2 recall hits
  Total: ~400 tokens instead of 1,830

2. Query Classification (Intent Routing)

Before any retrieval, run the user's message through a lightweight classifier to determine intent category. Each category maps to a set of relevant context blocks. This is much cheaper than embedding-based retrieval and works for obvious intent signals.

Lilith's codebase already has keyword-based intent routing for triggering tools (main.py lines 155–178). Extending that same list to also govern which context blocks to include is a small addition with large impact.

3. Hierarchical Memory (MemGPT / Letta Architecture)

The most sophisticated approach — treating memory as a first-class system with explicit tiers and model-driven read/write operations. Popularized by the MemGPT paper (Packer et al., 2023). The model itself decides when to move memories between tiers.

  ┌──────────────────────────────────────────────────────┐
  │  CONTEXT WINDOW  (what the model sees right now)    │
  │  ┌──────────────┐  ┌──────────────────────────────┐ │
  │  │  CORE MEMORY │  │  WORKING CONTEXT             │ │
  │  │  • Identity  │  │  • Recent conversation       │ │
  │  │  • User facts│  │  • Injected recall hits      │ │
  │  │  • Goals     │  │  • Active task state         │ │
  │  └──────────────┘  └──────────────────────────────┘ │
  └──────────────────────────────────────────────────────┘
          │ read/write                    │ read
          ▼                               ▼
  ┌──────────────────┐        ┌───────────────────────┐
  │  ARCHIVAL MEMORY │        │  RECALL MEMORY        │
  │  Long-term facts │        │  Vector search index  │
  │  Old conv logs   │        │  (recall.db today)    │
  └──────────────────┘        └───────────────────────┘

Lilith has the tiers (hot/warm/cold/recall). What MemGPT adds is an agent loop where the model explicitly calls memory_search() and memory_insert() tools as part of generating a response. The model manages its own memory rather than having it pre-assembled by the orchestrator.

4. Sliding Window + Summarization

The simplest practical approach for conversation history. Keep the last N full turns verbatim. Summarize everything older into a compact "session summary" that replaces the raw messages. The summary gets compressed again as the conversation continues.

  Turn 1 ──── turn 50  (compressed → SUMMARY A, ~200 tokens)
                │
  Turn 51 ─── turn 100 (compressed → SUMMARY B, ~200 tokens)
                │
  Turn 101 ── turn 110 (kept verbatim, ~1,500 tokens)

  Total history in context: ~2,000 tokens
  vs. current 50-turn raw: ~7,500 tokens

Infinite Conversations — Compression & Compaction

How to never lose what matters without paying for what doesn't

You described wanting Lilith to have no "end" to a conversation — she should never forget what you said and the conversation should be able to run indefinitely without degrading. Here's exactly how that's designed to work.

The Core Misunderstanding to Correct

Infinite conversation does not mean an infinite context window. It means an infinite external memory store paired with smart retrieval, so the context window stays bounded but nothing is truly lost. Think of it like RAM vs disk: the model's context window is RAM (fast, bounded), recall.db is disk (unlimited, indexed).

  ────────────────────────────────────────────────────────

  NAIVE APPROACH (what people assume):

  Conv starts: [msg1, msg2, msg3, ... msg500]
                ←────── context window ──────→ FULL → FAIL

  ────────────────────────────────────────────────────────

  CORRECT APPROACH:

  Conv starts: [msg1 ... msg50] → ARCHIVE → embed → recall.db
                                           ↓
  Next turn:   [SUMMARY of 1-50] + [msg51 ... msg100] → ARCHIVE
                                           ↓
  Right now:   [SUMMARY of 1-100] + [last 10 turns] + [recalled hits]
               ↑                                      ↑
               ~400 tokens                        ~580 tokens
               total history: CONSTANT, ~1,000 tokens regardless of
               how long the conversation has been running
  ────────────────────────────────────────────────────────

Compression Quality Is Everything

The compression step is where naive implementations fail. If you summarize poorly, you lose context and Lilith seems to "forget." The right approach uses lossy compression for specifics, lossless for facts:

TypeCompress toExample
Chit-chat / small talkDiscard"How's your day?" → dropped
Task discussionOne sentence summary"Discussed sprint plan for finance module"
Decisions madeFact stored in dossier"Steven prefers GIF over MP4 for demos"
Corrections givenFeedback memory entry"Don't use NodeJS.Timeout — use ReturnType<typeof setTimeout>"
Technical detailsEmbed in recall.dbFull searchable text, retrieved when relevant

For Local vs Cloud: Same Logic, Different Ceiling

The compaction pipeline works identically for both. The local model just has a lower "keep verbatim" budget (maybe 5 recent turns instead of 20) and a smaller overall context ceiling. The same compression artifacts go into recall.db either way, so nothing is truly lost — it just becomes retrieval-dependent rather than always-present.

What Claude Code Does (Anthropic's Published Approach)

The trillion-dollar company's answer to the same problem

You mentioned Claude releasing something — you're thinking of Claude Code and its /compact command. This is Anthropic's production solution to exactly the conversation-length problem, and it's worth dissecting because it's what you should build.

Claude Code's Compaction Strategy

When the context approaches the limit, Claude Code runs a structured summarization pass. The key insight: it doesn't just truncate or summarize everything equally. It preserves different things with different fidelity:

  WHAT CLAUDE CODE PRESERVES (verbatim):
  ├── All files currently open in editor
  ├── The most recent user message and response
  ├── Any pending tool results
  └── Explicitly pinned context (CLAUDE.md)

  WHAT IT COMPRESSES (to summary):
  ├── Previous conversation turns (collapsed to narrative)
  ├── Earlier tool call results (just the outcome, not the output)
  └── Superseded file contents (only latest version kept)

  WHAT IT DISCARDS:
  ├── Redundant exchanges
  ├── Error messages that were resolved
  └── Intermediate reasoning that led nowhere

The structural principle is: keep current state, compress history, discard resolved noise. Current state is always verbatim because the model needs it to act correctly right now. History is summarized because you might need to reference it. Resolved noise has no future value.

CLAUDE.md as Persistent Context

Anthropic's other approach is the CLAUDE.md file — a human-curated, always-loaded document that survives compaction. It's essentially the same concept as Lilith's EDICTS.md and MEMORY.md combined: stable facts about how to operate that don't need to be inferred from conversation history because they're explicitly authored.

Lilith already has this
Your vault's identity.md, EDICTS.md, VOICE.md, and MEMORY.md are exactly this pattern. They're the stable, always-loaded, human-curated context that survives any compression. The compaction strategy just needs to know not to touch these.

What to Build First

Ordered by impact-to-effort ratio
Already Done

Recall retrieval, context budget gate, Anthropic prompt caching

These are live. The recall system is your best context management tool. The Anthropic cache split is already wired correctly in cloud.py.

Highest Impact — Do First

Query-intent block selection

Extend the existing keyword intent list in main.py to also produce a block_whitelist. Weather questions skip finance and people. Coding questions skip everything except recall. This alone likely cuts local context from 22K → 5K tokens, making the local model useful again. One file, additive change.

Next

History compression (sliding window + summary)

After a conversation exceeds 20 turns, run a background summarization pass over the oldest 15 turns. Store the summary as a special "session_summary" message type. Load it instead of the raw turns. History budget drops from 7,500 → 2,000 tokens permanently.

After That

Post-response fact extraction

After every response, run a lightweight pass to extract any decisions, preferences, corrections, or new facts mentioned in the conversation. Write confirmed facts to dossier automatically. This feeds the long-term memory system without requiring you to explicitly say "remember this."

Later

Full MemGPT-style agent memory tools

Give the model memory_search(query) and memory_save(key, value) tools it can call during inference. Let the model manage its own retrieval rather than the orchestrator pre-assembling everything. This is the full vision — but the earlier steps need to be solid first.

Bottom Line
The context window manager you built four years ago and have been carrying forward is architecturally correct. The recall system, the L1/L2 enrichment pipeline, the TTL caching, the memory tiers — all right. What's missing is the selector layer between your data warehouse and the assembled prompt. Everything upstream of that is already built and working. The gap is smaller than it feels.
