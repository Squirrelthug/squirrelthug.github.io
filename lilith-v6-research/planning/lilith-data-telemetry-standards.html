1. Google SSO Onboarding — The Fastest Path to User Value

The best onboarding flow for a personal intelligence app isn't a form. It's a handshake with data the user already has. Google is the right first integration because almost everyone has a Google account and their most important data is already there: contacts, calendar, email.

  ONBOARDING FLOW (v6 v1)

  User installs Lilith
       │
       ▼
  "Sign in with Google" button
       │
       ▼
  OAuth 2.0 flow (browser opens, user consents)
  Requested scopes:
  ├── contacts.readonly      → People screen
  ├── calendar.readonly      → Calendar screen
  └── gmail.readonly         → Optional (later — richer context)
       │
       ▼
  Tokens stored in OS keychain (libsecret / Keychain / DPAPI)
       │
       ▼
  Initial sync:
  ├── Contacts → people DB
  ├── Calendar events (next 30 days + past 7 days) → calendar DB
  └── "Lilith is getting to know you..." (orb processing animation)
       │
       ▼
  First conversation starts with Lilith
  already knowing who you are (by name),
  your next appointment, and the people in your life

The power of this: within 60 seconds of installation, Lilith can say "Good morning [name]. You've got [appointment] at [time] with [person]." That's the moment the user feels it. Not a tutorial. Not a form. An immediate demonstration of value.

What explicit permission looks like

The consent screen should be transparent and specific — not a wall of legal text. Something like:

  • Contacts: "I'll read your Google contacts so I know who matters to you. This stays on your device."
  • Calendar: "I'll read your calendar so I can tell you what's coming and help you prepare."
  • Email (optional, later): "I'll read email subject lines and senders to give you briefings. I won't store the bodies."
The privacy-first framing
"This stays on your device" is true if the sync writes to SQLCipher local DB and the tokens stay in the OS keychain. The cloud never sees your contacts. The LLM call that uses contact names goes out to the cloud, but the raw data doesn't. This is meaningfully different from most apps, and worth saying clearly in the onboarding.

What comes later (not v6 v1)

  • Apple iCloud contacts/calendar (macOS version, v6.1)
  • Microsoft (Outlook / Exchange) — the corporate user path
  • Apple/Google Photos — "show me a photo from my trip to Denver"
  • Self-hosted migration: move from Google-hosted to Lilith-hosted Nextcloud, CardDAV, CalDAV

2. The Data Sovereignty Path

Most users will start with their data hosted by Google, Apple, or Microsoft. That's fine. The goal isn't to demand they self-host on day one — it's to give them a clear path from "Google hosts everything" to "I own everything," with Lilith managing the migration at whatever pace the user chooses.

Stage 1 — Default start
Cloud-sourced, local-stored

Data lives at Google/Apple/Microsoft. Lilith syncs a local copy (encrypted SQLCipher) and reads from there. You're still cloud-dependent for the source of truth, but Lilith adds a local encrypted layer. You can use Lilith offline; it just won't have new data from cloud sources until it reconnects.

Stage 2 — Hybrid control
Lilith-as-router

Lilith manages the pipeline. User can see where each data category lives and choose routing: "keep contacts at Google, migrate calendar to self-hosted CalDAV." The local encrypted copy is always authoritative for Lilith's reads — the cloud is now a sync source, not the source of truth.

Stage 3 — Full sovereignty (opt-in)
Local-first, self-hosted

User runs their own Nextcloud/Syncthing/Vaultwarden (or Lilith hosts its own sync server). Google is no longer in the loop. All data on devices you control. Lilith still handles sync across your own devices (desktop → phone → tablet) via E2E encrypted sync. This is the Obsidian model: you own the files, Lilith owns the experience layer.

Lilith doesn't force migration. She makes it easy when the user is ready. The UI for this is a "Data" screen that shows every data category, where it lives, and a button to migrate it. The backend supports all three stages simultaneously — different categories can be at different stages.

3. The Chat Screen in the Vertical Slice

The chat screen is the universal conversation view. Every interaction with Lilith — whether it started via voice, typed in the desktop app, or came from a voice unit in the kitchen — flows through the same conversation history. The chat screen shows all of it, organized by source.

What the chat screen needs to show

  • A standard chat UI (messages, timestamps, read receipts) — not revolutionary, just clean
  • Visual tagging of message source: which device or endpoint originated this message
  • Cross-device continuity: a conversation that started on the kitchen speaker can be continued in the desktop chat window. Context carries over.
  • Voice vs. text visual distinction: voice messages might show a waveform icon or audio-first styling
  • Artifact mode messages look different from conversational replies: longer, formatted, attached results

Cross-device conversation continuity

When a user picks up the conversation on a different device, Lilith needs to know two things: what was said before, and which device is now active. The context block for a resumed conversation should include:

  • Last N turns of conversation (from the persistent conversation log)
  • Any artifact results from the previous segment that were referenced but not acted on
  • The current device's capabilities (voice only? full screen? touchscreen?)

The conversation doesn't restart. Lilith doesn't say "how can I help you today?" She says "picking up where we left off" if the gap is short, or "it's been a while, here's what we had going" if it's been hours.

4. Source & Endpoint Management

Every message Lilith receives has an origin: a device, a mode (voice/text), and a session. Tracking this from the beginning prevents a mess later — when Lilith has 4 devices talking to her, you want to know which conversation came from where and be able to route responses back to the right endpoint.

  SOURCES / ENDPOINTS

  Desktop app
  ├── Voice input (microphone → STT → text)
  └── Chat window (typed text)

  Phone app (future — v6.x)
  ├── Voice input
  └── Chat window

  Kitchen speaker unit (appliance)
  └── Voice only (no screen)

  Browser / web thin client (future)
  └── Chat window only

  Each source has:
  ├── endpoint_id     (unique device identifier)
  ├── endpoint_type   (desktop, phone, speaker, browser)
  ├── capabilities    (voice, screen, touch, display_size)
  └── session_id      (current conversation session)

The endpoint model matters for response formatting. A response going to the kitchen speaker has no screen — it must be pure audio, no markdown, no "here's a table." A response going to the desktop can be a full artifact with formatted results. Lilith needs to know the destination before formatting her response.

# What the routing layer needs to know before generating a response
response_context = {
    "endpoint": "kitchen_speaker",
    "capabilities": ["voice_only"],
    "preferred_response_type": "conversational",
    "max_tokens": 150,         # Short for voice-only endpoints
    "markdown_supported": False
}

response_context = {
    "endpoint": "desktop_chat",
    "capabilities": ["screen", "voice", "touch"],
    "preferred_response_type": "artifact",  # Can show structured results
    "max_tokens": 2000,
    "markdown_supported": True
}

5. Telemetry as a First-Class Design Principle

This is not a "nice to have" observability feature. Telemetry is Lilith's nervous system. Without it, she's blind to her own behavior. With it, she can debug herself, tune herself, and eventually improve herself. The UCL (Universal Communication Log) database is where every event goes. It's the only way to answer: what did Lilith actually do, when, why, and did it work?

What metacognitive telemetry means

Lilith queries her own telemetry database. She doesn't need a human to look at a dashboard and tell her "you've been slow lately." She can look herself:

  Lilith receives: "You seem slower today"
         │
         ▼
  Lilith queries UCL DB:
  SELECT avg(response_latency_ms), event_type, endpoint
  FROM ucl_events
  WHERE timestamp > now() - interval '24 hours'
  GROUP BY event_type, endpoint
  ORDER BY avg_latency DESC
         │
         ▼
  "My average response time in the last 24 hours was 4.2 seconds,
   up from 1.8 seconds yesterday. The slowdown is concentrated in
   calendar tool calls — they're averaging 6.1 seconds each.
   The calendar sync ran at 11:15 PM last night and pulled
   847 events. That's probably the culprit."

This is not science fiction. It's a read query against a local SQLite database. The LLM has access to the query tool; the database has the data. The only requirement is that the data is actually there — which means every event must be logged when it happens, without exception.

What the UCL stores

  ucl_events table:

  id              INTEGER PRIMARY KEY
  timestamp       DATETIME (UTC, microsecond precision)
  event_type      TEXT (dot-namespaced: "calendar.sync", "tool.call", "llm.response")
  endpoint_id     TEXT (which device/session)
  session_id      TEXT (conversation session)
  payload         JSON (event-specific data)
  duration_ms     INTEGER (for latency tracking)
  success         BOOLEAN
  error_msg       TEXT (null if success)

Event taxonomy — the dot-namespaced system

NamespaceEventsWhy it matters
llm.* request, response, token_count, latency, model_used, cache_hit Cost tracking, latency profiling, cache performance
tool.* call, result, latency, error, tool_name Which tools get used, which fail, which are slow
voice.* stt.started, stt.complete, stt.latency, tts.started, tts.complete, amplitude_avg Voice pipeline quality, latency by model
router.* classify.result, mode_selected (conversational/artifact), confidence How often the classifier is right; what gets routed where
context.* blocks_selected, tokens_used, blocks_skipped, intent_detected Context window efficiency; which blocks get used
sync.* calendar.started, calendar.complete, contacts.started, contacts.complete, records_pulled Sync health; how fresh the data is
user.* message_sent, conversation_started, conversation_ended, session_length Usage patterns; which features get used
memory.* write, read, evict, recall_miss Memory system performance; recall quality
personality.* truth_detected, truth_written, truth_applied, model_call, model_latency How the personality system is working; truth capture rate
error.* type, source, recoverable, user_impact Error patterns; which failures affect users

6. Enhancement Layers — L1/L2 Pattern

Raw telemetry data is granular event logs. Enhancement layers transform raw events into purpose-built views that specific consumers can query without having to do complex aggregation themselves.

  DATA PIPELINE

  Raw event (logged at mutation point)
  ├── ucl_events.calendar.sync  {records: 847, duration_ms: 12000}
  ├── ucl_events.tool.call      {tool: "get_calendar", latency_ms: 6100}
  └── ucl_events.llm.response   {tokens: 1200, latency_ms: 2100}
         │
         ▼
  L1 Enhancement (run periodically or triggered)
  ├── calendar_sync_stats       {avg_records_per_sync, avg_duration, last_sync_age}
  ├── tool_performance          {p50_latency, p95_latency, error_rate per tool}
  └── llm_usage_daily           {total_tokens, total_cost_usd, avg_latency}
         │
         ▼
  L2 Enhancement (built on L1 — narrative/insight layer)
  ├── "Calendar sync is healthy" / "Calendar sync has degraded"
  ├── "get_calendar is your slowest tool (p95: 8.2s)"
  └── "LLM cost this week: $0.47 — up 23% from last week (more artifact calls)"
         │
         ▼
  LLM reads L2 data to answer questions about system state
  (never reads raw ucl_events directly — too much noise)

Why this matters for the LLM

If Lilith reads raw UCL events to answer "am I performing well?", she'd be reading thousands of rows and doing complex math in her context window. That's slow, expensive, and unreliable. Instead, L1 runs the aggregation as a background job and writes summary stats. L2 runs interpretations on those stats. The LLM reads L2 and gets a clean, pre-computed answer.

The same pattern applies to all data domains: contacts, finance, memory. The LLM never reads raw data tables. It reads enhanced views that have been purpose-built for LLM consumption: appropriately compressed, pre-summarized, contextually relevant.

7. Sprint Standards: Telemetry & Security as Done-Criteria

A task that ships functionality without telemetry is half-done. This must be a hard requirement in every sprint, not a nice-to-have polish item.

Standard: Every sprint task done-criteria must include

For any task that writes or mutates state:

  • A log_event() call at the point of mutation — not in a wrapper, at the actual write
  • The event type uses the dot-namespaced taxonomy (new types added to EventType enum if needed)
  • Payload includes enough data to reconstruct what happened without reading the actual record
  • Duration / latency tracked if the operation is non-trivial (>50ms)
  • Error path also logs — a failed write is still an event

For any task that exposes data to the LLM:

  • The data goes through an enhancement layer (L1 at minimum), not direct raw table reads
  • The enhancement layer is documented: what does it compute? For which consumer?
  • Staleness gate: if the data is older than a defined threshold, the context block says so

For any task that adds a new user-facing security boundary:

  • Credentials go to OS keychain — never to flat files or config JSON
  • User data that's sensitive goes to SQLCipher — never to plaintext SQLite
  • Any network call that sends user data documents what leaves the device and why

The review checklist addition

When reviewing any PR (yours or a delegated agent's), the security and telemetry check is not optional:

  1. Does this PR have a write/mutation path? → Check for log_event() call on that path.
  2. Does the write path go through an already-untelemetered route? → Flag it. The gap is inherited.
  3. Does any new data flow to the LLM? → Confirm it goes through an enhancement layer.
  4. Does this introduce any new credential, token, or sensitive data storage? → Must be in OS keychain or SQLCipher.

8. OS-Specific Platform Tools (iMessage Pattern)

The principle: when Lilith is installed on a specific OS, she gets access to OS-specific capabilities that are only available there. These are plugins that activate based on detected OS, not core features.

OSPlatform capabilityWhat Lilith can do with itStatus for v6 v1
macOS iMessage (via AppleScript or private API) Read messages, send messages, search contacts/messages v6.1+ (macOS-specific build)
macOS Notification Center Native system notifications, rich notification previews v6 v1 — all OSes have this equivalent
Windows Windows 11 Widgets / Live Tiles Brief summary on desktop, incoming call notification v6.1+ (Windows-specific)
Windows Cortana integration / Windows Search Surface Lilith results in Windows Search Not planned — too invasive
Linux D-Bus / systemd System-level notifications, startup service, file watch v6 v1 — Linux primary platform
All OS keychain Credential storage (libsecret / Keychain / DPAPI) v6 v1 — core requirement

How the plugin architecture works

  STARTUP (all platforms)

  detect_os() → "darwin" | "windows" | "linux"
       │
       ▼
  load_platform_plugins(os)
  ├── "linux" → load SystemdPlugin, DBusPlugin
  ├── "darwin" → load KeychainPlugin + check iMessage access
  └── "windows" → load DPAPIPlugin, WindowsNotificationPlugin
       │
       ▼
  Available tools list updated based on loaded plugins
  LLM system prompt includes: which platform tools are available

This means the v6 v1 codebase handles all three platforms in a single repo. OS-specific tools are just conditionally loaded plugins — they're not separate builds. The installer for macOS includes the iMessage plugin code even if it's disabled by default; a toggle enables it if the user grants access.

9. Distribution, Licensing & Piracy

The Obsidian model is the right reference here. Obsidian is local-first, free for personal use, paid for commercial use. Their Sync service is the revenue driver — not the app itself. Pirates can pirate the app; they can't pirate the sync service. Obsidian made $7M ARR with 1M users using this model.

Why piracy isn't an existential threat for Lilith

  • The premium value is cloud-connected features. BYOK gives users their own API keys. Sync keeps devices in sync. These are services, not file copies. A pirated copy can't get sync without a valid license.
  • Your target user is not a pirate. Someone who wants a personal AI assistant for their health, finances, and relationships is not the demographic looking for a cracked .exe on a forum. They want something that works, is maintained, and is trustworthy.
  • Piracy is free marketing at this scale. 100 pirated copies of Lilith means 100 people know it exists. At this stage of the product, distribution is the hard problem, not revenue leakage.
  • License keys for gated features. A simple license key for "Sync" feature, validated against a Lilith API endpoint. The core app works without it. Sync doesn't. No DRM needed on the app itself.
The practical licensing model
  • Personal use: Free. App works fully for a single device. No cloud sync. BYOK for API access.
  • Personal + Sync: $8–12/month. Cross-device sync (E2E encrypted, Lilith-hosted). License key enables the sync feature.
  • Self-hosted: One-time purchase or free (open source sync server). Power user path.

10. Documentation as Part of the Plan

Documentation isn't a post-launch cleanup task. A project this complex, built across multiple sessions with AI agents, needs documentation to stay coherent. The question is which documentation belongs in the plan and which can wait.

Documentation that's part of the plan (not optional)

Document typeWhen to write itWho uses it
Sprint docs (existing format) Before implementation starts — design tasks then implementation tasks Agents doing the implementation; orchestrator reviewing it
CLAUDE.md updates Whenever a workflow rule changes — not retroactively Every agent that starts a session in this repo
docs/history/ session narratives After significant sprint completion (every few tasks, not every task) You — reading back through what was built and why
Idea register (IDEA_REGISTER.md) When a good idea comes up that can't be done now — capture it before it evaporates You — at sprint planning time
Architecture diagrams (like v6-systems.html) When the system is complex enough that verbal description fails You and Steven — visual thinkers need visual maps
API docs / tool schemas When a tool or interface is stable (not before) Agents implementing callers; future contributors

Documentation that can wait

  • User-facing docs (README, getting started, FAQ) — write after vertical slice exists and works
  • Contribution guide — write if and when there are contributors
  • API reference for external developers — write if and when the API is public
  • Changelog / release notes — write at each release point, not continuously
The system map as living document
The visual architecture document (v6-systems.html, on the Desktop) should be updated when major new systems are added. It's not one-time documentation — it's the map of the territory. When the territory changes, update the map. A map from 6 months ago showing systems that no longer exist is worse than no map. Make it part of the sprint review: "did anything architecturally change? Update the systems diagram."
