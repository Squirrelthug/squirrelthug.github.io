1. The Non-Negotiable Requirements

Before comparing frameworks, the requirements that were underspecified in the previous document and which change the ranking:

  • The orb is a 3D animated asset. Not a 2D image stack. A real-time 3D object — potentially liquid, glowing, particle-based — that moves, reacts to voice input, and feels alive. This is a rendering requirement, not just a CSS animation.
  • Lilith controls the UI. She navigates between screens, points at elements, changes pages, runs a demo mode. The backend must be able to issue UI commands. This requires a well-defined backend-to-frontend event bus.
  • Touch and mouse/keyboard are interchangeable. No friction switching between them. Touch events and pointer events handled identically. This is a design requirement that the framework must support natively.
  • The orb lives inside the application. It does not float over the OS desktop or other applications. This removes the OS-overlay difficulty entirely — the orb is a widget inside a full-screen window.
  • Premium feel on all three desktop platforms. Not "it runs." Actually looks and feels like software worth paying for.
Clarification that changes the framework comparison
The orb being inside the application (not an OS-level overlay) removes what was listed as a "medium difficulty" challenge in the previous doc. Inside a full-screen window, every framework handles an animated floating element trivially. The orb complexity shifts entirely to rendering quality — can it be 3D? Can it react in real time?

2. Why Qt6 Was Underrated

The previous doc labeled Qt6 as a "strong candidate" but placed it below Tauri and Flutter. The reasoning was incomplete. Here's the case for Qt6 that was missed:

  • Qt Quick 3D is a production-grade 3D scene system built directly into Qt. It's what game engine UIs and premium embedded device interfaces use. The orb you're imagining — a real-time 3D animated sphere with shaders and particle effects — is exactly what Qt Quick 3D was designed for. This is not a workaround. It's the primary use case.
  • Native on all platforms means Qt draws its own widgets using native platform rendering primitives. On macOS it looks like macOS. On Windows it looks like Windows. On Linux it looks like Linux. Flutter draws everything with its own rendering engine (consistent across platforms but not native). Tauri uses OS webviews (which can look native or not depending on web CSS).
  • QML (Qt's UI language) is declarative, similar to CSS + a bit of JavaScript, and is genuinely pleasant to write for animated, reactive UIs. It's not C++. You write QML for the UI, Python for the logic, and they connect cleanly via PySide6.
  • Licensing was the unstated concern that knocked Qt6 down. Qt6 under LGPL is free for open source and most commercial use if you dynamically link. PySide6 (the official Qt Python binding) is under LGPL. For a commercial desktop app, this is fine — you can distribute Lilith without paying Qt licensing fees as long as you use LGPL components.
Why Qt6 was omitted from "top two"
The implicit assumption was that web tech (React/Vue) would be easier to hire for if the project grows. That's true but irrelevant for a solo developer. For one person who needs premium 3D UI, a clean Python integration, and genuine cross-platform native feel, Qt6 is arguably the strongest choice. It was underweighted in the previous analysis.

3. Tauri v2 — Deep Look

Tauri v2
Rust shell + OS webview + web frontend (React / Vue / Svelte)

What it is

Your app's logic lives in Rust. Your UI is HTML/CSS/JavaScript running in the OS's native webview (WebKit2GTK on Linux, WebKit on macOS, WebView2/Edge on Windows). Tauri exposes a bridge so the web frontend can call Rust functions and Rust can emit events to the web frontend.

The orb in Tauri

Three.js or React Three Fiber running in a WebGL canvas. This is actually the strongest 3D option of the three frameworks — the web 3D ecosystem (Three.js, Babylon.js, shader libraries, particle systems) is enormous, well-documented, and runs at full GPU speed via WebGL. Navi-like orbs have been built in Three.js extensively. Liquid metal, plasma spheres, glowing energy fields — all standard Three.js territory.

Backend-to-frontend events (Lilith controls the UI)

Tauri's emit API sends events from Rust to the webview. The Python backend sends a WebSocket message to the Tauri Rust layer, which emits a frontend event: navigate_to("finance"), highlight_element("balance_card"), run_demo_step(3). The frontend subscribes to these events and reacts. This is a clean, well-established pattern.

Touch support

Web touch events are native. pointer events unify mouse and touch in modern web APIs. No extra work needed.

4. Flutter Desktop — Deep Look

Flutter Desktop
Google's UI framework — renders everything with its own engine (Impeller/Skia)

What it is

Flutter doesn't use OS widgets or OS webviews. It renders everything itself via a GPU-accelerated canvas (Impeller on newer targets, Skia on older). Every pixel is Flutter's. This makes it perfectly consistent across platforms but not "native" in the traditional sense — it doesn't look like macOS or Windows specifically, it looks like Flutter.

The orb in Flutter

This is Flutter's weakest point for this use case. Flutter's 3D story is improving but immature. Options: (1) flutter_scene — Flutter's official 3D package, still early; (2) Rive — excellent 2D/vector animation, not true 3D; (3) Embedding a WebGL view — possible but defeats the point of Flutter; (4) Shader-based 2.5D effects — Flutter has fragment shader support that can produce impressive orb-like effects that aren't technically 3D but can look it. A plasma/liquid sphere effect in Flutter shaders is achievable and looks great.

Backend-to-frontend events (Lilith controls the UI)

Flutter uses MethodChannel and EventChannel to communicate between Dart and native code. For the Python backend, you'd route events through a WebSocket or IPC channel that the Dart code listens to. Workable but more layers than Tauri's direct emit approach.

Touch support

Flutter was designed mobile-first. Touch handling is its native language. Touch and pointer events are unified through Flutter's GestureDetector. The best touch story of the three frameworks.

5. Qt6 / PySide6 — Deep Look

Qt6 / PySide6
Industry-standard C++ UI framework with full Python bindings — the strongest 3D + native story

What it is

Qt is the framework behind VLC, Autodesk tools, the KDE desktop, Google Earth, and the interfaces on luxury car dashboards. Qt6 with PySide6 gives you full Qt in Python. You write the UI in QML (declarative, like a more powerful CSS) and the logic in Python. Qt Quick 3D is a first-class 3D scene system built into Qt, not a plugin.

The orb in Qt

Qt Quick 3D is exactly what you want. A real-time 3D scene with a sphere, custom shaders, particle systems, and lighting — all in QML, no C++ required for the scene definition. Performance is native GPU (Metal on macOS, Vulkan on Linux, D3D12 on Windows). An animated 3D orb with a glowing aura, particle field, and reactive pulse to voice input is a beginner-level Qt Quick 3D project. This is the strongest 3D story of the three frameworks.

// What the orb looks like in Qt Quick 3D (QML)
View3D {
    PerspectiveCamera { position: Qt.vector3d(0, 0, 300) }

    DirectionalLight { color: "#4fc3c9"; brightness: 2.0 }

    Model {
        source: "#Sphere"
        scale: Qt.vector3d(1.5, 1.5, 1.5)
        materials: PrincipledMaterial {
            baseColor: "#0a2030"
            metalness: 0.9
            roughness: 0.1
            emissiveColor: Qt.binding(() => orbColor)
            emissiveFactor: 2.0
        }
        // Animate rotation, pulse scale with voice amplitude
        NumberAnimation on eulerRotation.y { from: 0; to: 360; duration: 8000; loops: -1 }
    }
    ParticleSystem3D { /* Navi glow particles */ }
}

Backend-to-frontend events (Lilith controls the UI)

This is Qt's strongest suit for Lilith's architecture. PySide6 lets Python and QML share objects directly via QProperty and signals. The Python backend emits a signal; QML reacts immediately. No HTTP, no WebSocket between frontend and backend — they're in the same process. lilith.navigate.emit("finance") in Python triggers a QML screen transition instantly.

Touch support

Qt has had touchscreen support since Qt 4. QML's MultiPointTouchArea and TapHandler handle pointer events (mouse and touch) identically. Qt was used extensively in automotive touchscreens and embedded devices — touch is first-class.

Licensing

PySide6 is LGPL v3 (the official Qt Company Python binding). Free for commercial use when dynamically linked. You can ship Lilith commercially without paying Qt licensing fees. The components you need for this project are all in the free tier.

7. The Orb — 3D Asset Requirements

The orb is the most visible differentiator in Lilith's UI. It needs to communicate: presence, intelligence, listening state, processing state, speaking state. The v5 version is a 2D art stack rotating in a circle. The v6 version should feel alive.

ORB STATES AND VISUAL BEHAVIOR

  State           Visual behavior
  ────────────    ──────────────────────────────────────────
  Idle            Slow rotation, gentle pulse, ambient glow
  Listening       Outer particle field activates, amplitude ring expands
  Processing      Surface becomes turbulent, internal light pattern shifts
  Speaking        Pulse synced to TTS audio amplitude, warm color shift
  Error           Color drains to cool, brief sharp pulse
  Artifact mode   Splits attention — core speaks while background dim glow spins

What makes it feel like Navi vs. what makes it feel like a logo

The difference between a Navi-like orb and a generic animated logo is three things: reaction (it responds to what's happening in real time), personality (it has behavior that feels intentional, not random), and physicality (it has a sense of volume and material, not just a flat lit circle).

  • Reaction: Audio amplitude from the TTS stream modulates the orb's pulse and scale in real time. When Lilith is speaking louder (emphasis), the orb reacts. When she stops, it settles.
  • Personality: Idle behavior that's not perfectly looping — slight drift, occasional flicker, a pause and reset. Makes it feel like something's in there.
  • Physicality: A specular highlight that tracks a virtual light source. Sub-surface scattering that makes it look like there's something glowing inside. Particles that orbit at different speeds.

8. Demo Mode and Lilith Controls the UI

The vision: a markdown file of instructions, Lilith reads it, understands the steps, issues UI commands, navigates the app, points at elements, runs a complete demo of her own capabilities. She's not just a voice in the app — she lives in the app.

DEMO MODE PIPELINE

  markdown file
  "Show contacts screen → say 'I've got your contacts' →
   highlight 'Sarah' → navigate to finance → speak spending summary →
   return to home"
       │
       ▼
  Lilith parses instructions (LLM call, small context)
       │
       ▼
  Generates sequence of UI commands:
  { action: "navigate", screen: "contacts" }
  { action: "speak", text: "I've got your contacts loaded up." }
  { action: "highlight", element: "contact_sarah" }
  { action: "navigate", screen: "finance" }
  { action: "speak", text: "Here's your spending..." }
       │
       ▼
  Backend emits commands → frontend executes
  (each command is an event on the backend-to-UI bus)

Implementing "Lilith highlights an element"

This requires a semantic naming system on the UI side. Every interactive or notable element has an ID that the backend knows about. When the backend emits highlight("contact_sarah"), the frontend maps that ID to a visual element and applies a highlight animation (glow, border flash, camera pan to it).

This is actually a useful discipline beyond demo mode — it forces every UI element to be addressable by name, which makes automated testing and accessibility easier too.

Tauri v2app_handle.emit("ui_command", payload) from Rust (or Python → Rust → emit)JS event listener on listen("ui_command", handler)

Qt6 is the cleanest here — no bridge, no serialization overhead, Python and QML are in the same process and signals fire synchronously. Tauri requires going Python → WebSocket → Rust → webview emit, which is two hops but still fast (~1ms).

9. Touch + Mouse Parity

The requirement: a touchscreen desktop user and a mouse/keyboard user should have no friction switching between input methods at any point. The same button works with both. The same gesture works with a trackpad, a touchscreen, and a mouse.

This is not a hard engineering problem with any of the three frameworks — they all support it. It's a design discipline problem. The things that break touch/mouse parity are:

  • Hover states that have no touch equivalent (tooltips that appear on hover but never on tap)
  • Right-click menus that have no long-press equivalent
  • Drag targets that require pixel-perfect mouse precision
  • Keyboard shortcuts with no visible touch alternative

The discipline: design for touch first, add mouse convenience on top. If a control works by tap, it works by click. If a control requires hover or right-click, it also has a long-press or a tap-for-context-menu. If a target is too small for a finger, it's also too small for usability in general.

The touchscreen desktop market
Surface Pro (Windows), many Dell XPS and HP Spectre models, some Lenovo ThinkPads — all have touchscreens. This is not a niche use case on Windows. On macOS, no touchscreen (Apple has consistently not shipped one). On Linux, some developer workstations have it. Designing for touch parity is primarily a Windows differentiator, and it's worth doing right.

10. The Verdict

Framework recommendation

For the 3D orb and Python integration: Qt6 / PySide6

If the 3D orb as described — real-time, reactive to voice, physically present — is a design requirement and not a nice-to-have, Qt6 is the answer. Qt Quick 3D is purpose-built for exactly this. The Python integration (PySide6) means the backend and frontend share a process and share signals natively — no bridge, no serialization. The native platform feel on all three OSes is genuine, not approximated. The downside is learning QML, which is a real cost but not a steep one.

If web tech skills are more important than 3D quality: Tauri v2

If you're comfortable with React (or want to be) and the orb being Three.js/WebGL is acceptable (it's genuinely excellent), Tauri v2 is a strong choice. Smallest bundle, great cross-platform story, the three-hop Python → Rust → webview is fast enough. Premium feel requires design work that Qt6 provides by default, but it's achievable.

Flutter: not recommended for this use case

Flutter's strengths (touch parity, cross-platform consistency) are shared by the other two. Its weakness (3D) is a hard requirement for this project. Dart is a new language with no advantage over QML or React for this use case. Flutter excels at the mobile app you might build later — not the desktop hub.

The design step before picking a framework
Prototype the orb. Before committing to a framework, spend two days building the orb — the 3D sphere, the particle field, the voice-reactive pulse — in both Tauri (Three.js) and Qt6 (Qt Quick 3D). The one that looks right and feels right to build is the framework for v6. This is the kind of decision you shouldn't make from a comparison table.
