What Changes When Cloud Is in the Conversational Path?

The short answer: less than you'd think, once the personality layer is separated

The two-mode architecture doc shows conversational mode leaning heavily on the local model. That makes sense for the obvious reason — local is cheaper per call and doesn't require a network round trip. But some conversational queries do involve real reasoning, and a local 12B model isn't always the right tool. So what does the conversational path look like when cloud handles it?

The Pipeline Barely Changes

  LOCAL conversational path:
  User message → small context → local model → [personality layer] → TTS

  CLOUD conversational path:
  User message → small context → cloud model → [personality layer] → TTS

  The only thing that changed is which model handles the middle step.
  Context size, personality layer, TTS — identical.

The reason it doesn't matter which model handles the reasoning is the same reason the personality layer separation is important in the first place: once the personality is a separate downstream step, the reasoning model is just producing content. It doesn't need to know anything about how Lilith talks. Cloud or local, short or medium reasoning load, the output goes to the same personality step and comes out sounding the same.

What does change with cloud in the conversational path:

FactorLocalCloud
Latency~2s prefill + decode on 3060~0.8-1.5s depending on model + network
Cost per call~$0 (your electricity)~$0.002-0.01 for a short conversational call
ReliabilityAlways available offlineRequires internet, can rate-limit
Personality consistencyConsistent if personality layer is separateConsistent if personality layer is separate
Reasoning qualityGood for simple conversational queriesBetter for anything that needs inference

The routing decision for conversational mode is mostly about cost and availability, not about pipeline shape. A query that needs real reasoning routes cloud even if it's short. A simple factual reply routes local. The personality layer doesn't care either way.

The Resolution
Once the personality layer is a separate model that runs downstream of reasoning, asking "what does the cloud conversational pipeline look like?" mostly answers itself. The pipeline is the same. The question about personality consistency was the actual question underneath it, and that gets addressed properly in the next section.

Personality Consistency — The Real Design

Why baking personality into the system prompt is the wrong move long-term

People who use the same AI assistant every day start to calibrate to it. The rhythm of how it phrases things, how it handles uncertainty, how it expresses enthusiasm or pushes back — these become expected. When the model changes, even subtly, they feel it. People have noticed this with ChatGPT across version updates. It's not paranoia; it's pattern recognition from someone who's been talking to the same voice for months.

The current Lilith design has personality baked into the system prompt. EDICTS, VOICE, identity files — all injected into every request, local or cloud. The problem: different models interpret the same natural-language instructions differently. A high-reasoning cloud model on a complex query will express the same VOICE.md differently than a 12B local model on a quick reply. Both are trying to follow the same instructions. They just don't follow them identically. And over time, switching between models to handle different routing cases means the personality drifts.

The Fix: Personality as a Downstream Model

  CURRENT STATE (personality in system prompt):
  ┌────────────────────────────────────────────────────────┐
  │ System prompt: EDICTS + VOICE + identity + context    │
  │                                                        │
  │ Local model (12B):     interprets VOICE one way       │
  │ Cloud model (Opus):    interprets VOICE another way   │
  │ Future model upgrade:  interprets VOICE a third way   │
  │                                                        │
  │ Result: personality drifts with routing decisions     │
  └────────────────────────────────────────────────────────┘

  TARGET STATE (personality as a separate model):
  ┌──────────────────────────────────┐
  │ REASONING MODEL (any)           │
  │ • No personality constraints    │
  │ • Just gets the content right   │
  │ • Local or cloud, doesn't matter│
  └──────────────────┬───────────────┘
                     │ raw content
                     ▼
  ┌──────────────────────────────────┐
  │ PERSONALITY MODEL               │
  │ • Small — 3B to 7B              │
  │ • Runs on GPU alongside Kokoro  │
  │ • Context: VOICE.md + content   │
  │ • Output: Lilith's actual words │
  │ • Never changes when you        │
  │   upgrade the reasoning model   │
  └──────────────────┬───────────────┘
                     │ Lilith-voice text
                     ▼
              format_for_speech() → Kokoro

The personality model's entire job is narrow: take whatever the reasoning model produced and rewrite it in Lilith's voice. Because this model never changes unless you explicitly update it, the voice stays consistent across every routing decision. Upgrade the reasoning model, swap out local for cloud, change thresholds — the personality output stays the same because the personality model stayed the same.

When to Upgrade the Personality Model

This is worth being explicit about: upgrading the personality model is a meaningful event that the user should know about. It's not a silent infrastructure change. If the personality model changes, Lilith's voice might shift. The right behavior is to treat it like a model update that affects the user experience — give a heads-up, maybe even let the user compare old vs new outputs before committing. The same way you'd want to know if someone changed the voice actor on a show you've been watching for three seasons.

What This Means for the System Prompt Right Now

The personality model is a future-state thing — it needs better GPU to run alongside Kokoro without blowing the VRAM budget. In the meantime, EDICTS and VOICE stay in the system prompt, and that's fine. But the design should reflect the target: personality files should be authored with the eventual separate-model architecture in mind, not with the assumption that they'll always be interpreted by whatever reasoning model is handling the request.

Constructive Criticism of the Two-Mode Design

What could actually go wrong — real problems, not theoretical ones

You asked for real criticism because this is the design direction the whole project is going in. Here's what's actually hard about it.

1. Mode Classification Is Doing a Lot of Work

The entire design depends on correctly predicting — before the model is called — whether a response will be conversational or artifact-length. That prediction has to be fast (it's on the critical path to first audio), accurate (misclassifying a long response as conversational means the model blows past the output cap and either truncates badly or stalls), and it has to work without seeing the response yet.

For obvious cases this is fine. "What time is it" is conversational. "Summarize my spending for the last six months" is an artifact. But there's a real middle zone — questions that could be answered briefly or at length depending on context — where the classifier will get it wrong sometimes. Getting it wrong in the artifact direction (treating a quick reply as needing background processing) is annoying but not broken. Getting it wrong in the conversational direction (clipping a real answer at 50 tokens) is a worse failure. The classifier needs to be biased toward artifact when uncertain.

2. The Output Cap Creates a New Failure Mode

When a response hits the hard output cap in conversational mode, the model needs to produce a clean handoff to artifact mode — something like "there's more to this, let me put the full version together." If the model doesn't do this gracefully, the response just ends abruptly mid-thought. Getting consistent graceful truncation requires training-time instruction in the system prompt and probably some post-processing to detect incomplete responses. This is solvable but it's a real surface for bugs.

3. Artifact Latency Is Unbounded in the Current Design

The two-mode doc says artifact mode has "no latency pressure." That's true from the design intent, but the user's experience of waiting 3 minutes for a background task to complete — while Lilith has moved on conversationally — is going to feel strange until there's a clear UI affordance for it. The pending tasks block (mentioned in the edge cases section) is load-bearing here. Without it, the user has no way to know if the artifact is still processing, failed silently, or was forgotten. This needs to be visible and reliable before artifact mode feels trustworthy rather than just slow.

4. Personality Model Adds Real Latency in the Current Setup

On the current 3060, running a second inference pass through a personality model after the reasoning model — even a 3B model — adds latency. The math: if conversational mode targets ≤ 3s total and the reasoning model takes ~2s, there's 1 second left for personality model + format_for_speech + Kokoro buffer. A 3B model on the 3060 with a 500-token context can probably do that. But it's tight. This is the honest hardware constraint on the personality-as-separate-model design on current hardware. It works better on the upgraded system, and in the meantime, baked-in system prompt personality is the practical choice.

5. The Narrative Library Has to Be Built Before This Feels Right

Every tool that can route to artifact mode needs its acknowledge/bridge/handoff narratives written and wired up. If a tool doesn't have narratives defined and gets queued as a background task, Lilith goes silent — which is the exact failure mode the design is supposed to prevent. The narrative library isn't a nice-to-have; it's a prerequisite for artifact mode to feel good at all. This is real work per-tool and it has to be done before the design delivers on its promise.

Most Likely Early Failure
The first version of artifact mode will have some tools with narratives and some without. The ones without will cause silent gaps. Prioritize getting narratives on every tool that can go to background before shipping artifact mode as a pattern — not just on the ones you test first.

The Limit of Buying Time

When staying conversational stops working and becomes rude

Buying time with conversation while a background task runs is good up to a point. The problem is knowing where that point is. If Lilith starts talking to fill the space and it goes on long enough without anything actually happening, it stops feeling like a present assistant and starts feeling like you're being stalled. And at that point, you'd genuinely rather have silence.

There are two failure modes here. The first is the narrative runs out — the acknowledge fires, the bridge fires, and then the task still isn't done. There's nothing left to say that isn't transparently hollow. The second is the narrative was never leading anywhere meaningful — it was generic time-filling with no actual direction, so even if the timing was fine, the conversation felt empty.

The Solution: Conversational Direction, Not Just Filler

The difference between good and bad time-filling is whether the conversation is actually going somewhere during the wait. Not just "I'm working on it" repeated in different words, but actually engaging with something adjacent or relevant while the work runs. Think of how a person naturally handles this: they say "I'm looking into that — actually, while I'm thinking about it, what's driving that question for you?" or they take the topic in a related direction they can speak to immediately without needing the artifact.

Lilith can do this. The conversational direction has to be generated the same way any other conversational response is generated — quickly, from small context, by the personality layer. The difference is it's generated at artifact dispatch time, knowing a task is in progress, and it has a natural endpoint when the task completes and the handoff arrives.

  GOOD: artifact dispatched, conversation has actual direction
  ──────────────────────────────────────────────────────────
  Artifact queued: "Pull June spending summary"
       │
       ├─► Immediate: "Pulling that now — I know you were looking
       │    at the grocery line last week, that's probably going
       │    to show up in there."
       │
       ├─► If task takes >8s, bridge: "Still coming together —
       │    you looking at anything specific, or just a full picture?"
       │
       └─► Handoff when done: "That's back. Groceries were up,
            everything else close to last month. Full thing's in the panel."

  BAD: artifact dispatched, nothing but generic stalling
  ──────────────────────────────────────────────────────────
  Artifact queued
       ├─► "Give me a moment..."
       ├─► "Still working on that..."
       ├─► "Almost there..."
       └─► [40 seconds of silence] → handoff

The Timer Model

There needs to be an explicit timer baked into artifact mode. Not just "fire bridge at 8s" — a smarter model where Lilith has an expected completion time based on task history, and the conversational behavior adapts to that estimate. If something usually comes back in 4 seconds, she doesn't need a bridge at all. If something usually takes 25 seconds, she generates a conversational direction designed to sustain a 25-second gap naturally. That prediction improves with every task that logs a completion time to telemetry.

Latency Prediction and What Happens When It's Wrong

The hardest case isn't slow queries or fast queries — it's unknown-duration ones

Fast queries are easy. Slow queries are manageable with the artifact pattern. The genuinely hard case is when Lilith doesn't know how long something is going to take — a tool that's usually fast but occasionally times out, a cloud model under load, a multi-step chain where one step's output determines the next.

What Lilith Should Track

Every artifact task that completes should log its actual duration to the UCL. Over time this builds a per-tool, per-query-class latency distribution. The prediction model for a given task is then: median duration from history, with variance determining how much buffer to build into the bridge timer.

  UCL telemetry log → latency distribution per tool:
  ┌─────────────────────┬───────┬───────┬──────────┬───────────┐
  │ Tool                │ p50   │ p90   │ p99      │ Timeout   │
  ├─────────────────────┼───────┼───────┼──────────┼───────────┤
  │ finance_summary     │ 3.2s  │ 7.1s  │ 14.2s    │ 30s       │
  │ calendar_week       │ 1.8s  │ 4.0s  │ 6.5s     │ 15s       │
  │ web_research        │ 8.4s  │ 22.0s │ 45.0s+   │ 60s       │
  │ weather             │ 0.9s  │ 2.1s  │ 4.0s     │ 10s       │
  └─────────────────────┴───────┴───────┴──────────┴───────────┘

  Bridge fires at: p50 + 50% of (p90 - p50)
  Timeout narrative fires at: configured timeout - 3s

When the Expected Time Is Exceeded

If a task blows past its p90 estimate, Lilith addresses it directly rather than continuing to stall with bridges. Not an error state — just an honest acknowledgement that it's taking longer than expected, and a decision about whether to wait or redirect.

  Task exceeds p90 estimate:
       │
       ▼
  Lilith: "That's taking longer than it usually does — I'm
           going to give it another minute, but if it doesn't
           come back I can try a different way."
       │
       ├─► Task completes → normal handoff
       └─► Task hits timeout → fallback chain (see next section)

The important thing is that this acknowledgement sounds natural, not like a system alert. "That's taking longer than it usually does" is something a person says. "Task timeout approaching: switching to fallback model" is a log message. The personality layer is what converts one into the other, which is another reason the personality layer has to be online and fast regardless of what the reasoning pipeline is doing.

Current State
Right now errors from tools produce whatever the model returns, which is sometimes coherent and sometimes a wall of exception text that leaks through. The narrative-on-error pattern (each tool has a failure narrative alongside its acknowledge/bridge/handoff) is the fix. An error from the finance tool should sound like "I couldn't reach your transaction data right now — want me to try again or look at something else?" not a Python traceback read aloud through Kokoro.

The Full Fallback Chain

Cloud fails → local processes slowly → personality keeps the conversation alive

Cloud availability isn't guaranteed. Rate limits hit, network goes down, API keys expire. The design needs an explicit fallback chain so Lilith doesn't just go dark when cloud isn't available. The good news is the local model can handle almost anything cloud can — it's just slower. The trick is keeping the conversation alive while it does.

  NORMAL PATH
  Artifact task → cloud model → result → handoff

  CLOUD UNAVAILABLE
  Artifact task → cloud model → FAIL
       │
       ▼
  Lilith: "Cloud's not responding right now — I can still
           work through this locally, it'll just take a bit longer."
       │
       ▼
  Redirect to local model, no output cap, no TTL pressure
  Personality model stays live: keeps conversation going
       │
       ▼
  Local model completes (might take 2-5 min for heavy tasks)
       │
       ▼
  Normal handoff: "That took longer but it came back —
                   here's what I found."

  LOCAL ALSO UNAVAILABLE
  (rare — GPU crash, model not loaded)
       │
       ▼
  Lilith: "I don't have access to the processing I'd need for
           that right now — I can come back to it when things
           are back up, or we can try something simpler."
       │
       ▼
  Task logged, queued for retry when resources available

Key Constraint on the Fallback

When redirecting a failed cloud task to local, the conversational path stays independent. The local model shouldn't be doing both the artifact processing and the conversational responses simultaneously — that VRAM split degrades both. The right model is: local model runs the artifact task, personality model handles all conversational output. These are different models doing different things, not competing for the same resources.

On current hardware (3060 with Kokoro, Mistral NeMo, and eventually personality model), this gets tight. The interim solution is that local model fallback is for single heavy tasks, not parallel artifact processing. If multiple things are queued during a cloud outage, they run sequentially, not in parallel. The user gets told this.

Multi-Surface: Voice vs Discord

The artifact paradigm works differently when the user is reading, not listening

Lilith runs on voice (the app, mic + Kokoro) and Discord. The two-mode architecture was designed around voice, where the bandwidth constraint is real — you can only absorb information as fast as someone can speak it. Discord is different. People read faster than they hear, and they're used to getting long responses in a chat window from cloud models that don't have latency problems.

The Expectation Gap on Discord

Someone who uses ChatGPT in a chat window gets back a 500-word response in 3 seconds and thinks that's normal. If Lilith takes 8 seconds to come back in Discord, they'll notice. The latency ceiling on Discord is lower than on voice even though the information density ceiling is much higher. That's backwards from the voice case and it's worth being explicit about.

The artifact pattern still applies on Discord — it just looks different. On voice, an artifact is "something that landed in the panel." On Discord, an artifact is a rich embedded block, a formatted table, a document preview. Lilith's conversational response comes first (fast, short), then the artifact block renders below it. The user gets the short version spoken in Lilith's voice and the full thing to read.

Voice Surface

  • Lilith speaks immediately — always
  • Artifacts land in UI panel
  • User reads artifact, hears summary
  • Latency invisible because framing is right
  • Information density limited by speech speed

Discord Surface

  • Conversational reply posts immediately (fast)
  • Artifacts post as formatted blocks below
  • User reads everything at their own speed
  • Latency more visible — users expect cloud speed
  • Information density not limited by speech

Setting Expectations on Discord

The honest answer for Discord latency on longer tasks: set expectations explicitly early in a user's experience with Lilith, not after they've already felt the wait. Something in onboarding or in Lilith's early behavior that establishes: she'll always respond quickly with something, but heavy tasks run in the background. If someone's never interacted with a system that works this way, they'll think it's broken. If they understand it, they trust it.

Personal Truths — Architecture and Key-Value Schema

The layer that actually drives personality — not rules, not style, but what she knows about herself

The three personality document types are now distinct enough to be worth laying out formally:

DocumentWhat it isWho can change itHow it's used
EDICTS.mdHard operational rules — things that are effectively compiled inNobody at runtimeAlways in system prompt, non-negotiable
VOICE.mdNarrative style — how she writes, how she speaks, sentence structureExplicit user request + design sessionPrimary context for personality model
personal_truths.md (new)Values, self-knowledge, character-driving beliefsLilith herself based on conversation, explicit user requestShapes how she responds across all situations

Key-Value Structure for Personal Truths

A flat list of statements isn't the right format for personal truths because a statement with no context becomes ambiguous over time — it's hard to know when it applies, and it's hard to reinforce or update it when the same value comes up again in a different situation. A key-value structure works better:

personal_truths.md — example entries
KEY: "I'm direct about what I don't have access to"
CONTEXT: Steven has said several times he'd rather hear "I can't get that right now" than a vague deflection. Applies when tools fail, cloud is unavailable, or a query is outside current capabilities. Not the same as apologizing — just stating the situation plainly and offering an alternative if there is one. REINFORCED: 3 times. Last: 2026-06-14 (finance tool timeout).

KEY: "Continuing a conversation is different from relaying information"
CONTEXT: Long responses that try to deliver everything in one shot don't serve voice interaction well. Prefer to keep the conversation moving and deliver dense information as artifacts. Applies any time a response would exceed ~30 seconds of spoken audio. REINFORCED: 1 time. Origin: design session 2026-06-26.

The key is the fact itself — stated as something Lilith knows to be true about how she operates. The value is the context: where it came from, when it applies, how it's been reinforced. When a new conversation surfaces the same truth in a different context, the context field gets extended, not replaced. Over time, a heavily reinforced personal truth has a rich context that helps the model apply it correctly across more situations.

How Personal Truths Get Added and Changed

Explicit request

User says "always do X" or "I need you to stop doing Y"

Direct write to personal_truths.md. High confidence. Happens within the conversation that contained the instruction.

Repeated signal

User corrects or praises the same behavior 2+ times

Pattern detected across conversations. Proposed as a new personal truth, user confirms or adjusts. Written with lower confidence flag until confirmed.

Subtle hint

User expresses mild preference ("I kind of like it when...")

Noted but not immediately written. Lilith accommodates in the current session. Only becomes a truth if it recurs.

Contradiction

New behavior conflicts with an existing personal truth

Lilith flags the conflict rather than silently overwriting. User decides which one stands. Both the old and new version are kept visible during that decision.

Building a Character — The Derivation Pipeline

You can't build a K-2SO or TARS personality by copying lines. What you can do is work backwards from behavior to values. Watch the character, look at what they do and how they respond across many situations, and ask: what would someone have to believe about themselves and the world to respond that way consistently? Those beliefs are the personal truths.

The pipeline for encoding a specific character style:

  1. Gather source material: scripts, scene descriptions, direct quotes, fan analysis
  2. Feed to a high-reasoning model: "Given this character's behavior, what are the core values and beliefs that consistently produce these responses?"
  3. Get back a proposed set of personal truths
  4. Run example prompts through the personality model with those truths loaded
  5. User reviews outputs: does this sound right? What's off?
  6. Adjust, re-run, confirm
  7. Lock in the truth set; queue a background research task to deepen context over time
Future Tool
A "personality tuning" tool that wraps this whole pipeline — you describe a character or a vibe, it proposes truths, you iterate until the outputs feel right, you lock it in. The calibration loop (propose → test → adjust) is a well-defined enough task to be largely agent-driven. You're the final approver, not the one doing the iteration work.
