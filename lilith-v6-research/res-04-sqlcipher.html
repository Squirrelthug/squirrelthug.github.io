Q1. sqlcipher3 vs pysqlcipher3 — which to use?

  • pysqlcipher3 (Deprecated): Unmaintained for years. It fails to build on Python 3.10+ due to C API changes in Python's sqlite3 module.
  • sqlcipher3 (Active Fork): Active fork supporting Python 3.10, 3.11, and 3.12. However, a raw pip install sqlcipher3 compiles from source and will fail on Windows/macOS if SQLCipher and OpenSSL developer libraries are not pre-installed.
  • sqlcipher3-wheels & sqlcipher3-binary (Recommended): These forks publish pre-compiled wheels for all major platforms (including SQLCipher 4 and OpenSSL bundled in), removing compiling complexity from user environments. sqlcipher3-wheels is the preferred package for robust cross-platform developer setups.

Q2. Pre-built wheels — what exists and what requires compilation?

Platform Package Pre-built Wheel Status Compiling Requirements (If Wheels Fail)
Linux x86_64 sqlcipher3-wheels / -binary Available (cibuildwheel compiled) sudo apt install libsqlcipher-dev, GCC, Python headers
macOS arm64 sqlcipher3-wheels Available (M1/M2 compatible) brew install sqlcipher openssl, Xcode command-line tools

Design note — when wheels fail and what the backup is Pre-built wheels fail in three scenarios: (1) a new Python version ships before the wheel maintainer publishes for it (typically 2–4 weeks lag), (2) the user's CPU architecture is not covered (rare — only affects exotic platforms), (3) PyPI is temporarily unreachable during CI. For all three, the fallback is source compilation: libsqlcipher-dev + GCC on Linux, brew install sqlcipher openssl + Xcode CLI tools on macOS, Visual Studio C++ build tools + vcpkg on Windows. The design document should specify that CI tries wheel install first, then falls back to documented compilation steps on failure. This covers all realistic scenarios without blocking delivery.

Q4. PRAGMA key syntax and cipher configuration

When opening a SQLCipher database, the passphrase must be set immediately after connection before any queries are run:

import sqlcipher3

# Connect to database
conn = sqlcipher3.connect('contacts_db.db')

# 1. Text Passphrase Syntax
conn.execute("PRAGMA key = 'my_secure_passphrase'")

# 2. Raw 32-Byte Hex Key Syntax (bypasses KDF, fast)
conn.execute("PRAGMA key = \"x'2ddb9f8742d4a1ef186bd44e3e3b3c3d5e5f505152535455565758595a5b5c5d'\"")
  • Cipher Compatibility: SQLCipher 4 uses standard compatibility across platforms. If opening a legacy SQLCipher 3 database, run PRAGMA cipher_compatibility = 3; immediately after the key is applied.
  • SQLCipher 4.x Defaults:
    • Page Size: 4096 bytes (SQLCipher 3 used 1024)
    • Key Derivation (KDF): PBKDF2-HMAC-SHA512
    • KDF Iterations: 256,000 (SQLCipher 3 used 64,000)
    • HMAC Algorithm: HMAC-SHA512 (SQLCipher 3 used SHA1)
  • Incorrect Key Verification: SQLite performs lazy evaluation. To verify if a key is correct, run a dummy query (e.g., query sqlite_master) immediately after connection: 
    try:
    conn.execute("SELECT count(*) FROM sqlite_master;")
except sqlcipher3.DatabaseError as e:
    # Incorrect key raises: "sqlite3.DatabaseError: file is not a database"
    print("Invalid decryption key or corrupted database file.")

Q5. Performance overhead of AES-256 encryption

  • Overhead Percentages:
    • Single-row lookup (primary key): Negligible overhead (<5%).
    • 100-row scans: Moderate overhead (~10–15%) because SQLCipher must decrypt pages into memory.
    • Write operations (inserts/updates): ~20–30% overhead compared to plaintext SQLite due to encryption calculations. Wrap multiple inserts in a single transaction (BEGIN ... COMMIT) to reduce page-writing frequency.
  • WAL Mode: Yes, enabling Write-Ahead Logging (WAL) significantly increases SQLCipher performance by separating readers and writers. The WAL journal files are also fully encrypted.
  • Text Passphrase vs. Raw Hex Key: Using PRAGMA key with a text passphrase forces KDF derivation (256,000 PBKDF2 iterations), adding a 50–100ms CPU latency block during connection. Using a raw 32-byte hex key bypasses KDF, making connection establishment instantaneous (<1ms).
Decision — raw hex key confirmed; performance acceptable for conversational use The hot path (conversational queries) only touches single-row primary-key lookups — overhead is under 5%, well within any latency budget. Background writes (L1/L2 enhancement, sync) carry the ~20–30% write overhead but run asynchronously and never block the response pipeline. The 50–100ms connection-open penalty from text-passphrase KDF is eliminated by using the raw 32-byte hex key via PRAGMA key = "x'...'" (derived by Argon2id — see RES-13). WAL mode confirmed. Decision is locked.

Q6. Encrypted backup with VACUUM INTO

  • VACUUM INTO: Yes, VACUUM INTO 'path/to/backup.db' works in SQLCipher. The backup file is encrypted using the *same* key as the active connection.
  • Different Key Export: To back up a database with a different key, use the sqlcipher_export() API in the Python bindings: 
    conn.execute("ATTACH DATABASE 'backup.db' AS backup KEY 'new_secure_key';")
conn.execute("SELECT sqlcipher_export('backup');")
conn.execute("DETACH DATABASE backup;")
    This successfully duplicates schema and decrypted/re-encrypted records to the target database directly within python-sqlcipher3.

Q7. Thread safety

  • Check Same Thread: Yes, sqlcipher3 inherits Python sqlite3's check_same_thread parameter.
  • Concurrency Pattern (Lilith v6): For multiple readers and a single writer:
    1. Configure WAL mode and set a busy timeout of 5000ms: 
      conn.execute("PRAGMA journal_mode = WAL;")
conn.execute("PRAGMA synchronous = NORMAL;")
conn.execute("PRAGMA busy_timeout = 5000;")
    2. Do not share a single connection across threads (even with check_same_thread=False). SQLite transactions are connection-scoped. If multiple threads share a connection, their transactions will interfere.
    3. Recommended: Open a separate, local database connection for each thread/background job and close it when the job is completed.
