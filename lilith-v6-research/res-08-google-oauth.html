RES-08 — Google OAuth for Desktop Apps

Researched: June 28, 2026 · google-auth-oauthlib 1.4.0 · google-api-python-client 2.198.0

Q1. Correct OAuth 2.0 Flow for Locally Installed Desktop Apps

Use Google's installed-app OAuth 2.0 authorization code flow with PKCE and a loopback redirect. The deprecated out-of-band redirect urn:ietf:wg:oauth:2.0:oob must not be used. Google documents that desktop apps open the system browser, start a temporary HTTP listener on 127.0.0.1 or ::1 using a random available port, and receive the authorization code on that loopback URL.

The user should create an OAuth client of type Desktop app, not Web application. Installed apps cannot keep a client secret confidential; Google treats the downloaded secret as an app identifier, not a server-side secret. A local temporary port is normal for this flow; Lilith does not expose a persistent server.

Publishing status — move to Production for durable refresh tokens The app should not remain in external Testing mode indefinitely. Google documents refresh-token expiration for external apps in Testing; the Squirrelthug-managed OAuth app must be moved to Production before GA. For BYOK users, the same applies to their own GCP project's consent screen.

Q3. API Rate Limits

Google exposes Calendar and People API quotas in each Google Cloud project's Quotas page; exact values can vary by project and can change. Historically, Calendar has default per-minute limits suitable for personal sync, and People API enforces per-user and per-project limits. For one local user syncing Calendar and Contacts every 15 minutes with incremental tokens, default quotas are sufficient by a wide margin.

On quota exhaustion, Google APIs return HTTP 403 or 429 errors with reasons such as rateLimitExceeded or userRateLimitExceeded. Lilith should use exponential backoff with jitter and persist the last successful sync token so a failed poll does not force a full resync.

Q4. Incremental Sync for Google Calendar

Calendar incremental sync uses nextSyncToken. First, Lilith performs a full events.list request using the same stable query parameters it intends to use later, follows all nextPageToken pages, then stores the final nextSyncToken. Later polls call events.list(syncToken=stored_token). Changed and deleted events are returned; deleted events have status="cancelled", so Lilith must treat them as tombstones.

If Calendar returns HTTP 410 Gone, the sync token is invalid and the local cache must be cleared or reconciled by a new full sync. Token invalidation can happen after token expiration or query-parameter mismatch. Known issue to guard against: a calendar may omit some historical items if the initial full sync used restrictive time windows, so Lilith should document exactly what window it syncs and keep parameters stable.

from googleapiclient.discovery import build
from googleapiclient.errors import HttpError

def sync_calendar(creds, calendar_id, state):
    service = build("calendar", "v3", credentials=creds, cache_discovery=False)
    params = {"calendarId": calendar_id, "singleEvents": True, "showDeleted": True}
    if state.sync_token:
        params["syncToken"] = state.sync_token
    else:
        params["timeMin"] = state.initial_time_min_iso

    try:
        while True:
            resp = service.events().list(**params).execute()
            for event in resp.get("items", []):
                if event.get("status") == "cancelled":
                    upsert_calendar_tombstone(event["id"])
                else:
                    upsert_calendar_event(event)
            if "nextPageToken" not in resp:
                state.sync_token = resp["nextSyncToken"]
                save_sync_state(state)
                break
            params["pageToken"] = resp["nextPageToken"]
    except HttpError as exc:
        if exc.resp.status == 410:
            clear_calendar_cache(calendar_id)
            state.sync_token = None
            save_sync_state(state)
            return sync_calendar(creds, calendar_id, state)
        raise

Q5. Incremental Sync for Google Contacts

The People API people.connections.list supports requestSyncToken=true on the first full sync and returns nextSyncToken after all pages. Later calls pass syncToken. Deleted contacts are included as tombstones using PersonMetadata.deleted. Google documents that sync tokens expire after a limited period, so Lilith must handle EXPIRED_SYNC_TOKEN/410-style failures by running a new full sync.

The deprecated Contacts API had features not mirrored one-for-one in People API. For v6 v1, People API is still the correct API because Lilith only needs read-only contact records and tombstones.

Q6. Refresh Token Shelf Life

For a Desktop app OAuth client in Production, refresh tokens are intended to last until revoked or invalidated. They can stop working if the user revokes consent, changes a password for sensitive scopes, the token is unused for an extended period, the app exceeds per-user refresh-token limits, the OAuth app is deleted, or the app remains in external Testing status. A user returning after six months should usually still work, but Lilith must be ready to re-authenticate.

Graceful handling: catch RefreshError, clear only the stored Google token from keyring, mark sync disabled, and show a re-auth action. Do not delete local synced data unless the user asks.

Q7. BYOK OAuth Setup — User-Facing Flow

  1. User creates a Google Cloud project.
  2. Enable Google Calendar API and People API.
  3. Configure OAuth consent screen, add themselves as a test user during setup, then publish to Production for durable tokens if using an external app.
  4. Create OAuth Client ID with application type Desktop app.
  5. Download client_secret.json and import it into Lilith. Lilith extracts client ID/secret, obtains a refresh token, then stores serialized credentials in OS keychain via RES-07.

This is technically viable but high-friction for non-technical users. The simplification path is a guided setup wizard with screenshots, validation, and a future optional hosted OAuth broker if Lilith later wants first-party OAuth instead of strict BYOK.

import json
import keyring
from google.oauth2.credentials import Credentials
from google.auth.transport.requests import Request
from google_auth_oauthlib.flow import InstalledAppFlow

SCOPES = [
    "https://www.googleapis.com/auth/calendar.readonly",
    "https://www.googleapis.com/auth/contacts.readonly",
]
SERVICE = "lilith.google.oauth"

def authorize(client_secret_path: str, account: str) -> Credentials:
    flow = InstalledAppFlow.from_client_secrets_file(client_secret_path, SCOPES)
    creds = flow.run_local_server(host="127.0.0.1", port=0, access_type="offline", prompt="consent")
    keyring.set_password(SERVICE, account, creds.to_json())
    return creds

def load_credentials(account: str) -> Credentials:
    raw = keyring.get_password(SERVICE, account)
    if not raw:
        raise RuntimeError("Google account is not authorized")
    creds = Credentials.from_authorized_user_info(json.loads(raw), SCOPES)
    if creds.expired and creds.refresh_token:
        creds.refresh(Request())
        keyring.set_password(SERVICE, account, creds.to_json())
    return creds
Architectural note — BYOK OAuth should be one option, not the only path The current design requires users to create their own Google Cloud project and OAuth credentials — high-friction for non-technical users. Both paths should be available: (1) Squirrelthug-managed OAuth app — user clicks "Connect Google" and authorizes via standard Google Sign-In; no GCP project needed, but requires Squirrelthug's app to pass Google verification and remain active. (2) BYOK OAuth — user creates their own GCP project and credentials, full independence, suitable for power users and privacy-focused users. This principle extends to LLM API key onboarding as well — the vertical slice should minimize friction for non-technical users while keeping BYOK as an option.
Final recommendation: CLEAR — dual OAuth path resolves UX friction. Lilith ships two paths: (1) Squirrelthug-managed OAuth app — one-click "Connect Google" for non-technical users; requires Squirrelthug to pass Google OAuth verification and move the app to Production before GA. (2) BYOK OAuth — user creates their own GCP project, imports client_secret.json, full independence for power users. Both paths use the loopback PKCE flow and store the refresh token in the OS keychain via python-keyring. No blocker remains.
Action Plan — Google OAuth + APIs
  • Auth module: lilith/integrations/google.py — handles both BYOK and managed-app paths. Detects which path is configured and uses the correct client credentials.
  • BYOK path: Wizard UI imports client_secret.json, calls InstalledAppFlow.run_local_server(host="127.0.0.1", port=0), stores serialized credentials in OS keychain under KEYCHAIN_SERVICE / "google_credentials".
  • Managed-app path: Lilith ships a bundled client ID/secret (non-secret for desktop apps), user clicks Connect, same PKCE flow. Squirrelthug moves consent screen to Production before GA.
  • Token refresh: On load, check expiry; if expired and refresh token present, call creds.refresh(Request()) and re-store in keychain.
  • Incremental sync: Store nextSyncToken (Calendar) and People sync token in the local Calendar/Contacts SQLCipher DB. Handle 410 Gone by clearing cache and re-syncing.
  • Rate limits: Exponential backoff with jitter on 403/429. Persist last successful sync token on every page to survive partial failures.
  • Scopes — v6 v1: calendar.readonly and contacts.readonly only — read-only external APIs, no writes in v6 v1.
  • Error recovery: On RefreshError: clear Google token from keychain, disable sync, surface re-auth action in UI — do not delete local synced data.
