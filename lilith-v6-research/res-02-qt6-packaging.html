Q1. Which Python packaging tool works best with PySide6?

For PySide6 applications, packaging tools vary significantly in their maturity, support for Qt plugins, and ability to handle native C extensions.

Tool Out-of-the-Box PySide6 Support QML & Asset Gathering Active Maintenance Community Usage & Verdict
PyInstaller Excellent. Has built-in hooks for PySide6 that capture binaries and core plugins automatically. Needs explicit QML collection rules (via --collect-all PySide6 or spec files). Extremely Active Industry standard. Largest community, highly reliable for dynamic C extensions. Recommended.
cx_Freeze Moderate. Requires manual configuration of Qt directories and environment paths in setup.py. Requires manually adding QML directories to build options. Active Niche. Harder to configure and offers fewer advantages over PyInstaller.
Briefcase (BeeWare) Poor. Primarily designed for native macOS/iOS Cocoa or Windows Win32 apps via Toga. Extremely difficult to integrate with Qt Quick and QML bundles. Active Not recommended. Very little community footprint for PySide6 packaging.

Q2. How do you package QML files and Qt Quick 3D assets?

To distribute PySide6 applications that use QML and 3D assets, there are two primary patterns:

  • Qt Resource System (Recommended): Compile QML files, textures, and assets into a binary resource module using pyside6-rcc. Run: 
    pyside6-rcc resources.qrc -o resources_rc.py
    Import resources_rc.py in Python and load QML via qrc:///main.qml. This embeds assets directly into a Python import module, protecting source code and ensuring PyInstaller/Nuitka packages them without file paths failing.
  • External Directories: Copy raw .qml files using PyInstaller's --add-data. At runtime, resolve relative paths using pathlib.Path(__file__).parent. However, this exposes QML source code as plaintext to the user.
  • Qt Quick 3D Shaders: Qt Quick 3D shaders are compiled at runtime by default, which can cause frame drops. For production, pre-compile shaders using the qsb (Qt Shader Baker) tool and bundle them in the resource system.

Q3. Packaging C extension dependencies on each platform

Lilith v6 has complex C extensions: sqlcipher3, ctranslate2 (for faster-whisper), and sounddevice (linking to PortAudio).

Platform sqlcipher3 Status ctranslate2 / faster-whisper Status sounddevice / PortAudio Status
Linux Pre-built wheels on PyPI are rare. Recommendation: Use pip install sqlcipher3-wheels which bundles SQLCipher static binary. Official x86_64 wheels exist on PyPI. Bundles C++ libraries. Handles .so loading cleanly. PyPI wheel includes pre-compiled libportaudio.so. Handled automatically.
macOS (Apple Silicon) sqlcipher3-wheels provides pre-built Apple Silicon (arm64) wheels. Prebuilt arm64 wheels are available on PyPI. Native Metal/CPU support works. PyPI wheel includes pre-compiled Universal libportaudio.dylib. Handled automatically.
Windows Standard sqlcipher3 fails to compile. Must use sqlcipher3-wheels or manually package OpenSSL DLLs. Prebuilt x86_64 wheels exist on PyPI. Blocker: CUDA support requires packaging NVIDIA DLLs (cuBLAS/cuDNN). PyPI wheel includes pre-compiled portaudio.dll. Handled automatically.

NVIDIA CUDA DLLs on Windows/Linux: Bundling CUDA DLLs (e.g., cublas64_12.dll, cudnn_ops_infer64_8.dll) adds ~1.5 GB to the bundle. To keep size reasonable, the client app can be built CPU-only by default, or download CUDA DLLs dynamically on first launch if a GPU is detected.

Q4. macOS code signing and Apple notarization

To avoid macOS gatekeeper blocks, we must implement Apple Notarization in our GitHub Actions pipeline:

  1. Certificate: Obtain a Apple "Developer ID Application" certificate. Install it into the GHA runner keychain.
  2. Entitlements: Python dynamic loading requires specific exceptions. Create an entitlements.plist: 
    <?xml version="1.0" encoding="UTF-8"?>
<plist version="1.0">
<dict>
    <key>com.apple.security.cs.allow-unsigned-executable-memory</key>
    <true/>
    <key>com.apple.security.cs.disable-library-validation</key>
    <true/>
</dict>
</plist>
  3. Code Signing: Run `codesign` on every embedded .so/.dylib, then sign the main executable and the .app bundle: 
    codesign --force --options runtime --timestamp --entitlements entitlements.plist --sign "Developer ID Application: Company Name (TEAMID)" dist/Lilith.app
  4. Packaging & Submission: Package the app into a .dmg (using create-dmg), sign the .dmg, and submit for notarization: 
    xcrun notarytool submit Lilith.dmg --keychain-profile "notary-profile" --wait
  5. Stapling: Staple the notarization ticket to the DMG: 
    xcrun stapler staple Lilith.dmg

Q5. Windows code signing

  • EV vs Standard OV Certificates: EV (Extended Validation) certificates require strict verification and cloud HSM storage. They provide instant reputation bypassing Windows Defender SmartScreen. Standard OV certificates are cheaper but require building reputation over time, meaning users will see SmartScreen warnings initially.
  • Signing in GitHub Actions: Can use windows-latest runner which includes signtool.exe. Certificate credentials should be stored in GitHub Secrets, and signing executed via a tool like SignPath or direct Azure Key Vault integration (mandatory for modern HSM-backed certificates).
  • Indie options: There are no free code-signing certificates for commercial closed-source apps. SSL.com and Certum offer standard code signing certificates starting around $200–$300/year.

Q6. Linux AppImage generation

  • Tooling: Use linuxdeploy with the linuxdeploy-plugin-qt and linuxdeploy-plugin-python. This combination compiles the environment and correctly collects the PySide6 runtime dependencies.
  • GLIBC & OpenSSL Compatibility: AppImages are backwards compatible with older GLIBC versions, but not forwards. Therefore, we must build the AppImage on the oldest active Ubuntu LTS runner (Ubuntu 20.04 or 22.04) in GitHub Actions. If built on Ubuntu 24.04, the AppImage will crash when run on older distributions.
  • Flatpak Alternative: Flatpak is a robust alternative. The official KDE runtime includes Qt6 and PySide6, providing a sandboxed environment that resolves GLIBC/OpenSSL distribution mismatches.

Q7. Approximate final bundle size

A packaged PySide6 client application is inherently heavy due to Qt's framework size and native extensions:

  • Base Application Bundle (PySide6 + Python Runtime + App UI): ~110 - 130 MB compressed.
  • With Whisper/Kokoro CPU Inference (ctranslate2 + models): ~250 - 350 MB compressed (includes Whisper-Tiny/Base + Kokoro model weights).
  • With GPU support bundled (NVIDIA CUDA + cuDNN DLLs): ~1.5 - 2.0 GB compressed.

Conclusion: To manage installer sizes, the installer should exclude CUDA DLLs by default, and have the application download CUDA libraries dynamically on-demand if the user elects to use local GPU acceleration.

Q8. pyside6-deploy tool

pyside6-deploy (introduced in Qt 6.4) is a deployment CLI tool designed by The Qt Company. It acts as a wrapper around Nuitka to compile Python applications into C++ and produce standalone executables.

  • Viability: While excellent for simple PySide6 apps, it is currently not viable for Lilith v6. It lacks robust dependency tracking hooks for complex native C extensions (like ctranslate2 and sqlcipher3-wheels), resulting in compile failures or missing shared library bindings at runtime.
  • Recommendation: Use PyInstaller in One-Folder mode (--onedir) for the packaging step. It has a mature hook ecosystem that safely detects and packages complex binary extension modules.
Recommendation: PyInstaller (One-Folder Mode) + Platform installers The recommended packaging workflow for Lilith v6 is using PyInstaller in One-Folder (--onedir) mode to ensure LGPLv3 relinking compliance, and then wrapping the folder in platform-specific installers:
  • Linux: Wrap in an AppImage built on an Ubuntu 20.04 or 22.04 GitHub Actions runner.
  • macOS: Sign with Developer ID Application certificate and entitlements.plist, wrap in a .dmg, notarize via notarytool, and staple.
  • Windows: Compile utilizing sqlcipher3-wheels, and sign using an OV/EV certificate in GitHub Actions. Exclude CUDA DLLs from the base bundle, downloading them dynamically if required.
Key Risk: High installer bundle size if CUDA DLLs are bundled (~1.5 GB). Resolution is to fetch GPU runtime libraries on-demand.
